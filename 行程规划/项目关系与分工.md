# 行程规划：项目关系与分工

更准确地说：

- `dufu`：旅游行程业务平台 / 主后端
- `libai`：接在 `dufu` 之上的 AI 对话与工作流服务
- 两者共同组成同一套“旅游行程产品能力”

可以简单理解为：`libai` 调用 `dufu`，由 `libai` 负责 AI 对话与 Agent 编排，由 `dufu` 负责实际旅游业务能力承接。

## 项目一：libai

- 框架：`FastAPI + LangChain/LangGraph + OpenAI`
- 入口：`app/main.py`
- 角色：聊天与 Agent 编排层

### 结构特征

- 存在 `graph/`、`prompts/`、`tools/`
- 暴露接口：`/aiChat/sendMessage`、`/chat/sendMessage`
- 使用 `SSE` 流式输出
- 在 `app/services/common/http_request_service.py` 中统一封装对下游业务接口的调用

### 职责理解

`libai`` 更像 AI 交互层 / 编排层：

- 接收用户聊天请求
- 组织 Prompt、Graph、Tools、Agent 流程
- 将具体业务请求下发给后端业务服务
- 以流式方式把结果返回给前端或调用方

## 项目二：dufu

- 框架：`Spring Boot + MyBatis + OpenFeign + Dubbo + QMQ`
- 启动类：`Application.java`
- 角色：旅游业务主后端

### 结构特征

- Maven 多模块：`qal_dufu_web`、`qal_dufu_web-api`
- Controller 很多，覆盖路线、交通、酒店、聊天、攻略图、风控、公共能力
- 依赖数据库、Dubbo、Redis、QMQ 等企业基础设施

### 职责理解

`dufu` 更像业务中台 / 主后端：

- 承载旅游场景下的核心业务接口
- 提供路线、交通、酒店、风控、公共能力等服务
- 对接数据库及公司内部基础设施
- 为上层 AI 服务或前端业务提供实际数据与业务处理能力

## 两个项目的直接对应关系

可以理解成“`libai` 调 `dufu`”：

- `libai` 的天气能力 -> `dufu` `CommonController.java` `/dufu/client/common/queryWeather`
- `libai` 的文本风控 -> `dufu` 同文件 `/dufu/client/common/riskContentCheck`
- `libai` 的记录 cookie -> `dufu` 同文件 `/dufu/client/common/recordUserCookie`
- `libai` 的机票 / 火车 / 低价机票 -> `dufu` `TransportController.java`
- `libai` 的路线检索 / 行程调整 -> `dufu` `RouteController` 相关接口
- `libai` 的酒店问一问消息查询 / 保存，也有单独对接 `dufu/sushi` 风格接口的痕迹

## 当前可记忆的分工结论

### dufu 负责

- 旅游业务能力本身
- 路线、交通、酒店、攻略、聊天落库、风控、公共服务
- 企业级基础设施集成

### libai 负责

- AI 聊天入口
- Agent / Workflow 编排
- Prompt、Tools、Graph 组织
- 将用户自然语言需求映射到 `dufu` 的业务接口

## 一句话总结

`dufu` 是旅游业务主后端，`libai` 是其上层的 AI 对话与工作流服务；两者不是重复关系，而是上下游协作关系。
