# JVM 内存与垃圾回收面试题

## Q1. JVM 概览

JVM（Java 虚拟机）是实现 Java 一次编译、到处运行的关键。

**JVM 三大主要部分：**

1. **类加载子系统**：根据全限定名载入类和接口（`.java` → `.class`）。编译后的 `.class` 字节码文件在不同平台上，只需要对应版本的 JVM 即可运行，实现跨平台

2. **执行引擎**：执行载入类中方法的指令

3. **运行时数据区**：程序运行时，字节码、局部变量、对象、运算中间结果、返回值、参数都要存在数据区中

---

## Q2. JVM 运行时数据区（≠JMM）

**五个主要部分：**

| 区域    | 线程归属 | 作用                                    |
| ----- | ---- | ------------------------------------- |
| 程序计数器 | 线程独有 | 控制下一条指令的执行（循环、分支、跳转等）                 |
| 虚拟机栈  | 线程独有 | 保存 Java 方法运行时的数据结构，方法的调用和结束对应栈帧的入栈和出栈 |
| 本地方法栈 | 线程独有 | 和虚拟机栈类似，针对本地方法                        |
| 堆     | 线程共享 | 存储 new 出来的对象，垃圾回收器的重要管理区域             |
| 方法区   | 线程共享 | 存储全局变量、静态变量等                          |

---

## Q3. Java 内存模型（JMM）

JMM 是一种规范，屏蔽各种硬件和操作系统的内存访问差异：
- 程序中的变量存储在**主内存**中
- 每个线程拥有自己的**工作内存**并存放变量的拷贝
- 线程读写自己的工作内存，通过主内存与其他线程进行变量交互

**目的：** 解决多线程通信时的本地内存数据不一致、编译器指令重排、处理器乱序执行等问题，简化多线程编程，增强程序可移植性

---

## Q4. JVM 垃圾回收机制

**三个核心问题：**

**1. 什么样的数据是垃圾（需要回收）？**
- 类中不再被引用的对象空间
- 判断方法：引用计数法、可达性分析法

**2. 什么时间进行垃圾回收？**
- 内存不足时
- 定期进行

**3. 回收方式：**
- **分代回收：**
  - 新生代收集器：Serial、Parallel
  - 老年代收集器：CMS、Serial Old、Parallel Old
- **分区域回收：** G1（混合收集器）：对新生代和老年代混合回收，可以并发标记、并发回收

---

## Q5. JVM 垃圾回收算法

**三种算法：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 标记清除 | 扫描标记需要清除的对象，然后清除 | 简单 | 扫描开销大，空间碎片多，内存不连续 |
| 标记复制 | 内存分两半，只在一块存储，用完后将存活对象复制到另一块，清除使用过的内存 | 无内存碎片，效率高 | 内存只用一半，浪费 |
| 标记整理 | 标记存活对象，将存活对象移动到内存区，清除边界外未标记对象 | 无内存碎片 | 需要移动对象 |

**应用场景：**
- 新生代常用：标记复制算法
- 老年代常用：标记清除、标记整理算法

---

## Q6. GC 可达性分析法

**方法：** 从 GC Roots 根对象作为起始节点集，通过引用搜索，能搜索到的对象在引用链中，不在引用链中的对象视为不可达，被第一次标记为需要清除的对象

**后续处理：**
- 检查是否有执行 `finalize()` 的需要
- 没有需要则直接垃圾回收
- 有需要则加入 F-Queue 队列执行 `finalize()`，执行后仍不在引用链中的被垃圾回收

**GC Roots 通常是：**
- 栈中引用对象
- 方法区中静态引用变量、常量池中的引用对象
- 同步锁持有的对象

---

## Q7. Java 四种引用方式

| 引用类型 | 说明                  | 回收时机           |
| ---- | ------------------- | -------------- |
| 强引用  | new 出来的对象，引用关系在则不回收 | 永不回收（内存不足也不回收） |
| 软引用  | `SoftReference`     | 内存即将溢出时回收      |
| 弱引用  | `WeakReference`     | 只要垃圾回收就会被回收    |
| 虚引用  | `PhantomReference`  | 只要垃圾回收就会被回收    |

---

## Q8. CMS 垃圾回收器

**算法：** 标记清除算法

**四个步骤：**
1. **初始标记**：标记与 GC Roots 直接关联的对象（**STW** - Stop The World）
2. **并发标记**：标记 GC Roots 可以引用搜索到的对象（并发，不暂停用户线程）
3. **重新标记**：更新在并发标记阶段重新加入引用链的对象（**STW**）
4. **并发回收**：对标记需要清除的对象回收（并发）

**应用场景：** 服务端的老年代，以在垃圾回收期间用户线程低停顿为目标

**缺点：** 标记清除算法产生内存碎片

---

## Q9. G1 垃圾回收器

**原理：** 将内存区域划分为多个等大的区域（region），以 region 为回收单位，预估回收收益和停顿时间，维系优先级表，根据优先级表进行区域回收

**四个流程：**
1. 初始标记
2. 并发标记
3. 最终标记
4. 并发筛选回收（根据优先级表进行回收，最后进行空间整合）

**算法：** 
- 整体：标记整理算法（多 region 之间）
- 局部：标记复制算法（region 内）

| 步骤 | 说明 |

| 初始化 | 调用构造器的过程 |
| 使用 | 正常使用 |
| 卸载 | 卸载类 |

## Q11. JVM 双亲委派模型

**类加载器分为：** 启动类加载器、扩展类加载器、应用程序类加载器（逻辑上的父子关系）

**双亲委派模型：** 一个类加载器接收到类加载请求，不会直接加载，而是将请求**上交给父级**，所有类加载都会传到 JVM 由启动类加载器加载。只有当父类加载器表明自己无法完成加载，才转发给下一级加载器

**优点：** 不会出现类覆盖问题

## Q13. 内存溢出（OOM）

**定义：** 需要申请的内存空间大于系统能提供的内存空间，实实在在的内存不够

**主要原因：**
- 内存加载的数据量太大
- 存在死循环，或死循环中产生大量对象实体
- 启动内存值设定过小

---

## Q14. 内存泄露

**定义：** 该释放的内存没有释放，不再使用的对象仍然被引用，导致内存泄露

**多见于：** 自己的动态容器存储变量导致（如 ThreadLocal 未 remove）

---

## Q15. 如何打破双亲委派模型

**双亲委派模型的实现：** 在 `ClassLoader.loadClass()` 方法中实现的，该方法首先将请求委托给父类加载器

**打破方式：** 重写 `loadClass()` 方法，改写加载规则，不再先向上委托，而是直接自己加载

**典型场景：**
- **Tomcat**：每个 Web 应用有独立的 ClassLoader，隔离不同应用的类
- **JDBC / SPI 机制**：启动类加载器无法加载第三方驱动，使用线程上下文类加载器（Thread Context ClassLoader）打破
- **OSGi 框架**：实现模块化，各模块有自己的类加载器，类加载是网状结构而非树状

## Q16. 对象初始化顺序

**当创建一个子类对象时，执行顺序如下：**

1. 父类静态变量 / 静态代码块（按声明顺序）
2. 子类静态变量 / 静态代码块（按声明顺序）
3. 父类实例变量 / 非静态代码块（按声明顺序）
4. 父类构造方法
5. 子类实例变量 / 非静态代码块（按声明顺序）
6. 子类构造方法

**总结：** 静态内容先执行（只执行一次），然后父类非静态和构造方法，最后子类非静态和构造方法

## Q17. ZGC 垃圾回收器

ZGC（Z Garbage Collector）是 JDK 11 引入的低延迟垃圾回收器

**核心特点：**
- 停顿时间不超过 **10ms**，且不随堆大小增长而增长
- 支持 TB 级别的堆内存
- 使用**着色指针**（Colored Pointers）：将 GC 信息存储在对象引用指针的高位中
- 使用**读屏障**（Load Barrier）：在对象被读取时进行必要的处理

**与 G1 对比：**
| 特性 | G1 | ZGC |
|------|----|----|
| STW 停顿 | 几十到几百毫秒 | < 10ms |
| 堆大小 | GB 级 | TB 级 |
| 吞吐量 | 高 | 略低于 G1 |
| 适用场景 | 通用 | 超低延迟场景 |

## Q18. 如何排查 JVM Full GC 频繁

**排查步骤：**

1. `jstat -gcutil <pid> 1000`：实时监控 GC 情况，查看 FGC 频率
2. `jmap -heap <pid>`：查看堆内存使用情况
3. `jmap -histo <pid>`：查看各类对象数量和大小，找出大对象
4. Dump 堆快照：`jmap -dump:format=b,file=heap.hprof <pid>`，用 MAT 工具分析内存泄露

**常见原因：**
- 内存泄露：对象一直被引用无法回收
- 堆内存设置过小（`-Xmx` 不够）
- 大量大对象直接进入老年代
- 代码中存在静态集合不断增长
- 使用了不恰当的缓存策略
