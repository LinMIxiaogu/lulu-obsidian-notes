# IO 与 NIO 面试题

## Q1. BIO、NIO、AIO

Unix 提供了五个 I/O 模型：阻塞 I/O、非阻塞 I/O、异步 I/O、I/O 多路复用、I/O 信号驱动

| 模型 | 说明 | 类型 |
|------|------|------|
| BIO（阻塞 I/O）| 一次只能操作一个 IO 请求，阻塞等待数据准备 | 同步 IO |
| NIO（非阻塞 IO）| 请求后可以做其他事情，数据准备好后复制数据时处于阻塞状态 | 同步 IO |
| AIO（异步 IO）| 请求数据后内核返回回调函数，数据准备好后会发送过来，不用自己复制读取 | 异步 IO |

## Q2. IO 多路复用

**定义：** 一个线程可以监听处理多个客户端的 IO 请求

Linux 提供了三种调用方式：**select、poll、epoll**

| 方式 | 底层数据结构 | 文件描述符数量上限 | 工作原理 |
|------|------------|-----------------|---------|
| select | 数组 | 有上限 | 文件描述符集合从用户空间拷贝到内核空间，轮询询问，再返回给用户空间处理 |
| poll | 链表 | 无上限 | 与 select 基本相同，只是底层数据结构改为链表 |
| epoll | 链表 + 红黑树 | 无上限 | 在内核空间创建数据结构，有 IO 请求的文件描述符放入链表返回给用户空间处理 |

**epoll 的优势：** 少了用户空间复制到内核这一步，减少了开销

---

## Q3. epoll 原理详解

**底层：** 链表 + 红黑树，直接在内核空间创建数据结构

**三个主要方法：**
- `epoll_create()`：创建 epoll 对象，包含红黑树和链表
- `epoll_ctl()`：增、删、改要监听的文件描述符
- `epoll_wait()`：检测就绪的文件描述符，将就绪的放入链表里

**两种触发机制：**
- **水平触发（LT）**：内核检测到就绪的文件描述符，如果一直不处理 IO 请求，内核会**持续通知**
- **边沿触发（ET）**：内核检测到就绪文件描述符，**通知一次**，不处理也不再通知

## Q4. Java NIO

**NIO：** 同步非阻塞 IO

**三个核心组件：**

| 组件 | 说明 |
|------|------|
| Buffer（缓冲区）| 数据的写入写出都是对缓冲区操作 |
| Channel（通道）| 全双工双向通道（流是单向的），是数据输入输出管道 |
| Selector（多路复用器）| 不断轮询注册其上的 Channel，如有 Channel 接入新 TCP 连接，该 Channel 处于就绪状态，通过 SelectionKey 获取就绪 Channel 集合，进行后续 IO 操作 |
