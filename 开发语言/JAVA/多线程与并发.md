# 多线程与并发面试题

## Q1. 多线程基础

**进程 vs 线程：**
- 进程是操作系统资源调度的基本单位；线程是处理器任务调度的基本单位
- 一个进程可以包含多个线程
- 线程共享进程内的资源（堆、方法区），但线程有独立的程序计数器、本地栈、局部变量

**多线程优缺点：**
- 优点：提高 CPU 利用率，提升程序响应速度，线程的创建切换开销低
- 缺点：可能产生死锁；频繁上下文切换也是资源的浪费

---

## Q2. 线程的创建方式

**四种方式：Thread、Runnable、Callable、线程池**

1. **Thread 类**：继承 Thread 类，重写 `run()` 作为线程执行体，创建子类实例，调用 `.start()` 启动

2. **Runnable 接口**：实现 Runnable 接口，实现 `run()` 方法，创建实现类实例作为参数创建 Thread 对象，`.start()` 启动

3. **Callable 接口**：实现 `call()` 方法，创建实现类实例 → FutureTask 对象 → Thread 对象，`.start()` 启动，`FutureTask.get()` 可获取返回值

4. **线程池**：实现 Runnable 的 `run()` 方法，通过 Executors 创建线程池对象，调用 `executor()` 方法执行

---

## Q3. 线程的状态

线程的状态分为 6 种：

| 状态 | 说明 |
|------|------|
| 创建 | 线程已 new 出来，但还没有 start |
| 运行 | 已 start，被 JVM 运行或处于就绪状态 |
| 阻塞 | 等待获取监视器锁 |
| 等待 | 等待其他线程的中断和通知才能返回运行状态 |
| 超时等待 | 超过一定时间后可以自行返回运行状态 |
| 销毁 | 线程运行完成 |

---

## Q4. wait() 和 sleep() 的区别

| 特性 | wait() | sleep() |
|------|--------|---------|
| 所属类 | Object 实例方法 | Thread 静态方法 |
| 是否释放锁 | 释放锁 | 不释放锁 |
| 状态 | 进入等待状态 | 进入超时等待状态 |
| 唤醒方式 | 需要 notify/notifyAll | 一定时间后自行唤醒 |

两者都可以暂停线程。

---

## Q5. 保证线程安全

**引起线程安全的三个原因：**
- **原子性**：一个或多个操作不可被分割，线程切换可能破坏原子性
- **可见性**：一个线程对共享变量改变，其他线程可以实时可见
- **有序性**：编译优化可能打乱指令顺序破坏有序性

**解决方案：**
- `synchronized`：可以保证原子性、可见性、有序性
- `volatile`：保证可见性、有序性（禁止指令重排）
- `Lock` 和 `Atomic` 开头的类：保证原子性

---

## Q6. 线程同步方式

**两种锁：synchronized 关键字和 Lock 接口**

**synchronized：** 依靠 CAS + 排他锁
- 加在普通方法上，锁是对象实例（`this`）
- 加在静态方法上，锁是类的 class 对象
- 加在代码块上，需要在关键字后的小括号中显示指定锁对象

**Lock：** 依靠 CAS + AQS（线程同步队列）实现，更加灵活
- 提供排他锁和乐观锁等多种形式
- 支持响应中断、超时机制、非阻塞方式获取锁
- 支持多个条件变量

---

## Q7. 线程通信方式

线程通信的前提是线程同步，两种方式：**Monitor、Condition**

**Monitor（监视器）：** 配合 synchronized 使用
- 调用 `Object.wait()`、`notify()`、`notifyAll()` 进行线程通信

**Condition：** 依赖 Lock 对象创建
- 调用 `await()`、`signal()`、`signalAll()` 进行线程通信

---

## Q8. synchronized 和 Lock 的区别

| 特性 | synchronized | Lock |
|------|-------------|------|
| 释放锁 | 自动释放（执行完或遇到异常）| 需要手动释放 |
| 中断 | 不支持 | 支持中断获取锁 |
| 超时 | 不支持 | 支持超时机制 |
| 非阻塞 | 不支持 | 支持非阻塞获取锁 |
| 条件队列 | 只能通过 Monitor | 可以创建多个 Condition |
| 是否成功加锁 | 不知道 | 可通过 tryLock() 知道 |

---

## Q9. synchronized 的用法和原理

**作用位置：** 实例方法、静态方法、代码块

**原理：** 底层通过 Java 对象头存储锁信息，支持锁升级：
- 无锁 → 偏向锁 → 轻量级锁 → 重量级锁

保证同一时刻只有一个线程执行该代码，保证线程安全。

---

## Q10. AQS（队列同步器）

- AQS（AbstractQueuedSynchronizer）是构建锁的基本框架，Lock 实现类基于 AQS 实现
- 基于**模板方法模式**设计，实现锁需要继承并重写 AQS 中指定的方法
- AQS 中定义了一个 **FIFO 的同步队列**实现线程同步
- 定义了同步状态记录锁信息
- 模板方法：独占式获取/释放同步状态；共享式获取/释放同步状态

---

## Q11. 线程池

**重要参数：** 核心线程数、等待队列、最大线程数、拒绝策略、空闲线程存活时间

**优点：** 线程复用，减少线程创建销毁的开销，提高效率

**生命周期状态：** `Running(-1)` → `Shutdown(0)` → `STOP(1)` → `Tidying(2)` → `Terminated(3)`（状态只能从小到大迁移，不可逆）

> **注意：** JDK 提供的 Executors 创建线程池的等待队列是**无界的**，容易造成内存溢出（OOM），**不建议使用**

---

## Q12. volatile 的用法和原理

**作用：** 保证变量**可见性**和**有序性**
- 由**内存屏障**禁止执行器的并行级指令重排，保证有序性
- 由 **CPU 缓存一致性协议（MESI 协议）** 保证可见性

---

## Q13. ThreadLocal

- ThreadLocal 是线程变量，每个线程对 ThreadLocal 变量维持一个副本，隔离了多线程对共享变量的竞争
- 每个 Thread 对象都可以通过 ThreadLocal 创建 ThreadLocalMap 对象，以 ThreadLocal 对象为 key，通过 `get()` 和 `set()` 方法存取

> **注意：** ThreadLocal 使用完必须 `remove()`，避免内存泄露。线程池中的线程不会立即回收，其中的 ThreadLocal 不会被回收，需要手动 remove

---

## Q14. JUC（java.util.concurrent）

**五个主要工具：**

1. **原子类（atomic 子包）：** 原子更新基本数据类型、引用类型、属性、数组

2. **锁（Lock 接口）：** 与 synchronized 类似，但更灵活：中断获取锁、超时获取锁、非阻塞获取锁

3. **线程池：**
   - `ThreadPoolExecutor`（普通线程池）
   - `ScheduledThreadPoolExecutor`（可定时任务）
   - Executors 提供的线程池等待队列无界，慎重使用

4. **并发容器（三类）：**
   - 降低锁粒度：`ConcurrentHashMap`
   - 写时复制技术：`CopyOnWriteArrayList`
   - Lock 实现的阻塞队列

5. **并发工具：**
   - `Semaphore`：控制并发量
   - `CyclicBarrier`：让一组线程都到达屏障，屏障才打开放行
   - `CountDownLatch`：允许一个或多个线程等待其他线程执行完成

---

## Q15. Java 中哪些地方使用了 CAS

**CAS（Compare and Swap）：比较并交换**

三个典型场景：

1. **原子类：** `atomic` 类下自增方法通过 CAS 对值进行修改，修改失败则不断 CAS 尝试直到成功

2. **AQS：** Lock 锁基于 AQS 实现加锁解锁，维持变量 `state` 代表锁是否被获取，多个线程通过 CAS 竞争修改这个变量来竞争锁；向同步队列尾部追加节点也是通过 CAS 尝试（失败则自旋反复尝试）

3. **并发容器：** 以 `ConcurrentHashMap` 为例，CAS 方式对数据进行初始化，CAS 方式将初始化好的节点设置到指定槽的首位

---

## Q16. Java 内存模型（JMM）

JMM 定义了线程和主内存的抽象关系，屏蔽各种硬件和操作系统的内存访问差异：
- 程序中的变量存储在主内存中
- 每个线程拥有自己的工作内存并存放变量的拷贝
- 线程读写自己的工作内存，通过主内存与其他线程进行变量交互

**主要目的：** 解决多线程通过共享内存进行通信时，存在的本地内存数据不一致、编译器指令重排、处理器乱序执行等问题，简化多线程编程，增强程序可移植性

---

## Q17. synchronized 和 ReentrantLock 的区别

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 类型 | 关键字 | 类（java.util.concurrent.locks）|
| 加锁/解锁 | 自动（进入/退出同步块自动释放）| 手动（必须显式 unlock()）|
| 底层实现 | 对象头监视器（Monitor）| AQS（抽象同步队列）|
| 公平锁 | 不支持 | 支持（`new ReentrantLock(true)`）|
| 中断等待 | 不支持 | 支持（`lockInterruptibly()`）|
| 超时获取 | 不支持 | 支持（`tryLock(timeout)`）|
| 条件队列 | 单个（wait/notify）| 多个（`newCondition()`）|
| 作用范围 | 静态方法、实例方法、代码块 | 代码块 |

---

## Q18. Java 中有哪些锁

**按乐观/悲观分：**
- **乐观锁（CAS）**：先操作，提交时检查冲突（AtomicInteger 等原子类）
- **悲观锁**：先加锁再操作（synchronized、ReentrantLock）

**按公平性分：**
- **公平锁**：按等待顺序获取锁
- **非公平锁**：随机竞争，效率更高（默认）

**按独占/共享分：**
- **独占锁（排他锁）**：同一时刻只有一个线程持有（synchronized、ReentrantLock）
- **共享锁**：多个线程可同时读（ReentrantReadWriteLock 的读锁）

**其他类型：**
- **可重入锁**：同一线程可多次获取（synchronized、ReentrantLock 都是可重入的）
- **自旋锁**：获取不到锁时循环尝试，不挂起线程
- **分段锁**：ConcurrentHashMap 对每个槽位的头节点单独加锁
- **偏向锁 / 轻量级锁 / 重量级锁**：synchronized 的锁升级优化技术

---

## Q19. 线程池 execute() 和 submit() 的区别

| 特性 | execute() | submit() |
|------|-----------|----------|
| 返回值 | 无返回值（void）| 返回 Future 对象 |
| 任务类型 | 只接受 Runnable | 接受 Runnable 和 Callable |
| 异常处理 | 直接抛出 | 异常存储在 Future 中，调用 get() 时抛出 |
| 获取结果 | 不能 | `future.get()` 可以获取执行结果 |

## Q20. 线程池如何关闭

| 方法 | 行为 | 状态 |
|------|------|------|
| `shutdown()` | 不接受新任务提交，正在执行的和等待队列的任务执行完成后关闭 | 进入 SHUTDOWN 状态 |
| `shutdownNow()` | 不接受新任务，等待队列任务也不执行，正在执行的任务发送中断信号（interrupt）| 进入 STOP 状态 |

## Q21. JVM 三色标记法

三色标记法用于并发垃圾回收（CMS 和 G1 使用），将对象分为三类：

| 颜色 | 含义 |
|------|------|
| **白色** | 未被垃圾回收器扫描过的对象；任意对象在初始状态下均为白色；回收阶段仍为白色的对象会被回收 |
| **灰色** | 正在被回收器处理的对象，自身已被扫描，但其引用的子对象还未扫描完 |
| **黑色** | 已被垃圾回收器扫描并确认存活的对象，自身及其所有子对象都已被处理完 |

**问题：** 并发标记期间用户线程可能修改引用，导致"漏标"（黑色对象新引用了白色对象）。
**解决：** 增量更新（CMS）或原始快照 SATB（G1）。
