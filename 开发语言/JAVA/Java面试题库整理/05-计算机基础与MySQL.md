## 线程、进程、协程

### 线程和进程

- 进程更重，资源隔离更强。
- 线程更轻，调度更高效。
- 进程崩溃一般不影响其他进程。
- 线程崩溃可能带崩整个进程。

### 线程和协程

- 线程由操作系统管理，创建切换开销更高。
- 协程通常由用户态调度，开销更小。
- 线程能利用多核并行，协程更偏并发协作。

## 死锁

- 定义：多个执行单元相互等待资源，无法推进。
- 四个必要条件：
- 互斥
- 请求并保持
- 不可剥夺
- 循环等待

## 虚拟内存与物理内存

- 物理内存是真实硬件内存。
- 虚拟内存是逻辑地址空间，可映射到物理内存或磁盘。
- 目的在于扩大可用地址空间、提高内存利用率。

## MySQL 事务隔离级别

- 读未提交
- 读已提交：解决脏读
- 可重复读：MySQL 默认，解决脏读和不可重复读
- 串行化：隔离最强，也最影响并发

## InnoDB 和 MyISAM

- InnoDB 支持事务、外键、行锁、崩溃恢复。
- MyISAM 不支持事务，主要是表锁，查询场景历史上较多。
- 现在绝大多数业务表优先 InnoDB。

## MySQL 索引

### 索引的好处

- 提升查询效率。
- 提升排序和分组效率。

### 索引的代价

- 占用存储空间。
- 增删改需要维护索引。

### 常见索引

- 普通索引
- 唯一索引
- 主键索引
- 联合索引
- 全文索引

## B 树与 B+ 树

- 都是多路平衡查找树。
- B+ 树只有叶子节点保存完整数据。
- B+ 树叶子节点天然适合范围查询和排序。
- MySQL 大多采用 B+ 树而不是红黑树或普通二叉树。

## 数据库为什么不用红黑树或哈希做主流索引

- 红黑树层级深，磁盘 IO 次数更高。
- 哈希不适合范围查询和排序。
- B+ 树能兼顾磁盘 IO、范围扫描、稳定查找性能。

## 聚簇索引与非聚簇索引

- 聚簇索引：索引和数据在一起，叶子节点直接存数据。
- 非聚簇索引：叶子节点存主键或地址，需要回表。

## 主从同步

- 主库写 binlog。
- 从库拉取并写入 relay log。
- 从库重放 relay log，保持数据一致。

## MVCC

- 多版本并发控制。
- 目标是读写不冲突，提高并发能力。
- 核心通常依赖：
