# 计算机网络面试题

## Q1. Cookie 和 Session 的联系与区别

**联系：** Cookie 中存储的是 SessionId

**区别：**

| 特性 | Cookie | Session |
|------|--------|---------|
| 存储位置 | 浏览器端 | 服务器端 |
| 安全性 | 较差（客户端可修改）| 较好 |
| 存储内容 | 只能存 ASCII 码字符串 | 可以存储任何数据类型 |
| 有效期 | 可设置为长时间（如"记住我"）| 客户端关闭或 Session 超时则失效 |
| 存储大小 | 单个 Cookie 不超过 4KB | 远大于 Cookie |

## Q2. HTTP、TCP、Socket 三者的关系

- **IP 协议**：连接网络，负责寻址
- **TCP 协议**：在 IP 协议基础上，提供可靠的数据传输机制（连接、校验、重传）
- **HTTP 协议**：应用层协议，Web 服务器传输超文本到浏览器；HTTP 使用 TCP 协议来传输数据
- **Socket（套接字）**：对 TCP/IP 协议的封装，是一个调用接口（门面模式），通过 Socket 使用 TCP/IP 协议

## Q3. UDP 和 TCP 的区别

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 有连接（三次握手）| 无连接 |
| 可靠性 | 可靠（确认、重传）| 不可靠 |
| 传输方式 | 字节流 | 数据报文 |
| 通信模式 | 一对一 | 一对多、多对多 |
| 首部开销 | 通常 20 字节 | 8 字节 |
| 速度 | 较慢 | 较快 |
| 适用场景 | 文件传输、收发邮件 | 视频、语音、广播 |

## Q4. TCP 的拥塞控制

拥塞控制是为了避免数据填满网络造成拥堵，包含四个算法：

| 算法 | 说明 |
|------|------|
| **慢启动** | 拥塞窗口 cwnd 从 1 开始，每个 RTT 内指数增长 |
| **拥塞避免** | cwnd 达到慢启动阈值 ssthresh 后，改为每次 +1 线性增长 |
| **快重传** | 收到 3 个重复 ACK，立即重传，不等超时 |
| **快恢复** | 收到 3 个重复 ACK：`ssthresh = cwnd / 2`，`cwnd = ssthresh`，进入拥塞避免 |

**超时重传：** `ssthresh = cwnd / 2`，`cwnd = 1`，重新进入慢启动

## Q5. TCP 如何保证可靠性

1. **建立连接（三次握手）**：确保双方都有可靠的收发能力
2. **序列号 + 确认应答机制**：接收方对每个数据包返回 ACK，发送方知道哪些已送达
3. **校验和**：检测数据在传输过程中是否发生错误
4. **超时重传**：发送方等待 ACK 超时则重传
5. **流量控制**：滑动窗口机制，接收方通告窗口大小，避免接收方被压垮
6. **拥塞控制**：避免网络拥堵

## Q6. HTTP 和 HTTPS 的区别

| 特性 | HTTP | HTTPS |
|------|------|-------|
| 传输方式 | 明文传输 | 加密传输（TLS/SSL）|
| 端口 | 80 | 443 |
| 安全性 | 不安全 | 安全（防窃听、防篡改）|
| 速度 | 快 | 略慢（握手有开销）|

**HTTPS 数据安全传输流程：**
1. 服务器向 CA（证书权威机构）申请**数字证书**
2. 服务器将数字证书（含公钥）发送给客户端
3. 客户端验证证书合法性，生成**对称密钥**，用**公钥加密**对称密钥发送给服务器
4. 服务器用私钥解密，得到对称密钥
5. 之后双方用对称密钥加密传输数据

## Q7. HTTP 不同版本的区别

| 版本 | 特性 |
|------|------|
| **HTTP/1.0** | 短连接，每次请求都需要建立 TCP 连接 |
| **HTTP/1.1** | 长连接（持久连接），管道机制（一个连接可以同时发送多个请求，但必须顺序响应）|
| **HTTP/2.0** | 二进制传输，完全多路复用（不按顺序响应），报头压缩，服务端推送 |
| **HTTP/3.0** | 传输层基于 UDP，使用 QUIC 协议保证可靠性，TLS 握手加密，减少握手延迟 |

## Q9. DNS 解析过程

1. 查找**浏览器缓存**，寻找域名对应的 IP
2. 找不到，请求**本地 DNS 服务器**（通常是运营商提供的）
3. 本地 DNS 向**根域名服务器**请求
4. 根域名服务器返回**顶级域名服务器**地址（如 `.com` 对应的服务器）
5. 本地 DNS 请求顶级域名服务器，得到**权限域名服务器**地址
6. 本地 DNS 请求权限域名服务器，得到最终 IP 地址
7. 本地 DNS 将 IP 返回给客户端并**缓存**

## Q11. TCP 三次握手

```
客户端                              服务端
  |  ──── SYN (seq=x) ────────→   |  服务端: SYN-RCVD
  |  ←── SYN+ACK (seq=y,ack=x+1) |  客户端: SYN-SENT → ESTABLISHED
  |  ──── ACK (ack=y+1) ──────→   |  服务端: ESTABLISHED
```

**为什么不是两次握手？**
两次握手如果第二次握手丢失，服务端认为连接已建立并占用端口，客户端实际没有建立连接，造成资源浪费。三次握手通过双方返回的确认序列号，验证对方具有可靠的收发能力。

**最后一次握手可以携带数据吗？**
可以。前两次不可以（防止恶意攻击伪造请求时让服务端处理大量数据），第三次握手时连接已建立可以携带数据。

**SYN 洪泛攻击：**
恶意攻击者伪造大量 IP 向服务端请求建立连接，服务端半连接队列被占满，真正的请求无法建立连接。
**解决：** SYN Cookie（将源 IP、端口打包为 cookie 作为序列号）；SYN Proxy 防火墙代答。

## Q12. TCP 四次挥手

```
主动方                              被动方
  |  ──── FIN (seq=u) ────────→   |  被动方: CLOSE-WAIT
  |  ←── ACK (ack=u+1) ──────── |  主动方: FIN-WAIT-2（只收不发）
  |  ←── FIN (seq=v) ──────────  |  被动方: LAST-ACK
  |  ──── ACK (ack=v+1) ──────→  |  主动方: TIME-WAIT → CLOSED（等待 2MSL）
```

**为什么 TIME-WAIT 需要等待 2MSL？**
1. **保证最后的 ACK 能到达服务端**：如果 ACK 丢失，服务端会超时重发 FIN，主动方在 2MSL 内可以重新发送 ACK
2. **让当前连接的报文全部消失**：MSL 是报文在网络中存活的最长时间，2MSL 确保本轮连接的所有报文消失，避免影响下一轮连接
