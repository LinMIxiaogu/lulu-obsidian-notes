# MySQL 深入面试题

## Q1. MySQL 三大范式

| 范式 | 要求 |
|------|------|
| 第一范式（1NF）| 数据表中每个字段不可以再拆分（原子性）|
| 第二范式（2NF）| 在 1NF 基础上，非主键列完全依赖于主键，而不能只依赖于主键的一部分（消除部分依赖）|
| 第三范式（3NF）| 在 2NF 基础上，非主键只依赖于主键，而不依赖于其他非主键（消除传递依赖）|

## Q2. 一条 SQL 查询语句的执行顺序

```sql
FROM → JOIN → ON → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

**说明：**
1. `FROM`：确定数据来源
2. `JOIN / ON`：连接表，过滤连接条件
3. `WHERE`：过滤行（不能使用聚合函数结果）
4. `GROUP BY`：分组
5. `HAVING`：过滤分组（可以使用聚合函数）
6. `SELECT`：选取列（别名在此定义，HAVING 不能用别名）
7. `DISTINCT`：去重
8. `ORDER BY`：排序（可以使用 SELECT 中的别名）
9. `LIMIT`：限制行数

## Q3. MySQL 常见日志

**Server 层日志：**
- **慢查询日志**：记录执行时间超过 `long_query_time` 的 SQL
- **binlog（二进制日志）**：记录所有更改数据的 SQL 操作，用于主从复制和数据恢复

**InnoDB 存储引擎层日志：**
- **redo log（重做日志）**：记录数据页的物理修改，用于崩溃恢复，保证持久性（D）
- **undo log（回滚日志）**：记录数据修改前的状态，用于事务回滚和 MVCC，保证原子性（A）

---

## Q4. redo log 和 binlog 的区别

| 特性   | redo log                 | binlog               |
| ---- | ------------------------ | -------------------- |
| 归属   | InnoDB 存储引擎层             | MySQL Server 层       |
| 内容   | 记录数据页的物理修改（"第X页第Y字节改为Z"） | 记录逻辑操作（SQL 语句或行变化）   |
| 写入方式 | 循环写（大小固定，会覆盖旧日志）         | 追加写（不会覆盖，一个写满了就新建一个） |
| 用途   | 崩溃恢复（crash-safe）         | 主从复制、全量数据恢复          |

**两阶段提交（保证两个日志一致性）：**
1. InnoDB 写 redo log，标记为 **prepare** 状态
2. Server 层写 binlog
3. InnoDB 将 redo log 标记为 **commit** 状态

崩溃恢复时：有 commit 标记则提交；否则判断 binlog 是否完整决定提交还是回滚

## Q5. MySQL 索引分类

**按数据结构：**
- B+ 树索引（最常用）
- Hash 索引（Memory 引擎，不支持范围查找）
- 全文索引

**按物理存储：**
- **聚簇索引（主键索引）**：叶子节点存储完整行数据，InnoDB 必须有主键
- **非聚簇索引（辅助索引）**：叶子节点存储主键值，查询需要回表

**按字段特性：**
- 主键索引、唯一索引、普通索引、组合索引（联合索引）、全文索引

---

## Q6. 创建索引的注意点

**适合创建索引的字段：**
- 查询时频繁使用的字段（`WHERE`、`ORDER BY`、`JOIN ON`）
- 区分度高的字段（如性别不适合，用户 ID 适合）
- 组合索引中，区分度高的字段放前面

**不适合创建索引的情况：**
- 不频繁更新的字段（更新时需要维护索引）
- 不建议使用无序值做索引（如 UUID、身份证号——可以，但不如自增 ID）
- 索引个数要适量，太多影响写入性能

## Q7. 索引失效的场景

1. `OR` 两边一个无索引，则不走索引
2. 字符串查询未使用引号（隐式类型转换导致失效）
3. 组合索引**不满足最左前缀原则**
4. 索引列上使用内置函数（如 `WHERE YEAR(create_time) = 2023`）
5. 索引列进行四则运算（如 `WHERE id + 1 = 10`）
6. `!=`、`<>`、`NOT IN` 可能失效（取决于数据分布）
7. `IS NULL`、`IS NOT NULL` 可能失效
8. `%` 开头的 LIKE（如 `LIKE '%abc'`）导致索引失效
9. **优化器**认为全表扫描比索引更快时（如数据量少，或返回行数占比过高）

## Q8. 回表和覆盖索引

**回表：**
- InnoDB 辅助索引的叶子节点存储的是**主键值**
- 通过辅助索引找到主键值后，还需要再到**主键索引树**中查到完整行数据
- 这个额外的查询过程叫**回表**，比基于主键索引的查询多扫描一棵索引树

**覆盖索引：**
- 在辅助索引的叶子节点中就能直接找到 `SELECT` 需要的所有字段数据
- **不需要回表**，性能更好
- `EXPLAIN` 中 `Extra` 列显示 `Using index` 表示使用了覆盖索引

---

## Q9. 索引下推优化（ICP）

**不使用索引下推：**
存储引擎通过索引检索数据 → 回表查询完整行 → 返回给 Server 层 → Server 层进行条件过滤

**使用索引下推（Index Condition Pushdown）：**
- 将某些过滤判断条件（如果这些条件也是索引列）**下推**给存储引擎处理
- 存储引擎在索引遍历阶段就过滤掉不满足条件的行
- 减少回表次数，减少返回给 Server 层的数据量

**示例：** 组合索引 `(name, age)`，查询 `WHERE name LIKE '张%' AND age = 18`
- 没有 ICP：先用 name 查到所有姓张的，回表，Server 层再过滤 age
- 有 ICP：在索引层同时判断 age=18，减少回表

## Q10. MySQL 锁类型

**按粒度：**
- **表锁**：锁整张表，开销小，并发低（MyISAM 只支持表锁）
- **行锁**：锁一行或多行，开销大，并发高（InnoDB 支持）

**行锁的模式（InnoDB）：**
| 锁类型 | 场景 |
|--------|------|
| 记录锁（Record Lock）| 唯一索引等值查询到一条记录 |
| 间隙锁（Gap Lock）| 等值查询或范围查询没有命中索引，锁住索引间隙（左开右开）|
| 临键锁（Next-Key Lock）| 范围查询命中部分索引记录（左开右闭），是 Record Lock + Gap Lock 的组合 |
| 插入意向锁 | 插入数据前判断位置上是否有其他事务的间隙锁，有则等待 |

**其他锁：**
- **共享锁（S Lock）**：读锁，多个事务可以同时持有
- **排他锁（X Lock）**：写锁，独占
- **意向锁**：表级锁，表示某个事务在行级加了 S 或 X 锁，避免加表锁时遍历所有行

---

## Q11. MySQL 意向锁

**作用：** 解决表锁和行锁共存的效率问题

**问题场景：** 事务 B 申请表级写锁时，需要确认没有行级锁。如果没有意向锁，B 需要遍历所有行确认是否有行锁。

**解决：**
- 事务 A 申请行锁（写锁）之前，先在**表级**加一个**意向排他锁（IX）**
- 事务 B 申请表锁时，发现意向排他锁，直接判断冲突，不需要遍历所有行

> 注意：**意向锁**（表级）和**插入意向锁**（行级）是两个不同的概念

## Q12. DELETE、DROP、TRUNCATE 的区别

| 操作 | 说明 | 事务 | 速度 |
|------|------|------|------|
| `DELETE` | 删除表中符合条件的记录行，可加 WHERE | 属于 DML，可回滚 | 慢（逐行删除，记录 undo log）|
| `TRUNCATE` | 删除表中全部数据，保留表结构 | 属于 DDL，不可回滚 | 快（直接删除数据页）|
| `DROP` | 删除整张表（包括结构和数据）| 属于 DDL，不可回滚 | 快 |

## Q13. 事务 ACID 如何保证

| 特性 | 保证机制 |
|------|---------|
| **A 原子性** | undo log（回滚日志）：事务失败时利用 undo log 回滚到事务开始前的状态 |
| **I 隔离性** | MVCC（多版本并发控制）+ 锁机制 |
| **D 持久性** | redo log（重做日志）：事务提交后 redo log 持久化，崩溃后可恢复 |
| **C 一致性** | 由 A、I、D 共同保证；保证了 AID 就保证了 C |

## Q14. 脏读、不可重复读、幻读

| 问题 | 定义 | 示例 |
|------|------|------|
| **脏读** | 读取了另一个事务未提交的数据，该事务最终回滚 | 事务 A 修改了 name，事务 B 读取了 name，事务 A 回滚 |
| **不可重复读** | 同一事务内多次读取同一行，结果不同 | 事务 A 中两次读取 name，事务 B 在其中修改了 name |
| **幻读** | 同一事务内两次查询，第二次多出了行（INSERT 导致）| 事务 A 两次查询 age=18 的用户，事务 B 在其中插入了一条 age=18 的数据 |

**如何解决：**
- **快照读（普通 SELECT）**：生成 ReadView 快照，MVCC 从快照读取，天然避免脏读、不可重复读和幻读
- **当前读（SELECT FOR UPDATE 等）**：通过间隙锁/临键锁锁住区间，防止其他事务插入数据

## Q15. MVCC 详解

**关键点：** 隐式字段 + undo log 版本链 + ReadView

**隐式字段（每行数据都有）：**
- `trx_id`：最近更新该行的事务 ID
- `roll_pointer`：回滚指针，指向 undo log 中的上一个版本

**undo log 版本链：** 每次修改都会在 undo log 中记录旧版本，通过 roll_pointer 串成版本链

**ReadView（读视图）：**
- 记录创建时活跃的事务 ID 列表和事务范围
- 通过比较事务 ID 和 ReadView，确定从版本链中读取哪个版本

**事务隔离级别对应 ReadView 创建时机：**
- **读已提交**：每次 SELECT 前都生成新的 ReadView
- **可重复读**：只在第一次 SELECT 时生成 ReadView，后续复用

## Q16. 慢 SQL 优化

**避免不必要的列：** 避免 `SELECT *`，只查询需要的列

**索引优化：**
- 利用覆盖索引（避免回表）
- 避免 OR、不等于、列上的函数运算（导致索引失效）
- 适当使用联合索引、前缀索引

**分页优化（深分页）：**
- 延迟关联：先在索引列上完成分页，再回表获取需要的列
- 书签方式：子查询找到 LIMIT 第一个参数的主键值，根据主键过滤并 LIMIT

**JOIN 连接优化：**
- 用 JOIN 替代子查询（子查询会创建临时表，性能更差）
- 小表驱动大表：驱动表小，连接大表的次数少

**排序优化：** 利用索引扫描做排序，避免 filesort

**UNION 优化：** 手动将 WHERE 和 LIMIT 下推到 UNION 的子查询中

## Q17. EXPLAIN 关键字段

| 字段 | 说明 |
|------|------|
| `id` | 每个 SELECT 一个 id，id 越大越先执行 |
| `select_type` | 查询类型：SIMPLE（简单）、PRIMARY（主查询）、UNION、SUBQUERY 等 |
| `table` | 访问的表 |
| `type` | **访问类型（重要）**，性能从好到差：`system > const > eq_ref > ref > range > index > ALL` |
| `possible_keys` | 可能用到的索引 |
| `key` | 实际使用的索引 |
| `key_len` | 实际使用索引的长度 |
| `rows` | 预估扫描的行数 |
| `Extra` | 额外信息：`Using index`（覆盖索引）、`Using where`（回表过滤）、`Using filesort`（文件排序，需优化）、`Using temporary`（使用临时表，需优化）|

**type 重点：**
- `const`：命中主键/唯一索引
- `ref`：非主键非唯一索引等值扫描
- `range`：范围扫描（BETWEEN、IN、>、<）
- `ALL`：全表扫描，需要优化

## Q18. InnoDB 和 MyISAM 的区别（详细）

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | 支持 | 不支持 |
| 外键 | 支持 | 不支持 |
| 锁级别 | 行锁 | 表锁 |
| 索引类型 | 聚簇索引（主键索引存数据）| 非聚簇索引（索引和数据分开）|
| 主键要求 | 必须有主键，没有会自动创建 | 不要求 |
| COUNT(*) | 需要遍历 | 直接返回（存储了行数），但加 WHERE 也要遍历 |
| 崩溃修复 | 支持（redo log）| 不支持 |
| 全文索引 | MySQL 5.6+ 支持 | 默认支持 |
| 适用场景 | 需要事务、高并发写入 | 以查询为主，读多写少 |

## Q19. MySQL 主从复制流程

1. 主服务器（master）上的数据更改写入 **binlog（二进制日志）**
2. 主服务器创建一个 **dump 线程**，将 binlog 推送给从服务器
3. 从服务器（slave）上的 **IO 线程**接收 binlog，写入本地的 **relay log（中继日志）**
4. 从服务器上的 **SQL 线程**读取 relay log，重放其中的 SQL 操作，完成数据同步
