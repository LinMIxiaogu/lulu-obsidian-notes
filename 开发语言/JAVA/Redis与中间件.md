# Redis 与中间件面试题

## Q1. Redis 概览

Redis 是基于键值对的 NoSQL（非关系型数据库）

**特点：**
- 数据都在**内存**中，存取速度快，适合作为缓存
- 可以给数据设置**过期时间**，适用于定时任务
- 高速缓存特性常用于**热点数据**
- 自带 SortedSet 可以为热点数据排序，适合做**排行榜**
- 适合实现点赞、关注、排行等对性能要求极高的需求
- 适合存储频繁使用的信息（登录信息、用户状态）
- 可以实现**分布式锁**、发布订阅（作为消息中间件）、Lua 脚本等

## Q2. Redis 数据类型和底层实现

**五大数据结构：**

| 数据类型   | 特性                             | 底层实现               |
| ------ | ------------------------------ | ------------------ |
| String | 最基本的数据结构                       | 动态字符串（支持扩容）        |
| List   | 存储数据有序可重复                      | 双向链表 / 压缩列表        |
| Hash   | 键值对                            | 压缩列表 + Hash        |
| Set    | 存储数据无序不可重复                     | 整型数组 + Hash        |
| Zset   | 存储数据有序不可重复（每个数据有 score 属性用来排序） | ziplist + skiplist |

**其他数据结构：** Bitmap（位图）、Geo（地理信息）

## Q3. Redis 单线程架构

Redis 采用**单线程 + IO 多路复用技术**：
- Redis 的读写操作和网络 IO 都是单线程的
- 其他的持久化操作、异步删除等依赖其他线程执行

**单线程优势：**
- 消除了线程切换和竞争锁的开销
- 通过 IO 多路复用技术，一个单线程处理多个 IO 请求，高吞吐量

**Redis 快的三个原因：** 在内存、单线程、IO 复用技术

## Q4. Redis 持久化策略

### RDB（快照持久化）
- 间隔性地将数据生成快照存储到磁盘上
- Redis **默认**的持久化策略
- **优点：** RDB 文件小，恢复数据快
- **缺点：** 不能做到实时持久化，可能丢失最后一次快照的修改

### AOF（日志追加）
- 将所有命令追加写入日志中，Redis 启动时完整执行文件中的命令恢复数据
- **优点：** 不会丢失数据，安全性高
- **缺点：** 文件大，恢复速度慢

## Q5. Redis 主从同步机制

**两种同步方式：**

**全量同步（第一次连接）：**
1. 从机第一次连接到主机，主机校验从机 ID 发现是第一次
2. 主机将所有数据生成 RDB 快照发送给从机
3. 快照发送完后，将发送期间缓存中存储的写命令也发送给从机
4. 从机接收后存储到磁盘再加载到内存，执行传过来的缓存写命令

**增量同步（非第一次连接）：**
1. 主机检验从机 ID，发现不是第一次连接
2. 根据从机的**数据偏移量**，从自己的缓存中将新增的数据发送给从机
3. 从机进行同步

## Q6. Redis 缓存淘汰策略

**三种策略：**

1. **惰性删除：** 当访问数据时，检查是否过期，过期则删除（对 CPU 友好）

2. **定期删除：** 每隔一段时间，进行贪心删除过期的 key（对内存友好）
   - 贪心策略：每次随机从过期 key 中取 20 个，删除其中的过期 key
   - 如果需要删除的过期 key 比例高，则重复上一步

3. **Maxmemory-policy：** 达到最大内存时
   - **LRU（最近最少使用）**：淘汰最长时间没有使用的数据
   - **LFU（使用次数最少）**：淘汰使用次数最少的数据

## Q7. 缓存穿透、击穿、雪崩

| 问题 | 定义 | 解决方案 |
|------|------|---------|
| **穿透** | 客户端请求大量不存在的数据，直接落到数据库层，数据库崩溃 | 布隆过滤器；缓存空值 |
| **击穿** | 热点数据过期的瞬间，大量请求落到数据库层，数据库崩溃 | 设置热点数据不过期；加锁（过期时一个线程访问数据库并在缓存中重建）|
| **雪崩** | 大量数据同时过期，大量请求落到数据库；或缓存故障 | 数据设置随机过期时间；设置热点数据不过期；采用 Redis 集群避免故障不可用 |

---

## Q8. Redis 缓存与数据库双写一致性

**几种方案（后者更优）：**

1. 先更新缓存再更新数据库
2. 先更新数据库再更新缓存
3. 先删除缓存再更新数据库
4. **先更新数据库再删除缓存**（较好，删除缓存失败就重试）
5. **延迟双删**：先删除缓存，再更新数据库，延迟一段时间后再删除缓存

## Q9. Redis 高可用方案

**三种方案：**

### 主从复制
- 主服务器负责读写操作，数据同步给从服务器
- 从服务器只能读，帮主服务器分担压力
- **缺点：** 主服务器挂掉，会丢失部分数据，需要重启，不能自动切换到从服务器

### 哨兵模式（Sentinel）
- 单独一套哨兵系统监视主从服务器
- 主服务器挂掉，自动选出从服务器顶上
- **缺点：** 选取期间无法确定主从，不能进行工作

### 集群模式（Cluster）
- 数据槽，支持高并发并容纳海量数据
- 数据分布在不同的集群，实现分布式存储
- 去中心化控制

## Q10. Redis 实现分布式锁

**背景：** 分布式环境下，多个 server 并发修改一个资源时，Java 自带的锁限于自己的 JRE，在分布式场景下无效

**实现方式（由低到高）：**

| 方式 | 问题 |
|------|------|
| `setnx + expire` | 两条命令非原子操作，可能死锁 |
| `SET ex px nx` | 原子操作，但释放锁可能出错 |
| `SET ex px nx + 校验唯一随机值` 再释放锁 | 较安全，但校验和释放不是原子操作 |
| **Lua 脚本**（包含 setnx 和 expire 两条命令）| Lua 脚本默认原子操作，**推荐** |

**推荐使用 Redisson 框架提供的分布式锁实现**

## Q11. 如何处理 Redis 热 Key

**监控热 Key：**
- 业务经验预估（如大促期间的商品详情页）
- 客户端统计（在客户端代码中统计 key 访问频率）
- 代理层统计（基于代理架构的 Redis 可在代理端统计）
- `Redis MONITOR` 命令监控所有命令，然后分析统计

**处理热 Key：**
1. **在多个 Redis 节点备份**：将热 key 打散到不同服务器（如 `hot_key_1`、`hot_key_2`...），访问时随机选择
2. **本地二级缓存**：将热 key 加载到 JVM 本地缓存（如 Caffeine/Guava），减少 Redis 访问压力

## Q12. Redis 是单线程还是多线程？

**Redis 6.0 之前：** 核心是单线程
- Redis 基于内存操作，CPU 不是瓶颈，瓶颈是内存大小和网络带宽
- 单线程消除了线程切换和竞争锁的开销
- IO 多路复用处理大量并发连接

**Redis 6.0 之后：** 引入多线程
- 多线程用于处理**网络 IO**（读取请求、发送响应），提升网络吞吐量
- 多线程用于处理**缓慢操作**：大 key 的异步删除、无用连接的释放
- **命令的执行依然是单线程**，保证原子性

## Q13. Redis 事务

**三个关键命令：**
- `MULTI`：事务开始
- `EXEC`：事务结束，一次性执行所有命令，返回结果列表
- `DISCARD`：取消事务

**特点：**
- Redis 单线程执行事务，不会被其他线程打断，保证隔离性
- **不支持回滚**：事务中某条命令失败，其他命令仍会执行

**Lua 脚本（更推荐）：**
- Lua 脚本原子执行（整个脚本作为一个原子操作）
- 常驻 Redis 内存，可以复用（使用 `EVALSHA`）
- 多条命令一次性打包，减少网络开销

## Q14. Redis 删除策略

Redis 同时采用**惰性删除 + 定期删除**两种策略配合使用：

| 策略 | 说明 | 优点 |
|------|------|------|
| 惰性删除 | 访问 key 时才检查是否过期，过期则删除 | 对 CPU 友好，减少 CPU 开销 |
| 定期删除 | 每隔一段时间，随机抽取一批 key，删除其中过期的 | 对内存友好 |

**定期删除的贪心策略：**
每次随机从过期 key 中取 20 个，删除其中的过期 key；如果需要删除的过期 key 比例 > 25%，则重复上一步

## Q15. 跳表（Skip List）原理

跳表是 Redis Zset（有序集合）的底层数据结构之一

**基本结构：**
- 底层是一个有序链表
- 在链表上方建立多层索引，每层索引节点随机出现（概率约 1/2）

**查找过程：** 从最高层索引开始，找到比目标小的最大节点，下降一层继续查找，直到找到目标

**时间复杂度：** 平均 O(log n)，与平衡树相近

**跳表 vs 红黑树：**
| 特性 | 跳表 | 红黑树 |
|------|------|--------|
| 实现复杂度 | 简单 | 复杂 |
| 范围查找 | 简单高效（直接遍历底层链表）| 需要中序遍历 |
| 并发控制 | 更容易实现无锁并发 | 复杂 |
| 内存占用 | 略高 | 较低 |

Redis 选择跳表而非红黑树，主要原因：范围查询性能更好，实现更简单
