# 计算机基础与数据库面试题

## Q1. 进程间的通信方式

| 方式 | 说明 |
|------|------|
| 管道 | 本质是内核中的一块缓存区，单向通信，FIFO |
| 共享内存 | 两个进程共享用户空间的一块内存 |
| 消息队列 | 读消息、写消息 |
| 信号 | 某事件发生会通知进程 |
| 信号量 | P（-1）V（+1）操作对信号量操作获取共享资源 |
| 内存映射 | 磁盘文件映射到内存上，修改内存即可修改磁盘数据 |
| 套接字（Socket）| 用于两个主机之间的进程通信 |

## Q2. 线程和进程的区别

| 对比 | 进程 | 线程 |
|------|------|------|
| 定义 | 操作系统分配资源的最小单元 | CPU 调度任务的最小单元 |
| 内存空间 | 有独立的空间 | 共享进程内的堆和方法区（程序计数器、本地方法栈、虚拟机栈独立）|
| 并发性 | 低 | 高 |
| 健壮性 | 崩溃不影响其他进程 | 崩溃会导致进程死掉 |
| 上下文切换 | 开销大 | 开销小 |

## Q4. 死锁定义和发生条件

**死锁：** 两个及两个以上的线程持有自己的资源并请求对方的资源，陷入了永久循环等待状态，没有外力作用下无法离开

**四个必要条件：**
1. **互斥条件**：资源不能被共享
2. **请求并保持**：持有资源并请求其他资源
3. **不可剥夺条件**：已获得的资源不能被强制剥夺
4. **循环等待**：存在一个循环等待链

## Q5. 内存管理

**连续内存管理：**
- 块式管理：只需要小的内存可能会分配大的内存块，造成较多内存浪费

**非连续内存管理：**
- **页式**：按页划分内存，内存碎片较小
- **段式**：按程序逻辑结构划分为段，更有实际意义
- **段页式**：结合以上两种

---

## Q6. 虚拟内存和物理内存的区别

- **物理内存**：实际的内存大小
- **虚拟内存**：一种内存管理技术，是逻辑上的内存空间。程序认为是连续的空间，实际上是逻辑地址映射到物理地址，可能是离散的内存空间和磁盘上的空间，在需要时调入调出，逻辑上扩充了内存大小

## Q7. MySQL 事务隔离级别

事务隔离级别是为了处理：**脏读、不可重复读、幻读**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|----------|------|
| 读未提交 | ✗ | ✗ | ✗ |
| 读已提交 | ✓ | ✗ | ✗ |
| 可重复读（MySQL 默认）| ✓ | ✓ | ✗ |
| 串行化 | ✓ | ✓ | ✓ |

## Q9. MySQL 索引

**优点：** 加快查询记录的速度，加速排序和分组

**缺点：** 占用存储空间，增删改操作时需要动态维护索引

## Q10. B 树和 B+ 树

B 树和 B+ 树都是**多路平衡查找树**（二分查找树扩展为多叉）

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 节点存储内容 | 存储索引和数据 | 只有叶子节点存储数据，其余节点只存索引 |
| 叶子节点连接 | 无 | 叶子节点连成链表，方便范围查找 |
| 查找深度 | 不稳定 | 每次必须查找到叶子节点，查找深度相同，效率稳定 |

MySQL 采用 **B+ 树**，原因：
- B+ 树是多叉树，可以保证树的层数控制在 2~4 层
- 层数代表磁盘 IO 次数，层数少意味着 IO 少，性能高
- 叶子节点链表方便范围查找

## Q11. MySQL 主从同步

主从同步依靠**复制（Replication）**实现：

1. 主服务器（master）更新数据的操作记录到**二进制日志（binlog）**文件中
2. 从服务器将主服务器的二进制日志复制到自己的**中继日志（relay log）**中
3. 从服务器重做复制来的日志中的更新操作，使自己的数据与主服务器保持一致

---

## Q12. 数据库索引底层数据结构

**可用的数据结构：** 二叉树、红黑树、Hash 表、B+ 树

**MySQL 采用 B+ 树的原因：**
- 二叉树在某些情境下会退化为链表
- 红黑树本质还是二叉树，大量数据下层数高，层数代表磁盘 IO 次数，不合适
- B+ 树是多叉树，可以保证树的层数控制在 2~4 层，控制了 IO 次数

**为什么不用哈希表：**
1. 哈希表查询数据需要全部读入内存，数据量大时内存放不下
2. 哈希表可能产生哈希冲突
3. 哈希表不可以顺序和范围查找

## Q15. 数据库 ACID

| 属性 | 说明 |
|------|------|
| A（原子性）| 事务由多个操作组成：要么都执行，中途出错则回滚 |
| C（一致性）| 数据库在事务执行前后状态保持一致，保证数据读取正确 |
| I（隔离性）| 一个事务的更新在没有提交之前，不会被其他事务看到 |
| D（持久性）| 事务更新一旦提交，对数据库是永久的，即便崩溃也可通过日志恢复 |

## Q16. InnoDB 的 MVCC

**MVCC（多版本并发控制）：** 无锁多版本并发控制，读不加锁，读写不冲突，并发性好

**工作原理：**
- MVCC 为每个数据根据事务维护了多个版本
- 在并发事务中解决了读写冲突，同时使用**快照读**提供非阻塞读功能
- InnoDB 默认隔离级别是可重复读，MVCC 替它解决了幻读问题

**实现基础：**
- 每行数据的隐藏列
- 隐藏列中指向 undo log 的指针（将数据恢复到指定版本）
- ReadView（确定恢复到哪个版本）

## Q17. 乐观锁和悲观锁

| 特性 | 乐观锁 | 悲观锁 |
|------|--------|--------|
| 读取时 | 不加锁 | 总是加锁 |
| 更新时 | 检查期间是否有其他人修改了数据 | 阻塞其他人读取 |
| 适合场景 | 多读场景，并发性好 | 多写场景 |
| 典型应用 | GIT 代码版本控制 | MySQL 排他锁、共享锁 |
