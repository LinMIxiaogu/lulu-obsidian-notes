# 分布式与微服务面试题

## Q1. CAP 定理

在一个分布式系统中，**Consistency（一致性）**、**Availability（可用性）**、**Partition tolerance（分区容错性）** 这 3 个基本需求，**最多只能同时满足其中的 2 个**。

| 特性 | 说明 |
|------|------|
| **一致性（C）** | 所有节点在同一时刻的数据是一致的（访问任何节点得到相同的数据）|
| **可用性（A）** | 每个请求都能得到响应，但不保证数据是最新的 |
| **分区容错性（P）** | 网络分区（节点间通信失败）时，系统仍然能继续工作 |

**结论：** 对于分布式系统，**网络分区（P）是必然存在的**，因此必须满足 P。在 P 的前提下，**一致性（C）和可用性（A）是矛盾的**：
- 选 CP（如 ZooKeeper）：保证数据强一致，牺牲可用性
- 选 AP（如 Redis、Eureka）：保证高可用，允许短暂数据不一致

## Q2. BASE 理论

BASE 理论是对 CAP 中 AP 的延伸，是互联网系统设计的核心理论：

| 特性 | 说明 |
|------|------|
| **BA（基本可用）** | 系统出现故障时仍然可用，但允许响应时间损失或功能降级 |
| **S（软状态）** | 允许系统中的数据存在中间状态（不同节点间的数据副本允许短暂延迟）|
| **E（最终一致性）** | 不允许一直处于软状态；在一定时间后，所有副本达到数据一致性 |

## Q3. 分布式锁的实现方式

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| **MySQL 分布式锁** | 专门一张表，通过唯一索引字段实现互斥 | 简单，但性能差，依赖数据库 |
| **ZooKeeper 分布式锁** | 在 ZNode 下创建有序临时节点，序号最小的获得锁；释放锁通知后续节点 | 可靠，强一致性，但性能一般 |
| **Redis 分布式锁** | `SET key value NX EX seconds` 原子命令，NX 保证互斥，EX 保证锁自动过期 | 性能高，但存在主从切换时锁丢失问题 |

**Redis 分布式锁的演进：**
1. `setnx + expire`：两条命令非原子，可能死锁
2. `SET key value NX EX`：原子操作，解决死锁，但释放锁可能误删他人的锁
3. `SET key uuid NX EX` + 释放时校验 uuid：防止误删，但校验和删除不是原子操作
4. **Lua 脚本**（`setnx + expire` + `get + del`）：完整原子操作，推荐
5. **Redisson**：封装好的 Redis 分布式锁客户端，支持可重入、看门狗自动续期

## Q4. 分布式事务的实现方案

**2PC（两阶段提交）：**
- **阶段 1（准备）：** 事务管理器通知所有参与者预提交，参与者执行操作并锁资源，返回 OK/FAIL
- **阶段 2（提交）：** 所有参与者都 OK 则提交；否则回滚
- **缺点：** 同步阻塞，事务管理器单点故障，数据不一致风险

**3PC（三阶段提交）：**
- canCommit → preCommit（记录 undo/redo 日志）→ doCommit
- 加入超时机制：参与者超时未收到 commit，则自动提交
- **优点：** 减少阻塞；**缺点：** 仍然存在数据不一致问题

**TCC（Try-Confirm-Cancel）：**
- **Try**：尝试执行任务（预留资源）
- **Confirm**：确认执行业务（正式提交）
- **Cancel**：取消待执行的业务（释放预留资源）
- **特点：** 业务层面的分布式事务，保证最终一致性，侵入性强

**MQ 消息事务（可靠消息最终一致性）：**
- 业务操作成功后发送消息，消费者消费消息完成另一部分操作
- 两个本地事务通过消息中间件异步解耦
- **特点：** 高可用，实现最终一致性，对业务侵入性小

## Q5. 设计模式六大原则

| 原则 | 说明 |
|------|------|
| **开放封闭原则** | 发生需求变化时，对扩展开放，对修改封闭 |
| **单一职责原则** | 一个类/方法只负责一个职责，相互不影响，降低耦合 |
| **接口隔离原则** | 一个类对另一个类的依赖应建立在最小接口上（不应强迫客户端依赖它不需要的接口）|
| **迪米特法则** | 只和直接朋友交流，不和陌生人说话（最少知识原则，减少对其他对象的依赖）|
| **里氏替换原则** | 任何基类可以出现的地方，子类一定可以出现（子类不改变父类行为）|
| **依赖倒转原则** | 高层模块不依赖低层模块，两者都依赖抽象（面向接口编程）|

## Q6. 微服务是什么，有什么优势？

**定义：** 将单一应用程序划分为一组小的服务，每个服务运行在独立的进程中，服务之间通过轻量级 API（通常是 HTTP/RPC）相互协作

**优势：**
- **单一职责**：每个服务专注一个业务功能
- **独立部署**：各服务可以独立开发、测试、部署、扩展
- **业务解耦**：业务与业务之间完全解耦
- **横向扩展**：可以针对某一服务单独扩容
- **技术多样性**：各服务可以使用不同的技术栈

**劣势：** 分布式系统复杂度高，服务间通信、事务、监控难度增加

## Q7. DDD 架构和 MVC 架构的区别

| 特性 | MVC 架构 | DDD 架构 |
|------|---------|---------|
| 驱动方式 | 以数据库表为驱动，从表结构开始设计 | 以业务领域模型为驱动 |
| 分层结构 | DAO → Service → Controller | 领域层（model + 仓储 + 服务）→ 应用层 → 接口层 |
| 问题 | Service 层臃肿，DAO/PO/VO 类随项目膨胀，不易维护 | 需要对业务有深入理解，设计成本高 |
| 适用场景 | 业务简单，CRUD 为主 | 复杂业务，领域逻辑丰富 |

## Q8. 分布式限流

**常见限流算法：**

| 算法 | 说明 | 特点 |
|------|------|------|
| **固定窗口** | 固定时间窗口内限制请求数量 | 简单，但时间窗口边界可能突发 2 倍流量 |
| **滑动窗口** | 以当前时间为基准，向前看固定时间内的请求数 | 解决固定窗口边界问题 |
| **漏桶算法** | 请求放入桶中，以固定速率流出，桶满则拒绝 | 输出速率固定，无法应对突发流量 |
| **令牌桶算法** | 以固定速率向桶中添加令牌，请求需要获取令牌才能通过，桶满时多余令牌丢弃 | 允许一定程度的突发流量（如 Guava RateLimiter）|

**Redis 实现限流：** 使用 `INCR` + `EXPIRE` 或 Lua 脚本实现滑动窗口限流；也可使用 Redis 的 `ZSET` 实现精确的滑动窗口
