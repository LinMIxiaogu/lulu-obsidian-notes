## 一、推理优化全景图

> "这节课只记住一件事的话，那就是：推理速度的瓶颈在内存，不在算力。"
> — Percy Liang（原话：*"If that's one thing you take away from this lecture, it's like all about the memory for speed."*）

### 核心认知：推理的瓶颈是内存，不是算力

一次 LLM 推理分为两个阶段，计算特征截然不同：

|      | Prefill（处理 prompt） | Decode（逐 token 生成） |
| ---- | ------------------ | ------------------ |
| 并行性  | 所有 prompt token 并行 | 严格逐 token 串行       |
| 瓶颈   | **计算受限**（GPU 在拼命算） | **内存受限**（GPU 在等数据） |
| 用户感知 | TTFT（首 token 延迟）   | 每 token 延迟（打字速度）   |

![Prefill 和 Decode 两阶段](assets/A2-A5/prefill-decode.png)

**为什么 Decode 是内存受限的？**

以 Llama 2 13B 为例：每生成一个 token，GPU 需要从 HBM（High Bandwidth Memory，高带宽内存，即 GPU 显存）加载全部 13B 参数（bf16 下约 26GB），却只做一次矩阵向量乘法。H100 的 HBM 带宽 3.35 TB/s，光读参数就要约 8ms，这就是每 token 延迟的下限。

而且 **Attention 层增大 batch size 也无法改善**。MLP 权重在所有 token 序列间共享（读一次服务 B 个 token 序列），但 Attention 每个 token 序列有独立的 KV Cache（读写量与 B 成正比，分子分母抵消）。

**判断瓶颈的工具：Roofline 模型**

- H100 bf16 算力 ~989 TFLOPS，HBM 带宽 3.35 TB/s
- **加速器强度** = 989 / 3.35 ≈ **295 FLOPs/Byte**
- 操作的算术强度 > 295：compute-bound（好，GPU 在拼命算）
- 操作的算术强度 < 295：memory-bound（坏，GPU 在等数据）
- Decode 阶段 B=1 时算术强度约 1，**远低于 295**

过去十年 Tensor Core 算力增长了 ~100,000 倍，HBM 带宽仅增长 ~100 倍。内存瓶颈只会越来越严重，所有减少内存搬运的优化（量化、算子融合、FlashAttention）的价值只会越来越大。

### 六大优化技术速览

#### 1. KV Cache：为什么你的长对话越来越慢

LLM 逐 token 生成。假设输入"今天天气"要生成"真不错"，没有缓存时每步都要重算所有 token 的 Key/Value：

```
第1步：算 [今,天,天,气] 的 K,V           → 输出 "真"
第2步：算 [今,天,天,气,真] 的 K,V         → 输出 "不"   ← 前4个白算了
第3步：算 [今,天,天,气,真,不] 的 K,V      → 输出 "错"   ← 前5个白算了
```

KV Cache 把算好的 Key/Value 存下来，下一步直接复用：

```
第1步：算 [今,天,天,气] 的 K,V → 缓存 → 输出 "真"
第2步：只算 [真] 的 K,V     → 拼到缓存 → 输出 "不"
第3步：只算 [不] 的 K,V     → 拼到缓存 → 输出 "错"
```

每步只算 1 个新 token 的 K/V，计算量从 O(T³) 回到 O(T²)。代价是用**显存换速度**：所有历史 K/V 常驻 GPU 显存。

**内存占用公式**：`KV Cache = 序列长度 × KV头数 × 头维度 × 层数 × 2(K+V) × 2(bf16) 字节`

Llama 2 13B 实例（S=1024, K=40, H=128, L=40）：

- B=1 时 KV Cache 约 0.8GB，尚可
- B=256 时 KV Cache 约 ~210GB，**单张 H100 (80GB) 直接爆**

**对 Agent 的影响**：10K token 的对话历史 + 2K 系统提示，KV Cache 吃掉 GB 级 HBM，直接限制并发数。

缩减方案：

| 方案             | 原理              | 压缩比   | 代表          |
| -------------- | --------------- | ----- | ----------- |
| GQA            | 多个 Q 头共享 K/V 头  | N/K 倍 | Llama 3     |
| MLA            | KV 投影到低维空间      | ~32x  | DeepSeek V2 |
| Sliding Window | 只保留最近 K 个 token | O(1)  | Mistral     |

**选模型时务必关注是否用了 GQA**，这直接决定推理成本。

##### KV Cache vs Prefix Caching

上面的 KV Cache 是**单请求内部**的优化，请求结束就释放。Prefix Caching 是**跨请求**的优化：推理框架把 KV Cache block 保留在显存里，后续请求前缀 token 相同就直接复用，跳过 prefill。

匹配条件为：**同一个推理实例** + **token 序列前缀相同**。跟哪个用户、哪个 conversation 无关。

| 条件 | 是否共享 |
|------|---------|
| 同实例 + token 前缀相同 | 共享（无论是否同一个用户/conversation） |
| 同实例 + token 前缀不同 | 不共享（差一个 token 就断开） |
| 不同实例（不同机器） | 不共享（各自维护显存） |

system prompt 里插了用户名等个性化内容，即使只差一个 token，匹配就从那个位置断开。

这两个层次的区分也体现在 API 计费上：

KV Cache 相关（单请求内部）：

- 输入 token（Prefill，计算 KV Cache，全价）
- 输出 token（Decode，逐 token 读权重 + 追加 KV Cache）

Prefix Caching 相关（跨请求复用）：

- 缓存写入（首次 Prefill 并存入缓存）
- 缓存读取（命中缓存，跳过计算，大幅折扣）

##### Prompt 按变化频率分层

Prefix Caching 按前缀匹配，所以 prompt 组织原则是**越稳定的内容越靠前，变化越快的越靠后**。以"告诉模型当前时间"为例，错误写法：

```
你是旅行助手。当前时间：2026-03-29 14:20:35。  ← 每秒都变
你可以调用以下工具：...（2000 token 全部失效）
```

正确做法，按变化频率分层：

```
── 不变（所有请求共享缓存）──────────
你是旅行助手。你可以调用以下工具：...（2000 token）

── 每天变一次（同一天共享）──────────
当前日期：2026-03-29（星期日）

── 每轮都变 ─────────────────────
当前时间：14:20
用户：帮我查明天上海飞大阪的航班
```

同理：RAG 结果放末尾，few-shot 示例放中间，早期对话压缩为摘要。日期精度够用就不传秒级时间。

#### 2. 量化：用精度换速度

推理是 memory-bound，减少内存传输量就能加速。

| 方法 | 精度 | 效果 | 注意事项 |
|------|------|------|---------|
| bf16（基线） | 2B/参数 | — | 默认 |
| LLM.int8() | 1B/参数 | 内存 2x 压缩 | **速度反而慢 15-23%**（混合精度开销） |
| AWQ / GPTQ | int4 | **内存 4x 压缩，3.2x 加速** | Agent 场景质量损失可忽略 |

![AWQ 量化原理](assets/A2-A5/awq-schema.png)

- **(a) RTN 量化**：直接把 FP16 权重四舍五入到 INT3，PPL（Perplexity，困惑度，越低越好）从正常水平飙到 43.2，质量严重下降
- **(b) 混合精度**：通过激活值找到 1% 的"显著权重"保留 FP16，其余量化到 INT3，PPL 降到 13.0；但矩阵里混了两种精度，硬件执行效率差（图中红字 "bad hardware efficiency"）
- **(c) AWQ 的做法**：量化前先按通道缩放权重（scale before quantize），让显著权重的数值范围变小、量化损失更小，然后全部统一量化到 INT3。PPL 同样 13.0，且硬件友好

**结论**：直接上 AWQ/GPTQ int4，跳过 int8。

#### 3. 投机采样（Speculative Decoding）：唯一无损的加速

核心洞察：验证（Prefill）远快于生成（Decode）。

1. 小模型（如 8B）快速生成 K 个候选 token
2. 大模型（如 70B）一次并行验证（本质是 prefill）
3. 通过拒绝采样决定接受或拒绝

**关键**：输出分布与目标模型完全一致，**无损加速约 2x**。

![投机采样结果](assets/A2-A5/speculative-sampling-results.png)

- **ArS（Autoregressive Sampling）**：常规自回归采样，每次生成 1 个 token，作为基线（1x）
- **SpS（Speculative Sampling）**：投机采样，小模型草拟 K=4 个 token，大模型一次验证
- **结果对比**：质量几乎不变（Result 列数值接近），速度提升 1.92x ~ 2.46x
- **HumanEval（代码生成）加速最明显**（2.46x）：代码输出模式规律，小模型猜中率更高

Agent 输出格式规律（JSON、函数调用），draft 模型接受率更高，天然适配。

#### 4. PagedAttention / vLLM：操作系统思想的复用

传统方案为每个请求预分配最大长度的 KV Cache，导致内存碎片严重。

PagedAttention 借鉴虚拟内存：把 KV Cache 切成固定大小的 block，按需分配。

![PagedAttention block 示意](assets/A2-A5/paged-attention-blocks.png)

- **右侧表格**：物理内存中的 KV Cache 被切成固定大小的 block（每个 block 存 4 个 token 的 Key/Value 向量），block 之间不需要连续存放
- **左侧 Query vector**：当前要生成的 token "forth" 需要与之前所有 token 做 attention 计算
- **箭头指向**：逻辑上连续的序列 "Four score and seven years ago our fathers brought forth" 在物理内存中分散在 Block 0、Block 1、Block 2，通过类似页表的映射找到对应 block
- **核心收益**：不用预分配最大长度的连续内存，按需分配 block，用完即回收，解决内存碎片问题

Agent 特别受益的场景：
- **系统提示词共享**：所有请求共享 2000+ token 的 system prompt（copy-on-write）
- **多路采样**：同一 prompt 生成多个候选方案，几乎不占额外内存
- 内存节省 **30-50%**

#### 5. Continuous Batching：Agent 必备的调度策略

推理服务（如 vLLM）同时服务多个用户请求时，会把多个请求打包成一个 batch 送进 GPU 并行计算，以提高吞吐量。比如同一时刻有 3 个用户发了请求，推理服务把这 3 个请求组成一个 batch，GPU 一次前向传播同时为 3 个请求各生成一个 token。

**静态批处理的问题**：一个 batch 内的所有请求必须全部生成完毕，才能开始处理下一个 batch。batch 内先完成的短请求只能干等最慢的长请求，GPU 在空转。

```
静态批处理：
  时间 →  t1    t2    t3     t4    t5    t6    t7     t8
  请求A: [生成] [生成] [完成] [空等] [空等] [空等] [空等] [空等]  ← 生成2个token就结束，但被迫等到t8
  请求B: [生成] [生成] [生成] [生成] [生成] [生成] [生成] [完成]  ← 生成8个token
  请求C:  ————————— 等待整个 batch 完成后才能进入 —————————

Continuous Batching：
  时间 → t1     t2    t3    t4     t5    t6    t7     t8
  请求A: [生成] [生成] [完成]
  请求B: [生成] [生成] [生成] [生成] [生成] [生成] [生成] [完成]
  请求C:                    [生成] [生成] [完成]                ← A完成后下一步加入
  请求D:                                       [生成] [...]    ← C完成后下一步加入
```

**Continuous Batching**（Orca 方案）：每生成一个 token，控制权返回调度器。完成的请求立即移除释放资源，新请求立即加入。

配合 **Selective Batching** 处理变长序列：
- **Attention 部分**：每个请求用自己的 KV Cache 独立计算（因为每个请求长度不同）
- **MLP 部分**：将所有请求的 token 拼接为一个大张量统一计算（MLP 权重共享，token 间不交互，拼在一起算更高效）

Agent 的请求天然异构（工具调用返回几个 token，思考链生成几百 token），Continuous Batching 几乎是必须的基础设施。

#### 6. FlashAttention：长上下文的基础设施

**问题**：标准 Attention 需要在 HBM 中实体化完整的 N×N 注意力矩阵。N=128K 时，仅这一个中间矩阵就需要 128K × 128K × 2 bytes ≈ **32GB**，接近 H100 的 HBM 上限。

FlashAttention 的核心思路是把计算搬到 SRAM（Static RAM，GPU 片上缓存，容量小但速度比 HBM 快一个数量级）中完成，避免 N×N 矩阵在 HBM 上反复读写。

```
标准 Attention 的数据流：

  HBM（慢）                              SRAM（快，但很小）
  ┌──────────┐         读 Q, K          ┌─────────────┐
  │ Q, K, V  │ ───────────────────────→ │  计算 QK^T  │
  └──────────┘                          └──────┬──────┘
                                               │
  ┌──────────┐         写回 S                  │
  │  S(N×N)  │ ←───────────────────────────────┘
  └────┬─────┘  ↑ S = QK^T，未归一化的注意力分数，完整 N×N 矩阵写回 HBM
       │
       │               再读取 S          ┌─────────────┐
       └─────────────────────────────────→  Softmax    │
                                         └──────┬──────┘
                                                │
  ┌──────────┐         写回 P                   │
  │  P(N×N)  │ ←────────────────────────────────┘
  └──────────┘  ↑ P = Softmax(S)，归一化后的注意力权重，又一个 N×N 矩阵写回 HBM

  问题：两个 N×N 矩阵在 HBM 上反复读写，内存访问 O(N²)


FlashAttention 的做法：

  HBM（慢）                              SRAM（快）
  ┌──────────┐        按块读取           ┌─────────────────────────┐
  │ Q, K, V  │ ───────────────────────→ │  加载一小块 Qi, Kj, Vj    │
  └──────────┘                          │                         │
                                        │  在 SRAM 内完成：         │
  ┌──────────┐        只写结果           │    局部 QK^T             │
  │  O(输出) │ ←─────────────────────── │    Online Softmax        │
  └──────────┘                          │    累加输出               │
                                        └─────────────────────────┘

  N×N 矩阵从不实体化，HBM 访问降到亚二次方
```

FlashAttention 的三个关键技术：
- **Tiling（分块）**：将 Q、K、V 切成小块，逐块加载到 SRAM 中计算，不生成完整 N×N 矩阵
- **Online Softmax**：不需要看到所有值就能正确计算 softmax，支持逐块流式处理
- **重计算**：反向传播时不存储中间注意力矩阵，而是重新算一遍，用多余的算力换内存带宽

FlashAttention 使 128K 甚至 1M 上下文窗口成为可能。vLLM、TGI 等主流推理框架已默认集成。

## 二、缩放定律与模型选型

### 核心公式

```
L(N, D) = A / N^α + B / D^β + E
最优配比：N_opt ∝ C^0.50，D_opt ∝ C^0.50
其中 C ≈ 6·N·D
```

- **L**：模型的损失（loss），越低越好
- **N**：模型参数量（Parameters），**A / N^α** 是参数不足项，模型太小时这项主导
- **D**：训练数据量（Data，token 数），**B / D^β** 是数据不足项，数据太少时这项主导
- **E**：不可约误差（irreducible error），即使模型无限大、数据无限多也无法消除的损失下限（语言本身的熵）
- **C**：总计算量（FLOPs）。6 的来源：前向传播 2N + 反向传播 4N（对激活值梯度 2N + 对权重梯度 2N）= 6N FLOPs/token
- **N_opt ∝ C^0.50，D_opt ∝ C^0.50**：给定固定计算预算，参数量和数据量应等比例增长。计算预算翻 4 倍，参数量翻 2 倍，数据量也翻 2 倍

**[Chinchilla 定律](https://arxiv.org/abs/2203.15556)**（DeepMind, 2022）：固定计算预算下，模型和数据等比例缩放，最优约 20 tokens/parameter。之前 GPT-3 的做法是模型做得很大但数据给得不够（175B 参数只用 300B token），Chinchilla 证明同样的计算预算下，更小的模型（70B）+ 更多数据（1.4T token）效果更好。

![IsoFLOP 分析的 U 形曲线及最优点](assets/A2-A5/chinchilla-isoflop.png)

- **左图（IsoFLOP 曲线）**：每条曲线代表一个固定的计算预算（从 6e18 到 3e21 FLOPs），横轴是参数量，纵轴是训练损失。每条曲线都呈 U 形：参数太少欠拟合，参数太多则数据不够也会变差。U 形底部的最优点随计算预算增大而右移
- **中图（最优参数量 vs FLOPs）**：把每条 U 形曲线的最优点取出来，画出计算预算与最优参数量的关系。斜率约 0.50，即 N_opt ∝ C^0.50。
- **右图（最优数据量 vs FLOPs）**：同理画出计算预算与最优数据量的关系，斜率同样约 0.50，即 D_opt ∝ C^0.50。

### 但实际已经变了：从训练最优到推理最优

Chinchilla 定律给出的是**训练计算最优**，但现实中模型训练一次、推理无数次，真正的成本大头在推理侧。

| 时代 | tokens/param | 策略 | 代表模型 |
|------|-------------|------|---------|
| GPT-3 | ~2 | 大模型少训练 | GPT-3（175B 参数，300B token） |
| Chinchilla | ~20 | 训练计算最优 | Chinchilla（70B 参数，1.4T token） |
| **当前** | **100~200+** | 小模型多训练 | Llama 3 8B（15T token），Qwen 2.5 |

**为什么要"过度训练"？**

按 Chinchilla 定律，8B 模型的最优数据量约 160B token，但 Llama 3 8B 实际用了 15T token，接近最优量的 100 倍。这看起来"浪费"了训练计算，但考虑推理成本就合理了：

- Agent 一个任务可能触发 10~50 次 LLM 调用，推理量远大于训练量
- 70B 模型的单次推理成本是 8B 的近 10 倍（参数量决定了每次推理的内存和计算开销）
- "过度训练"8B 模型的额外训练成本是一次性的，但每次推理都在省钱
- 实测过度训练后的小模型性能可以接近按 Chinchilla 最优训练的大模型

### 推理时计算缩放（Test-time Compute Scaling）

```
总性能 = f(模型大小) + g(推理时计算量)
```

三种策略：
- 大模型 + 直接输出：单步质量高，但延迟和成本都高
- 小模型 + CoT 推理：单步便宜，但需要更多 token 来"思考"，总 token 多
- **混合策略**：简单任务直接输出，复杂任务启用 CoT（Qwen3 等模型已支持 thinking budget 控制）

### 选型建议

- 关注 **tokens/param 比率**：>100 说明训练充分，同参数量下通常更好
- 缩放定律在 loss 上完美，但在具体能力上可能存在**涌现阈值**，别只看 loss

## 三、评估方法论

> "评估决定了语言模型的未来。"
> — Percy Liang（原话：*"Evaluation determines the future of language models."*）

### 四维评估框架

1. 输入：评估集覆盖了哪些用例？有没有长尾困难样本？
2. 调用方式：zero-shot / few-shot / CoT / 带工具？评的是裸模型还是完整系统？
3. 输出评估：什么指标？pass@k？精确匹配？LLM 打分？
4. 结果解读：91% 意味着什么？能上线吗？有没有数据污染？

### Agent Benchmark：远未饱和

| Benchmark | 测什么 |
|-----------|--------|
| SWE-bench | 给 GitHub issue 生成修复 patch |
| CyBench | CTF 安全挑战 |
| MLEBench | Kaggle 竞赛全流程 |

当 MMLU（Massive Multitask Language Understanding，大规模多任务语言理解，用选择题测试模型在 57 个学科上的知识水平）已经精确到小数点后一位时，Agent benchmark 远未饱和，提升空间很大。

![SWE-bench 的任务结构](assets/A2-A5/swebench.png)

以 SWE-bench 为例，一个完整的评估任务包含：

Test Sample / Input（**测试样本**，左侧 Model Input）：
- Instructions：任务说明
- Issue：GitHub issue 的完整描述，含问题代码片段
- Code：相关源文件
- Additional Instructions：补充约束

Ground Truth / Gold Label（**标准答案**，右上 Gold Patch）：
- 人类开发者提交的真实修复 patch

Prediction / Model Output（**模型预测**，右中 Generated Patch）：
- 模型生成的 patch，这个例子中模型只做了单行替换，没有处理完整的条件逻辑

Evaluation Metric（**评估指标**，右下 Test Results）：
- 用项目自带的单元测试验证模型的 patch，45 passed / 2 failed，说明修复不完整
- 注意：这里的 "Test" 是软件工程的单元测试，不是 ML 的测试集。它在评估中充当的角色是评估指标，类似 MMLU 用"答案是否匹配"打分，SWE-bench 用"单元测试是否通过"打分

这个评估流程直接在真实代码库中完成端到端的 bug 修复，用项目自身的测试来判定对错。

### 评估的常见坑

- 数据污染：模型可能在训练时已见过测试题
- Prompt 敏感性：few-shot 示例的顺序、格式都大幅影响分数
- 成本被忽略：o3 可能性能最高但贵 10 倍，要看 Pareto frontier
- 评估对象混淆：你以为在评模型，实际在评 prompt 模板

![Artificial Analysis 的性能-成本 Pareto frontier](assets/A2-A5/artificial-analysis.png)

- **纵轴（Intelligence Index）**：Artificial Analysis 的综合智能指数，聚合了 MMLU-Pro、GPQA Diamond、LiveCodeBench、AIME、MATH-500 等多个 benchmark 的得分
- **横轴（Price）**：每百万 token 的价格（USD），按输入:输出 = 3:1 的比例混合计算
- **绿色区域（左上角）**：性能高且价格低的"甜区"
- **如何读图**：选模型时应关注左上角方向（性能高、价格低）。绿色区域外的模型被绿色区域内的模型同时在性能和价格上超越
- **离群点 o3**：性能不是最高但价格远超其他模型（~$20/M token），是"成本被忽略"这个坑的典型案例

### 实操建议

1. 不要迷信排行榜，构建自己的 200-500 条评估集
2. 关注成本-性能 Pareto frontier，不只看准确率
3. 用 LLM-as-Judge 做规模化评估（注意长度偏好、位置偏差）
4. 从真实用户请求采样，持续评估，不要一次性静态测试

## 四、数据与对齐

### 预训练数据：1.4% 的精华

DCLM（DataComp for Language Models，语言模型数据竞赛，一个开源的预训练数据筛选基准）从 **240 万亿** tokens 过滤到 **3.8 万亿**（仅保留 1.4%），效果大幅提升。

![DCLM 从 240T tokens 过滤到 3.8T 的完整 pipeline](assets/A2-A5/dclm-filter.png)

这张图从左到右展示了三阶段过滤流水线，百分比是相对原始文档总数的保留比例：

- **第一阶段：Heuristic Cleaning（启发式清洗）**：用规则过滤，从 100% 砍到 19.9%
- **第二阶段：Deduplication（去重）**：从 19.9% 降到 13.7%
- **第三阶段：Model-based Filtering（模型过滤）**：从 13.7% 降到 1.4%

关键启发：
- HTML 转 Text 用 trafilatura 比直接用 Common Crawl 的 WET 格式高 ~4 个百分点
- "什么是高质量"没有绝对标准，取决于你选择的代理信号（即用来间接衡量数据质量的指标，比如用 Wikipedia 作为"高质量"参照训练 FastText 分类器，不同参照会筛选出风格不同的数据）
- 去重至关重要（C4 数据集某段文本重复 61,036 次）

**对 Agent 工程师的迁移**：
- RAG 知识库构建时，数据清洗的思路相同：提取正文、去重、质量过滤
- Fine-tuning 花 80% 时间在数据质量，20% 在训练

### 对齐技术演进（重点）

```
预训练模型（会续写，不会对话）
  → SFT（教模型"说人话"，学会输出格式）
    → RLHF（利用人类偏好，提升回答质量）
      → DPO（把 RL 简化为监督学习）
        → RLVR（用可验证奖励替代人类偏好）
          → GRPO / DeepSeek-R1（推理能力训练）
```

#### SFT：三个关键认知

1. 少量数据就够：几百~几千条高质量数据就能改变模型行为
2. 教的是格式，不是知识：输出格式靠 SFT，正确性靠预训练
3. 会鼓励幻觉：如果训练数据超出模型知识范围，模型学会"自信地编造"


#### 从 RLHF 到 DPO：从复杂到简洁

RLHF 的动机：人类验证能力 > 生成能力（Generator-Validator Gap）。与其让人写示范，不如让人判断"A 比 B 好"。

RLHF 的问题：需要同时跑 4 个模型（策略、参考、奖励、价值函数），工程极其复杂。

DPO 的突破：数学推导发现可以跳过奖励模型，直接用策略的 log ratio 隐式表达奖励。只需偏好数据 + 两个模型，标准监督学习框架就能跑。

#### RLVR + GRPO：推理模型的训练秘密

RLHF 的天花板：奖励模型有噪声，模型会学会讨好奖励模型而非真正变好（过优化问题）。

RLVR 的思路：在数学、代码等可验证域，直接用"答案是否正确"作为奖励。没有噪声，理论上可以超越人类水平。

DeepSeek-R1 的实验：
- 直接在基座模型上用 GRPO + 可验证奖励训练数学推理
- 不做 SFT、不用过程奖励模型
- 结果接近 OpenAI o1

GRPO vs PPO：对每个问题采样多个回答，用组内平均奖励做 baseline，省掉了价值网络（critic）。

#### Qwen3：推理预算控制

同一模型通过 `/think` 和 `/no_think` 控制推理深度。Agent 可以根据问题难度动态选择：简单问题直接回答，复杂问题深入推理。

### 对齐技术选型速查

| 场景 | 推荐方法 | 数据量 |
|------|---------|--------|
| 控制输出格式/风格 | SFT | 几百条 |
| 提升回答质量（主观） | DPO | 偏好对数据 |
| 提升推理正确率 | RLVR (GRPO) | 可验证题目 |
| 快速获得推理能力 | 蒸馏（大模型 CoT 数据 SFT 小模型） | ~1000 条就有效 |

### Agent 工程师的行动项

- 利用 Generator-Validator Gap：在 Agent 流程中设计"生成、自验证、修正"循环
- 给模型留"不知道"的退路：SFT 鼓励幻觉，提供搜索/RAG 工具能减少编造
- RLHF 模型的概率不可靠：不要用 token 概率当置信度，用多次采样一致性
- 推理模型和 Agent 天然契合：Agent 很多任务本身可验证（代码执行、API 调用），与 RLVR 训练范式一致

## 五、训练基础设施

你不需要自己训练 70B 模型，但理解成本结构帮你做选型决策。

### 三种并行策略

| 策略 | 切什么 | 适用场景 |
|------|--------|---------|
| 数据并行 | batch 维度 | 每卡装得下完整模型时 |
| 张量并行 | 权重矩阵宽度 | 同节点内 8 卡（NVLink） |
| 流水线并行 | 模型深度（按层） | 跨节点 |

**经验法则**：先在节点内做张量并行，再跨节点做流水线/FSDP，最后用数据并行扩吞吐

### 训练一个 70B 模型

- 每参数需要 **16 字节**内存（参数 + 梯度 + 优化器状态）
- 7B 模型需 ~112GB，单张 A100 (80GB) 装不下
- FSDP 把一切切片分散到多卡，8 张 A100 可装 ~50B 模型

![不同规模模型的 FLOPs 需求对照](assets/A2-A5/roller-flops.png)

**Napkin Math**：70B 模型充分训练（7T tokens）
- 总 FLOPs ≈ 3×10²⁴
- 1024 张 H100，约 **87 天**
- 成本约 **$5.3M**

### 对选型的实际影响

- 模型更新不是"月更"：前沿训练需要数月 + 评估对齐数月，半年周期正常
- 7B/13B/70B 不是随意选的，和 8 卡节点的张量并行上限直接相关
- 微调比预训练便宜得多：FSDP + 几张卡就行，7B 用 LoRA 单卡就够
