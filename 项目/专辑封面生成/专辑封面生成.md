## 项目名称：基于多模态大模型的专辑封面智能生成系统

- **技术栈：** RAG、LangChain、Lora、Python
    
- **项目描述：**构建自动化专辑封面生成系统，借助垂直知识库和大模型实现专辑封面的风格化生成。
    
- **核心职责与技术实现：**
    
    - RAG专辑知识检索系统：专辑知识库向量数据库存储，运用语义分块以提取文本特征，设计多向量融合检索与二次排序逻辑，并引入标签生成机制，解决“类型相似但内容差异大”及“风格时效性问题”。专辑检索准确率达92%。
        
    - 大模型工作流：主导LLM调用：先基于rag检索结果生成风格决策，整合信息生成精细化生图提示词，解决生图模型风格理解偏差问题。
        
    - cogview4模型垂直微调：基于赛道专辑优质封面，用cogkit包实现lora微调，强化不同分类专辑与视觉风格的关联性。
        

**结果**：ab实验上人工审核通过率达 89%。

### RAG 实现细节（含动态标签生成与语义分块）

###### 流程（用户输入——〉rag召回数据）

- 知识库构建
    
    - 准备专辑数据（标题，简介，图片上传时间，图片url，【标签列表】）。
        
        - 将【简介】——>大语言模型——>匹配标签列表。
            
        - 将（标题｜简介｜匹配标签列表）用语义分块，再模型BAAI/bge-large-zh-v1.5进行转化向量存到向量数据库milvus中。
            
    - 用户输入（专辑标题+专辑简介+专辑赛道/分类——目前只开放网文和相声评书）
        
    - llm优化用户输入[专辑简介]——>匹配标签列表；
        
    - rag检索：以（标题｜简介｜匹配标签列表）检索（计算余弦相似度）,返回top10
        
    - 二次重排：内容相似度的分*0.6+简介风格与返回图片风格匹配*0.3+图片上传时间*0.1（图片上传时间，1年内为1，往前推1年*0.8，0.5为最低值）
        
    - rag检索数据返回：取二次重排的得分top3返回。
        

###### 语义分块算法

**系统使用语义分块算法（对内容简介）：**

1. 将文档（内容简介）按段落或句子（标点）进行分段，为每个分段进行向量化(Embedding)
    
2. 从第一个向量化分段开始
    
3. 如果第一个分段与第二个分段具有高余弦相似度（相似度阈值0.6），则合并
    
4. 检查合并后的分块大小是否超过限制，如果超过则停止合并
    
5. 继续与后续分段比较，直到相似度低于阈值或达到大小限制
    
6. 重复步骤3-6，直到所有分块结束
    

  效果：相比于直接固定大小分块，检索效果提升7%。

  优点：语义连贯性：不会将连贯的语义强制切分，导致单块包含不完整信息。另外每个块中每个块中的内容核心相似，检索时匹配度更高。

**Q：使用的向****量化模型****，为什么选这个**

A：BAAI/bge-large-zh-v1.5，嵌入维度：1024维，专为中文场景设计，语义理解能力强，计算和存储资源消耗低。

**Q：使用的向量数据库，为什么选这个**

A：milvus，相同硬件配置下，**qps****更高**。完全开源。支持**多向量融合索引**。（标题0.4，简介0.4，标签0.2）

  标题，简介：bge语义编码——余弦相似度计算得分。

  标签：二元编码——点积匹配数量计算得分。

  FAISS：虽然单向量检索快，但本身不支持多向量融合检索场景，需要复杂代码实现，也没有分布式支持，暂时做算法的人使用更多。

  Chroma：不支持多向量融合检索场景

###### 标签生成逻辑：

触发机制：针对用户输入的专辑内容简介，调用 deepseek-r1 从该分类下提供标签库（含目前有风格、元素、受众三个维度，每个维度下30+标签（高频标签））中各选取 1个最匹配标签。

  优势：为长文本的简介提炼关键词，匹配准确率提升；

1. 问题：大模型匹配标签可能出现不完全遵从指令，会出现标签库之外的标签。
    
        影响：没有能正确匹配标签，导致标签分低，召回可能被遗漏，召回率降低。
    
        解决：代码筛选删除后补充该位置为“其它”。
    
2. 问题：当一个专辑是穿越在官场升级的类型，可能会匹配上穿越，也可能匹配官场。
    
        解决：prompt中增加标签匹配优先级具体规则。
    
        影响：匹配不稳定性导致召回遗漏，召回率降低。

###### RAG 检索效果评估：

1. 召回率：判断一次召回的专辑满足“相似”标准的概率
    
2. 准确率：判断最终的top3的专辑满足相似标准的概率（**本项目只关注了准确率**）
    
3. F1值: 准确率和召回率的调和平均数，用于综合衡量两者的平衡，避免单一指标的局限性
    
    1. 核心背景：准确率和召回率往往存在 “矛盾”（例如，提高准确率可能降低召回率，反之亦然），F1 值可作为两者的综合指标。
        
    2. 计算公式： $$\mathrm{F} 1=2 \times \frac{\text { Precision } \times \text { Recall }}{\text { Precision }+ \text { Recall }}$$
        
    3. 特点：F1 值范围为 0~1，值越高表示准确率和召回率的平衡越好。当准确率和召回率均高时，F1 值才会高；若一方极低，F1 值也会偏低。
        
4. 平均排序位置（MRR）:专注于排序质量，评估相关结果是否排在靠前位置。
    
5. 专辑召回的准确率（是否相似）：
    
    1. 相似规则：（标签强相似，弱相似+其它规则点>2点）——视为相似
        
        1. 标签维度：>=2个标签相同，强相似，>=1个标签，弱相似
            
        2. 标题维度：
            
            1. 关键词重合
                
            2. 隐含主题完全一致：如《时光机》与《重返过去》均指向 “时间回溯”）。
                
        3. 内容简介维度：
            
            1. 受众指向：
                
                1. 简介隐含的网文受众标签（如 “女频甜宠爱好者”“男频爽文读者”“无 CP 事业向”“强强党”“古风控”）完全一致
                    
            2. 核心元素>=3个：
                
                1. 简介中高频出现的网文标志性元素（非泛化词汇）重合≥3 个。常见元素分类：
                    
                    - **设定元素**：系统、空间、重生、穿越、异能、灵根、阵法、丧尸、机甲、穿书、马甲、直播、娱乐圈、电竞等；
                        
                    - **关系元素**：师徒、青梅竹马、霸总、甜宠、虐恋、强强联合、相爱相杀、白月光、替身等；
                        
                    - **情节元素**：逆袭、打脸、复仇、寻宝、闯关、权谋斗争、校园竞赛、末世生存等。
                        
            3. 风格与调性：
                
                1. **文风**：沙雕搞笑、正剧严肃、甜宠无虐、虐恋情深、爽文快节奏、慢热细腻、悬疑烧脑等；
                    
                2. **调性**：热血励志、轻松治愈、暗黑压抑、沙雕无厘头、现实向等。
                    
    2. 不相似规则：赛道不同；不满足相似规则。

###### Q:为什么选择cogview4模型

- cogview4是开源且支持中英文文字图片生成。在DPG-Bench(复杂语义对齐测试)中综合评分超过Stable Diffusion3和Flux.1，生图效果被认为可达到商用。
    

###### Q:为什么不直接使用第三方api服务，比如即梦，可灵

- 一开始是用的第三方api服务调用，但是风格不是很贴近我们平台一些分类想要的。尤其是相声评分，真人写实，人文国学类。它们有自己局限的风格。
    
    - 所以试图尝试自己准备垂直数据微调模型，一个是节省成本，另一个是更符合需要的风格类别。目前微调后的生成质量可替代三方的达到85%以上。
        

###### Q:为什么选择lora微调，而不是全参数微调，lora微调有什么优势和短板。

LoRA 通过冻结预训练模型参数权重，只训练少量新增的低秩适应矩阵（A 和 B），大幅减少可训练参数（如从 10B→0.5M），降低显存需求并加速训练。

###### Q:如何选择 LoRA 的秩（rank）参数？

Rank 的物理意义是低秩矩阵的 “维度”，决定了 LoRA 的表达能力：Rank 越大，可学习的模式越复杂，但参数越多（参数量 = 2×rank×in_dim×out_dim，按层计算）。

建议 Rank=4 或 8，核心逻辑是 “小数据配小 Rank，避免过拟合”：

50 张 / 类属于极小数据集，模型容易记住噪声而非规律：若 Rank 过大（如 16 或 32），LoRA 参数过多，会导致 “强行拟合少量样本的细节”（如某张图的特殊色彩偏差），泛化到新样本时效果极差。小 Rank（4/8）参数少，强制模型学习样本中的共性规律

###### Q：lora微调的可调节的核心训练参数

rank秩，缩放因子Alpha，学习率，权重衰减，训练轮次，应用层选择，优化器，批次大小

缩放因子：Alpha 是 LoRA 输出的缩放系数（公式：output = W0×x + (alpha/rank)×(W1×W2×x)），作用是平衡 LoRA 更新与原始模型输出的权重。默认设置 alpha=rank：此时缩放系数为 1（alpha/rank=1），LoRA 更新与原始模型输出 “强度对等”，适合大多数场景；

设置：batch=2,epoch=2,rank=4,weight_decay=0.1,learning_rate = 2e-5,训练数据50条

###### Q：如何评估lora微调模型的效果？

先客观筛选：用 FID、CLIP 分数、文字准确率排除明显不合格的模型（如 FID 过高说明风格偏离，文字准确率低于 70% 需重新微调）；

同一简介分别用 “微调后模型” 和 “未微调的 CogView4” 生成图像，让评审选择更符合 “站内标准” 的结果，统计偏好比例

对比微调模型与即梦、可灵等工具生成的同主题封面，评估在 “专辑场景适配性” 上的优势（如是否更符合中文用户审美）

生成图像被运营 用户审核通过可用的比例，反映 “落地实用性”。
