
## Q1. 多Agent协作架构中，Agent之间的协作调度是怎么实现的？

### 基础框架：LangGraph StateGraph

项目用 `StateGraph` 构建有向图，每个 Agent 是图中的一个节点（`builder.py:17`）：

```
coordinator → price_comparison_checker → extract_hotel_id → extract_hotel_data → operator → reporter
coordinator → ride_comparison_checker → operator → reporter
coordinator → app_type_detection → ...
```

---

### 共享状态传递（State TypedDict）

所有 Agent 通过统一的 `State` 对象通信（`types.py:17`），关键字段：

| 字段 | 作用 |
|------|------|
| `next_node` | 显式指定下一个要执行的节点 |
| `scenario` | 当前场景（酒店比价 / 打车比价） |
| `execution_loop` | 是否处于循环执行模式（给 operator 持续操作） |
| `process_node_list` | 已执行的节点记录链 |
| `messages` | Agent 间传递的消息历史 |

---

### 动态路由：`Command(goto=...)` 机制

每个节点**不直接调用下一个 Agent**，而是返回 `Command` 对象声明下一跳（`nodes.py:414`）：

```python
return Command(
    update={"next_node": "extract_hotel_data", "scenario": "price_comparison"},
    goto="extract_hotel_data"  # 由 LangGraph 框架负责调度
)
```

---

### 入口路由：`coordinator_node` 意图分类

`coordinator` 是统一入口，调用 LLM 分析用户意图，根据返回内容中的关键词决定路由（`nodes.py:761`）：

```python
if "text_price_comparison" in response_content:
    goto = "price_comparison_checker"
if "photo_price_comparison" in response_content:
    goto = "app_type_detection"
if "text_ride_comparison" in response_content:
    goto = "ride_comparison_checker"
```

---

### START 节点的条件边（支持断点续跑）

`route_start` 函数实现了"跳过 coordinator 直接恢复执行"的能力（`builder.py:20`）：

```python
def route_start(state):
    if state["next_node"]:          # 有指定节点 → 直接跳转（断点续跑）
        return state["next_node"]
    elif state["execution_loop"]:   # 执行循环中 → 回到 operator
        return "operator"
    else:                           # 默认 → 从 coordinator 开始
        return "coordinator"
```

---

### 不同 Agent 使用不同 LLM

`config/agents.py` 中配置了 Agent-LLM 映射，按能力分配模型：
- `coordinator` → `basic`（判断意图，不需要视觉）
- `operator` → `vision`（操作屏幕截图，需要多模态）
- `reporter` → `fast`（生成报告，追求速度）

---

### 总结

> 项目以 **LangGraph StateGraph** 为骨架，通过 **共享 State + Command(goto) 动态路由** 实现 Agent 间的协作调度，`coordinator` 负责意图识别与分发，各专属 Agent 处理完后通过 `Command` 显式声明下一跳，框架负责实际调度，整体是一个**有条件跳转的有向图工作流**。

## Q2. Agent编排框架为什么选LangGraph而不是LangChain?

### 项目中两者的实际使用情况

项目**同时使用了 LangChain 和 LangGraph**，各司其职：

- **LangChain**（`langchain-core`、`langchain-openai`、`langchain-community` 等）负责**底层基础设施**：
  - LLM 封装（`ChatOpenAI`、`ChatDeepSeek` 等）
  - 消息类型（`AIMessage`、`HumanMessage`）
  - Tool 装饰器（`@tool`）
  - Callback 回调（`BaseCallbackHandler`）
  - 工具集成（`TavilySearchResults`、`WriteFileTool`）

- **LangGraph** 负责**上层的 Agent 编排与调度**：
  - `StateGraph` 构建工作流图
  - `Command(goto=...)` 动态路由
  - 条件边、循环控制

---

### 核心差异

|      | LangChain       | LangGraph                  |
| ---- | --------------- | -------------------------- |
| 流程模型 | **线性链式**（Chain） | **有向图**（Graph，支持循环）        |
| 状态管理 | 无持久化共享状态        | `State` 对象贯穿全图             |
| 路由能力 | 固定顺序，分支有限       | 条件边 + `Command(goto)` 动态路由 |
| 循环支持 | 不原生支持           | 原生支持循环节点                   |

---

### 为什么编排层选 LangGraph：结合项目的具体原因

**1. operator 需要循环执行，LangChain 不支持**

`operator` 需要反复截图操作 APP 直到完成任务，项目用 `execution_loop` 实现循环回 operator（`scenario_builder.py:117`）：

```python
elif state.get("execution_loop", False):
    goto_node = "ride_operator"   # 循环回到 operator
```

LangChain 的 Chain 是单向的，无法原生表达循环。

**2. 流程需要运行时动态路由**

`coordinator` 根据 LLM 输出决定去哪个节点；`price_comparison_checker` 根据信息完整性决定继续追问还是推进。这种**运行时动态分支**需要 LangGraph 的条件边，LangChain 只能按预设顺序执行。

**3. 跨节点共享状态（支持断点续跑）**

前端可以传入 `next_node` 直接跳过 coordinator 恢复执行（`builder.py:22`）：

```python
if "next_node" in state and state["next_node"] is not None:
    goto_node = state["next_node"]   # 直接跳到指定节点
```

依赖 LangGraph 的全局 `State`；LangChain 没有跨节点持久状态。

**4. 多场景图拓扑不同**

`ScenarioGraphBuilder` 针对酒店比价、打车比价、预订等不同场景构建**不同结构的图**（`scenario_builder.py:42`）。LangGraph 支持动态组装节点和边，LangChain 的 Chain 结构是固定的。

---

### 总结

> LangChain 提供"零件"（LLM、消息、工具），LangGraph 负责"组装"（编排调度）。选 LangGraph 做编排是因为项目的核心难点——**operator 循环操作、动态分支路由、跨节点状态恢复**——都超出了 LangChain 线性链的能力边界，只有 LangGraph 的有向图 + 持久 State + 条件路由才能原生支持。

## Q3. AI Agent的评测体系是怎么搭建的？

**⚠️ 此题不完全适合这个项目。**

项目中有监控体系（基于 Qunar 内部的 Watcher + StatsD），但那是**运维监控**（统计各节点的调用次数、耗时、报错率），不是"AI Agent 评测体系"。

代码里没有发现：
- 针对 Agent 输出质量的自动化评测（如准确率、成功率的离线评估）

- LLM-as-judge 评判机制
- A/B 实验评测框架

### 如果面试官问到这里

诚实的回答是：项目目前的质量保障主要依靠**线上监控指标**（节点成功率、耗时分位值）和**人工测试**，还没有建立系统化的离线 AI 评测体系——这也可以作为项目未来的改进方向来讲。

如果在其他地方（比如公司内部平台）有搭建过评测体系，这道题可以结合那部分经验来回答，但不能直接对应到这个项目的代码。

## Q4. Agent 的记忆系统是怎么实现的？

### 长期记忆

**长期记忆的定义**：跨会话（session）持久存储，能在不同对话之间被召回的记忆。比如用户今天的偏好，明天开新会话时 Agent 仍然记得。

**这个项目没有实现长期记忆。** 历史消息是按 `session_uuid` 取的，新开一个 session 后历史就断了。

---

### 项目实际的记忆层次

**1. RAG（知识库检索）**

严格来说不是"记忆"，而是外部知识。每次用户发消息，在进入 Agent 工作流之前先调用 `/api/chatRag` 做预检索，命中则直接返回，不走 LLM。

**2. 会话记忆（Session Memory）**

每次请求开始时，通过 `/api/historyMessage?sessionUUID=xxx` 从 Java DB 取回当前 session 的完整历史对话，注入到 `messages` 中，让 Agent 能看到本次会话的上下文。

不会有 context 过长的问题——这是**任务型垂直场景**，用户说出酒店名和日期，Agent 操作返回比价结果，一个任务 3～5 轮就结束，不存在上百轮对话的情况。

**3. 会话内中间状态（Redis）**

比价参数（酒店名、入住日期等）提取后存 Redis，key 为 `{session_uuid}_price_comparison`，供同一 session 后续轮次直接读取，不用重复提取。

**4. 单次请求内短期记忆（LangGraph State）**

单次请求内所有 Agent 通过 `State.messages` 共享上下文，节点间追加写入。

## Q5. 项目中使用的基础模型是什么？选型依据是什么？

### 使用的模型

项目所有节点统一使用 **Qwen2.5-VL-72B**，内部自部署（`10.70.128.152:8089`）。Fallback 走公有云：basic 类节点备用 Claude 3.7 Sonnet（OpenRouter），vision 类节点备用 Qwen2.5-VL-72B（OpenRouter）。

---

### 各节点选型依据

**operator → vision（Qwen2.5-VL-72B）**

operator 输入包含手机 APP 截图，需要看图判断页面状态，输出 `click()`、`type()`、`scroll()` 等坐标操作指令，**必须用多模态视觉模型**。

**coordinator / price_comparison_checker / ride_comparison_checker → basic（Qwen2.5-VL-72B）**

这些节点处理纯文本：coordinator 做意图分类，checker 从对话历史中抽取酒店名、入住日期、目的地等结构化信息，核心需求是**中文理解 + 严格 JSON 输出**，不需要视觉能力。

---

### 为什么 basic 和 vision 都用同一个 Qwen2.5-VL-72B？

- Qwen2.5-VL-72B 本身是多模态模型，**既能处理纯文本也能看图**，一个模型覆盖两类任务
- 内部统一自部署，**数据不出内网**，满足企业安全合规
- 相比调用外部 API，**延迟更低、成本更可控**，只维护一套模型降低运维复杂度

---

### 总结

> 项目用 Qwen2.5-VL-72B 作为唯一主力模型统一承担文本和视觉两类任务，核心考量是**数据安全（内网自部署）+ 成本可控 + 运维简单**，所有节点配公有云 fallback 保障可用性。

## Q6. 多Agent系统中，用户发出请求后，整个后端系统的运行流程是什么？

### 总览

```
前端请求 → FastAPI /api/chat → 预处理 → RAG预检 
→ 历史消息注入 → 场景检测 → 构建图 → LangGraph执行 → 结果后处理 → 持久化
```

---

### 逐步拆解

**Step 1：接收请求（FastAPI `/api/chat`）**

前端携带 `messages`、`session_uuid`、`next_node`、`scenario`、`app_list`、`current_app` 等参数发起 POST 请求。

**Step 2：预处理**

图片 URL 格式转换、保存用户图片信息到 userInfo。

**Step 3：RAG 预检（可短路返回）**

消息 channel 为聊天类型且内容为纯文本时，先调用 Java `/api/chatRag` 检索知识库。**命中则直接返回，整个 Agent 流程不启动。** operator 循环时前端发的是截图，不触发此步。

**Step 4：注入历史消息**

每次请求都调用 `/api/historyMessage?sessionUUID=xxx` 取回本 session 历史对话，拼接到当前 messages 前面。

**Step 5：场景检测 + 过滤 app_list**

`request.scenario or detect_scenario_from_messages(messages)`——前端已传 `scenario` 时直接使用，跳过检测。同时过滤 app_list（酒店场景只留去哪儿、美团、携程；打车场景只留高德、滴滴）。

**Step 6：构建图并执行 LangGraph 工作流**

每次请求调用 `build_graph_for_scenario(scenario)` 重新构建 StateGraph，再用 `graph.astream_events(...)` 执行。**每次请求图只运行一轮**，路由由两个机制控制：

- **图内跳转（`goto`）**：节点通过 `Command(goto=...)` 在同一次请求内连续跳转。coordinator 识别意图后直接 `goto="price_comparison_checker"`，不回 `__end__`。
- **跨请求路由（`next_node`）**：operator 每次执行完返回 `goto="__end__"`，同时写 `next_node="operator"`，告知前端下次请求的入口节点。前端执行操作、截图后带着新截图和 `next_node` 再次发请求，实现**客户端驱动的多轮循环**。

operator 的退出完全由 LLM 决定：输出 `finished()` 且是最后一个 app 时，图内直接 `goto="reporter"`；还有下一个 app 时 `goto="__end__"`，前端打开下一个 app 继续。

**Step 7：结果后处理**

`post_chat_llm_result_construction` 注入按钮信息、酒店 business_id 等，组装成 `ChatResponse` 返回给前端。

**Step 8：持久化**

完整结果异步存入 Java DB（`/api/conversation/insert`）。LLM 调用通过 `LLMCallRecordCallBack` 在每次调用结束时实时上报到 `/api/llmCallLog`，Langfuse 同样通过回调实时记录 trace，**两者都不是请求结束才触发**。

---

### 补充子问题：为什么这个项目不使用流式传输？

项目后端存在 `/api/chat/stream` 流式接口，但 Android 前端实际调用的是非流式 `/api/chat`，原因如下：

**1. Action 必须完整才能执行**

operator 返回的是 `{"Action": "click([x1,y1,x2,y2])"}` 这样的结构化 JSON，Android 客户端需要解析完整坐标才能执行点击操作。流式下 JSON 是残缺的，无法解析执行。

**2. 响应是机器消费，不是人类阅读**

流式传输的价值是让用户看到文字逐渐出现。这个项目的响应是给 Android 无障碍服务消费的结构化数据，不是展示性文字，流式对用户体验没有提升。

**3. 路由依赖完整响应**

Android 需要从完整响应里拿到 `next_node`、`final_result` 才能决定下一步走哪个节点处理器，这些字段在响应末尾才确定，流式无法提前路由。

**4. 任务型交互，不是对话型交互**

每轮交互是"截图 → 后端决策 → 执行一个操作"的离散指令模式，每次只需要一个简短的 Action，没有需要流式展示的长文本场景。

---

### 总结

> 一次请求的核心链路：**RAG短路 → 历史注入 → 场景路由 → LangGraph单轮执行 → 持久化**。图内用 `goto` 控制同请求内的节点跳转，跨请求用 `next_node` 告知前端下次入口；operator 的循环本质是**前端驱动的多轮请求**，退出由 LLM 输出 `finished()` 决定；项目使用非流式接口，因为 GUI 操作指令必须原子完整才可执行。

## Q7. 最后输出给用户的是什么格式的信息？整体交互架构是怎样的？

### 整体交互架构

```
H5聊天页(WebView)  ←── HTTP /api/chat ──→  后端Python
       ↕ JS Bridge (tomo.start / callback)
   SystemOperator (HyPlugin)
       ↕ 本地广播(MESSAGE_SENT / SIMULATE_MESSAGE)
 GlobalAccessibilityService
   （UI操作：click / scroll / screenshot）
```

- **H5 ↔ 后端**：H5 聊天页（`TOMOHyWebviewActivity` 打开 `https://qai.qunar.com/chat/#/pages/tripChat`）直接发起 HTTP 请求到 `/api/chat`，获取后端响应后渲染结果、驱动下一步流程
- **H5 ↔ 无障碍服务**：H5 通过 JS Bridge 调用 `tomo.start`（`SystemOperator.kt`），原生层将数据打包成本地广播 `MESSAGE_SENT` 发给 `GlobalAccessibilityService`；无障碍服务执行完 UI 操作后，通过 `SystemOperator.callback()` 经 JS Bridge 将结果回传给 H5

---

### 最终输出给用户的格式

每个节点执行后将结果写入 `final_result[node_name]`，包含 `type`、`show_detail`、`data` 三个字段：

| 节点 | type | show_detail 内容 |
|------|------|-----------------|
| coordinator / checker / extract_hotel_data 等 | `TEXT` | 纯文字提示或追问 |
| operator | `JSON_OBJECT` | `{"Thought": "...", "Action": "click([x1,y1,x2,y2])"}` |
| reporter | `JSON_COMPARISON_TABLE` | 结构化比价结果 JSON |

reporter 最终输出的数据结构（`nodes_utils.py:750`）：

```json
{
  "hotel_name": "汉庭酒店",
  "check_in": "2025-08-01",
  "check_out": "2025-08-02",
  "recommendations": {
    "low_price": {
      "lowest_price": 238,
      "platform": "去哪儿",
      "platform_summary": [{"platform": "去哪儿", "lowest_price": 238}, ...],
      "comparison": [{"platform": "去哪儿", "rooms": [...]}, ...]
    }
  }
}
```

H5 页面收到后端响应，根据 `type` 字段决定渲染方式：`TEXT` 渲染为对话气泡，`JSON_COMPARISON_TABLE` 渲染为比价表格，并附带领券/下单按钮。

---

### 节点之间是串行还是并行？

**节点之间全程串行，不存在节点并行。**

LangGraph 图按 `goto` 依次执行每个节点，前一个节点完成才进入下一个。

唯一的异步发生在节点内部：`app_type_detection` 和 operator 节点在执行时，各自向线程池提交一个**后台酒店信息提取任务**（`nodes.py:1083`、`nodes.py:1917`），提交后立即返回，主流程继续串行推进。后台任务将各平台数据缓存到 Redis，reporter 最后直接从 Redis 读取，无需临时等待。

---

### 总结

> 后端通过 HTTP 与 H5 前端交互，H5 通过 JS Bridge + 本地广播驱动无障碍服务执行 UI 操作。最终展示给用户的是 reporter 产出的 `JSON_COMPARISON_TABLE`——各平台比价数据，H5 渲染为比价表格并附带操作按钮。节点间全程串行，`app_type_detection` 和 operator 节点在执行时向线程池提交后台任务，实现数据缓存与主流程的异步并行。
