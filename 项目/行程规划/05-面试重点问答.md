# 面试重点问答

> 以下是针对 LibAI 项目最高频的面试考察点，每个问题都提供了清晰的回答框架。

---

## 架构设计类

### Q1：为什么选择 LangGraph 而不是 LangChain 的 AgentExecutor？

**答：**
LangChain AgentExecutor 适合简单的单轮工具调用场景。但行程规划有以下复杂度：

1. **多步骤依赖**：意图识别 → 路线规划 → 酒店推荐，步骤之间有数据依赖
2. **条件分支**：用户问酒店和用户问交通走完全不同的处理路径
3. **状态持久化**：需要在多轮对话中记住用户的旅行参数
4. **并行处理**：路线/酒店/交通/天气可以同时查询

LangGraph 提供了**有向图**的执行模型，天然支持以上需求。每个节点是独立模块，状态在节点间流转，方便单独测试和维护。

---

### Q2：整个系统的状态是怎么管理的？

**答：**

使用 LangGraph 的 `State` 对象贯穿整个工作流：

```
单个请求的完整生命周期：

1. 请求进入 → 从 Redis/DB 加载 conversationId 对应的历史状态
2. 初始化 State：
   - messages = 历史消息 + 新用户消息
   - common_state = {conversationId, messageId, userId...}
   - 其他字段初始为空
3. 图开始执行，每个节点读取 State、更新自己负责的字段
4. 图执行完毕 → 将关键状态（消息历史、旅行意图）持久化到 DB/Redis
5. 返回 SSE 事件流给前端
```

持久化的原则：**只存需要跨轮对话复用的状态**，临时计算结果不存储。

---

### Q3：并行规划是怎么实现的？

**答：**

LangGraph 支持 `fan-out / fan-in` 模式：

```python
# fan-out：travel_plan_trigger 节点完成后，4个节点并行启动
graph.add_edge("travel_plan_trigger", [
    "route_planner", "hotel_planner", 
    "transport_planner", "weather_planner"
])

# fan-in：等 4 个都完成后，才进入 travel_planner 汇总
graph.add_edge([
    "route_planner", "hotel_planner",
    "transport_planner", "weather_planner"
], "travel_planner")
```

结合 FastAPI 的异步框架，4 个规划节点真正并发执行，性能提升明显（从顺序的 4x 变为约 1x 最慢任务时间）。

## LLM 应用类

### Q4：Function Calling 是怎么工作的？

**答：**

Function Calling 是 OpenAI/DeepSeek 等模型的能力，允许 LLM 在输出中"决定调用函数"：

1. **注册工具**：在 `function_tools_register.py` 中定义工具的名称、描述、参数 schema
2. **传给 LLM**：每次请求时把工具列表作为参数传给 LLM
3. **LLM 决策**：LLM 如果认为需要查酒店，它不输出文字，而是输出 `tool_call` JSON
4. **执行工具**：系统解析 `tool_call`，调用对应函数（如调用携程 API）
5. **结果回注**：把函数结果作为 `ToolMessage` 加入消息历史
6. **LLM 继续**：LLM 看到工具结果，基于真实数据生成最终回答

**关键点**：这是 LLM 主动决定"需要查哪些数据"，而不是代码硬编码调用顺序。

---

### Q5：流式输出是怎么实现的？

**答：**

三层机制配合：

**第一层：LLM 流式生成**
LangChain 的 `astream()` 方法让 LLM 逐 token 输出，不等待全部生成完毕。

**第二层：事件队列**
`OutputQueueService` 收集 LangGraph 执行过程中产生的各类事件（文本块、卡片数据、错误信息），放入内存队列。

**第三层：SSE 推送**
FastAPI 的 `StreamingResponse` 不断从队列取事件，格式化为 SSE 格式，推送给前端。

不同内容有不同的推送策略：
- 文本内容：逐 token 推（MESSAGE 事件）
- 结构化卡片（行程、酒店）：等完整数据就绪后一次推送（TRIP_CARD 事件）
- 思考过程：独立流，前端可选择是否展示（THINKING 事件）

---

### Q6：如何保证多轮对话的上下文连续性？

**答：**

三个层次：

1. **短期：消息历史** - `State.messages` 包含当前会话的完整消息历史，每个请求都会带上来
2. **中期：上下文存储** - `transcript_context_store` 将旅行意图（出发地、目的地、日期）持久化到 Redis，下次对话直接复用，不用重新解析
3. **长期：消息压缩** - 消息越来越多时，`message_context.py` 的精简器将早期历史压缩为摘要，既保留关键信息又控制 token 消耗

**挑战与应对**：用户可能在一次会话里改变主意（"不去三亚了，改去北京"）。系统通过 `transcriptor_node` 在每轮对话重新解析意图，新意图会覆盖旧意图。

## 业务逻辑类

### Q7：路线是怎么生成的？用的什么算法？

**答：**

路线生成有三种模式，按优先级依次尝试：

**模式1：召回路线**（效果最稳定）
- 从历史行程数据库中检索与用户需求相似的路线
- 基于目的地、天数、风格标签等条件匹配
- 优点：真实用户验证过的路线，质量高

**模式2：AI 生成路线**（最灵活）
- 调用 LLM，传入用户意图和 `generate_travel_guide.md` 提示词
- LLM 输出完整的路线文本
- 解析文本提取景点列表
- 调用 EasyMap 获取景点间距离/时间，验证路线合理性

**模式3：小红书 Embedding 路线**（特色功能）
- 将用户需求转为 Embedding 向量
- 与小红书笔记的 Embedding 计算相似度
- 基于最相似笔记生成路线

召回失败时才降级到 AI 生成。

---

### Q8：小红书导入功能有什么技术难点？

**答：**

**难点1：内容提取**
小红书笔记包含大量无关内容（广告、导航、评论区）。使用 `Readability` 库自动识别主体内容，配合 Jina API 进一步提取。

**难点2：非结构化转结构化**
笔记文字是自然语言描述，格式各异。LLM 需要从"第一天我们去了亚龙湾，然后下午去了天涯海角，晚上吃了海鲜大排档"这类文字中提取出标准的 `{day: 1, spots: ["亚龙湾", "天涯海角"]}` 结构。

**难点3：笔记内容不完整**
用户发的笔记可能只写了景点，没有交通和酒店信息。系统需要自动补全：基于笔记的路线，独立查询合适的交通方式和酒店推荐。

---

### Q9：系统如何处理"调整路线"这种需求？

**答：**

调整路线和新建路线走不同的节点路径：

1. **识别调整意图**：`reception_node` 检测到用户的 `route_id` 参数 + 调整类语言（"把第二天改成..."）
2. **加载原始路线**：从 `route_state` 或根据 `route_id` 重新加载当前行程
3. **进入调整节点**：`adjust_route_sector_node` 接收"原始路线 + 修改指令"
4. **LLM 局部修改**：LLM 只修改被指定修改的部分，保留其他天不变
5. **重新落库**：调用 `route_store` 保存新版本，返回新的 `route_id`

**关键设计**：修改是幂等的，每次修改都生成新的 `route_id`，用户可以回溯历史版本。

## 工程实践类

### Q10：怎么做监控和问题排查？

**答：**

**三层监控体系：**

1. **请求链路追踪**：`TraceMiddleware` 为每个请求注入 `qtraceid`，所有日志和外部调用都带这个 ID，方便串联

2. **LLM 调用监控（Langfuse）**：
   - 每次 LLM 调用的输入 prompt、输出、耗时、消耗 token 数
   - 可以对比不同提示词版本的效果
   - 成本统计

3. **业务指标（QMonitor）**：
   - 路线生成成功率
   - 工具调用成功率
   - 各节点 P50/P95/P99 耗时
   - 报警规则（成功率下降 → 触发告警）

**排查问题流程**：用 qtraceid 在 Langfuse 找到 LLM 调用链 → 查看哪一步的 prompt/response 有问题 → 查 OpenTelemetry 确认外部 API 调用情况。

### Q11：这个系统面对高并发怎么优化？

**答：**

**已有优化：**

1. **异步 IO**：全异步架构，LLM 等待期间线程不阻塞，可以处理其他请求
2. **并行规划**：4个规划子任务并行，而非串行
3. **Redis 缓存**：用户旅行意图、热门路线结果缓存，避免重复调用 LLM
4. **消息精简**：控制每次请求的 token 数，减少 LLM 响应时间

**可进一步优化的方向：**

1. **路线预生成**：热门目的地的路线提前生成好，用户请求时直接返回
2. **结果缓存**：相同意图的请求（同城市、同天数、同标签）复用缓存结果
3. **负载均衡**：多个 LLM API 账号轮询，避免单账号限速

### Q12：提示词是怎么管理和迭代的？

**答：**

**本地提示词**（`prompts/*.md`）：
- 基础版提示词，随代码一起部署
- 修改需要重新部署服务

**Langfuse 远程提示词**（生产使用）：
- 提示词存储在 Langfuse 平台
- 系统启动时拉取最新版本，也支持运行时动态更新
- 可以对比 A/B 两个版本的提示词效果（查看转化率、用户满意度）
- 修改提示词不需要重新部署服务

**迭代流程**：
1. 在 Langfuse 创建新版本提示词
2. 在测试环境验证效果（通过 `/eval` 接口批量测试）
3. 确认效果后推送到生产环境（远程配置生效）

## 项目难点总结表

| 难点 | 解决方案 | 相关文件 |
|------|---------|---------|
| 多步骤 Agent 编排 | LangGraph 有向图 | `graph/builder.py` |
| 长上下文 token 超限 | 消息精简器（STD/MINI/NANO）| `framework/message/message_context.py` |

| LLM 调用成本和耗时 | Langfuse 监控 + 缓存 | `app/qconfig/langfuse_config.py` |
| 提示词频繁调整 | 远程配置 + Langfuse 管理 | `app/qconfig/remote_config.py` |
