## 技术点 1：Function Calling（工具调用）

### 是什么？

Function Calling 允许 LLM 在生成文本时，决定"我需要调用某个工具"，暂停生成，等待工具结果，再继续生成。

### 在项目中的应用

所有可用工具集中注册在 `tools/function_tools_register.py`（70KB，核心文件）。

LLM 在处理请求时可以选择：

| 工具名 | 用途 |
|--------|------|
| `search_hotels` | 搜索酒店 |
| `search_flights` | 搜索机票 |
| `search_trains` | 搜索火车票 |
| `get_weather` | 获取天气预报 |
| `search_pois` | 搜索景点 |
| `web_search` | 网络搜索（Bing/Tavily）|
| `crawl_url` | 爬取网页内容 |

### 执行流程

```
1. LLM 收到用户问题
2. LLM 分析：我需要先查酒店价格
3. LLM 输出 tool_call：{"name": "search_hotels", "args": {"city": "三亚", "checkin": "2026-04-01"}}
4. 系统执行 search_hotels 函数，调用携程 API
5. 系统把结果作为 ToolMessage 追加到消息历史
6. LLM 继续生成，基于真实酒店数据生成推荐文本
```

## 技术点 2：流式输出（Streaming）

### 为什么需要流式输出？

LLM 生成一个完整回答可能需要 5-30 秒。流式输出让用户"看到 AI 在打字"，大幅提升体验。

### 实现机制

```python
# FastAPI SSE 端点（简化示意）
@app.post("/agent/chat")
async def chat(request: ChatRequest):
    async def event_generator():
        async for event in graph.astream_events(state):
            if event["event"] == "on_chat_model_stream":
                chunk = event["data"]["chunk"].content
                yield f"event: MESSAGE\ndata: {json.dumps({'text': chunk})}\n\n"
    
    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

### 不同内容类型的流式处理

```
思考内容 → THINKING 事件（独立流）
   ↓
正文文本 → MESSAGE 事件（主流）
   ↓
结构化卡片 → TRIP_CARD / HOTEL_CARD（完整 JSON，不分块）
   ↓
END_MESSAGE → 通知前端结束
```

**重点：** 思考（THINKING）和文本（MESSAGE）是独立的流，前端可以分开展示。

## 技术点 3：提示词工程（Prompt Engineering）

### 提示词管理策略

项目有 50+ 个 `.md` 提示词文件，每个功能模块一个：

```
prompts/
├── chatbot.md                    # 通用聊天机器人人设
├── generate_travel_guide.md      # 生成行程指南（核心）
├── travel_intent_sector.md       # 旅行意图识别
├── hotel_consult_sector.md       # 酒店咨询
├── transport_consult_sector.md   # 交通咨询
├── question_recommender.md       # 问题推荐
├── text_to_hotel_list_filters.md # NLP转酒店过滤条件
└── adjust_route_sector.md        # 路线调整
```

### 提示词的特点

- 用 Markdown 编写，可读性好
- 支持变量插值：`{departure}` `{arrival}` `{days}`
- 通过 `prompts/template.py` 在运行时填充参数
- 支持从 **Langfuse** 远程管理（不重启服务就能更新提示词）

### 提示词示例结构（generate_travel_guide.md）

```markdown
# 行程规划助手

## 用户信息
- 出发地：{departure}

- 出行日期：{start_date} 至 {end_date}（共{days}天）
- 人数：成人{adult_count}人，儿童{child_count}人
- 兴趣偏好：{select_tags}

## 输出格式要求
请按照以下格式输出每天的行程：
Day X | 主题名称

...
```

## 技术点 4：思考模式（Thinking Mode）

### 是什么？

部分 LLM 支持在生成正式回答之前，先进行内部推理（Chain-of-Thought）。这个推理过程对用户可见，帮助理解 AI 的决策依据。

### 在代码中的配置

```python
# ai_account_config.py
ENABLE_THINKING = True   # 开启思考模式
```

```python
# thinker_node 使用支持 thinking 的 LLM 配置
llm = ChatOpenAI(
    model=config.MODEL,
    extra_body={"thinking": {"type": "enabled", "budget_tokens": 2000}}
)
```

### 前端展示

```
[用户问] 帮我规划三亚5天亲子游

[AI思考中...]  ← 展示 THINKING 事件内容
"用户需要亲子游，应该优先考虑：
1. 海滩类景点（亚龙湾、蜈支洲岛）
2. 亲子乐园（海昌海洋公园）
3. 避开过于拥挤的景点..."

[AI回答]  ← 展示 MESSAGE 事件内容
根据您的需求，我为您规划了以下5天亲子游行程...
```

## 技术点 5：链路追踪（Distributed Tracing）

### 为什么需要链路追踪？

AI 系统的一次请求可能涉及：多次 LLM 调用 + 多个 API 调用 + 数据库操作。需要追踪每个环节的耗时和结果，方便排查问题。

### 实现方式

**TraceMiddleware**（`app/middleware/trace_middleware.py`）：

```python
# 每个请求进入时自动注入追踪ID
@app.middleware("http")
async def trace_middleware(request: Request, call_next):
    qtraceid = generate_trace_id()      # 生成全局追踪ID
    qspanid = generate_span_id()        # 生成当前节点ID
    request.state.qtraceid = qtraceid
    response = await call_next(request)
    return response
```

**多层追踪工具：**

| 工具                | 用途                      |
| ----------------- | ----------------------- |
| **Langfuse**      | 追踪每次 LLM 调用：输入、输出、耗时、成本 |
| **LangSmith**     | LangChain 调用监控          |
| **OpenTelemetry** | 标准化分布式追踪，接入公司监控平台       |
| **QMonitor**      | 自定义业务指标上报（Qunar 内部监控）   |

## 技术点 6：多模态支持

### 图片输入

用户可以上传图片（如旅游景点照片）：

```python
class ChatRequest(BaseModel):
    images: Optional[List[str]]  # 图片 URL 列表
```

图片会被转换为 `HumanMessage` 的多模态内容：

```python
content = [
    {"type": "text", "text": "这个地方在哪里，怎么去？"},
    {"type": "image_url", "image_url": {"url": "图片URL"}}
]
```

### 语音输入

- 前端通过 WebSocket 连接发送音频流
- 服务端使用 **WebRTC VAD**（Voice Activity Detection）检测语音片段
- 调用语音转文字服务，转为文本后进入正常对话流程
- `app/services/bidirectional_tts_service.py` 处理双向语音

## 技术点 7：消息上下文管理

### 问题：token 有限，历史消息会越来越多

解决方案：**消息精简器**（`framework/message/message_context.py`）

### 三个精简级别

| 级别 | 说明 | 使用场景 |
|------|------|---------|
| `STD` | 标准，保留大部分历史 | 短对话 |
| `MINI` | 精简，只保留关键消息 | 长对话 |
| `NANO` | 极度精简，只保留最近几轮 | 超长对话 |

### 消息转换逻辑

- 超过 N 轮的历史消息自动压缩为摘要
- Tool Message（工具调用结果）在使用后从历史中移除
- System Message 始终保留

## 技术点 8：并发与异步

### 全异步设计

FastAPI + Uvicorn 原生支持 `async/await`，所有 IO 操作（LLM 调用、API 请求、数据库读写）都是异步的。

### 并行规划

`travel_plan_trigger_node` 中，路线/酒店/交通/天气四个规划器**并行执行**：

```python
# LangGraph 中使用 fan-out 实现并行
# builder.py 配置
graph.add_edge("travel_plan_trigger", [
    "travel_route_planner_node",
    "travel_hotel_planner_node",
    "travel_transport_planner_node",
    "travel_weather_planner_node"
])
```

4 个 API 并行调用，总耗时 = max(4个耗时)，而非 sum(4个耗时)。

## 技术点 9：MCP（模型上下文协议）

`langchain-mcp-adapters` 集成了 MCP（Model Context Protocol），这是 Anthropic 提出的标准化工具调用协议。

**配置文件：** `config/mcp_config.json`

MCP 允许 LLM 通过标准化协议访问外部数据源和工具，比自定义 Function Calling 更灵活和可扩展。
