## Q1. 行程规划项目由哪几个子项目构成，各自承担什么角色？

行程规划大项目由三个子项目协同构成：**libai（李白）**、**dufu（杜甫）**、**sushi（苏轼）**。

**libai（李白）** 是 AI 对话层，基于 Python + FastAPI + LangGraph，对外暴露 SSE 流式聊天接口（`/chat/sendMessage`、`/aiChat/sendMessage`），通过 LangGraph Agent workflow 理解用户意图并调度工具（航班搜索、必应搜索、爬虫等），是整个系统的入口和 AI 编排中枢。

**dufu（杜甫）** 是核心业务后端，基于 Java + Spring Boot，提供 POI 管理、路线规划、酒店查询、机票/火车票查询、小红书内容抓取与分析、评论情感分析等具体业务能力，集成 LangChain4j + 通义千问处理细粒度 AI 任务，是 libai 的数据/服务提供者。

**sushi（苏轼）** 是评测与运营平台，基于 Java + Spring Boot，专注离线评测，提供数据集管理、评测集管理、测试记录、每日评分统计、排行榜等功能，不在主链路上。

三者关系：用户 → libai（对话/规划）→ dufu（业务数据），sushi 独立对 AI 效果做监控评测。

## Q2. 多Agent协作架构中，Agent之间的协作调度是怎么实现的？

### 核心路由机制

项目 90% 的路由通过**节点内 `Command(goto=...)` 在运行时自主决定下一跳**实现，而不是 `add_conditional_edges`——后者只用在图的入口 START 处（构建期静态定义）。这是两种方式的关键区别，本项目主要用前者，每个节点自主决定下一步，更灵活。

### 两层图结构

- **父图**：`START` 同时触发 `parallel`（子图）和 `thinker`，两者完全并行，thinker 独立做问题思考，不阻塞主流程
- **子图**：包含所有业务 Agent 节点，是实际调度核心

### Reception 层（v5 模式）

入口路由到 `trip_react_reception_node`，节点内有四级意图识别，优先级依次递减：

1. **`pre_router_sector` 规则路由**：看请求的结构化参数（extMap 字段，如 `planTrigger`、`invokeSource` 等），调用方已明确意图，直接跳转，不分析文本
2. **关键词路由**：仅首轮触发，用正则匹配用户消息文本，规则来自远程配置，命中直接路由
3. **BERT v2 分类**：置信度 ≥ 0.85 才采用结果
4. **LLM 分类**：兜底

识别完成后并行双发：`goto=["trip_react", "question_association"]`，业务执行和问题推荐同时触发，不影响主链路响应时间。

### trip_react 执行层

`trip_react_node` 根据 reception_result **动态绑定对应工具集**，调用 LLM with Function Calling，再根据工具返回结果决定下一跳（直接 END、走 `trip_function_call_output` 格式化卡片输出、或转发其他节点）。

### travel_plan_trigger 的三条出口

`travel_plan_trigger` 是路由决策节点，根据路线召回结果走不同分支：

1. **embedding 召回**（`embedding_by_xhs_note`）→ `generate_route_use_embedding_router`
2. **纯 AI 生成**（召回为空或返回 str）→ `generate_route_use_embedding_router`
3. **标准召回**（返回正常路线列表）→ `travel_route_planner`

`travel_route_planner` 完成后扇出目标动态决定，`travel_planner` 标记 `defer=True`，等所有并行分支汇聚后才执行。

### 核心机制总结

| 机制 | 实现方式 |
|------|---------|
| **主要路由** | 节点内 `Command(goto=...)` 运行时决定 |
| 图入口路由 | `add_conditional_edges`（仅 START） |
| 并行扇出 | `Command(goto=["node_a", "node_b"])` |
| 并行汇聚 | `add_node(..., defer=True)` |
| 节点间通信 | `Command(update={...})` 写 / `state.get(...)` 读共享 State |

## Q3. AI Agent 系统的评测系统是怎么实现的？

sushi 中有两套独立的评测流程，侧重不同。

### 流程一：TripEvalSet — 评测"AI生成的行程路线质量"

**评测的是什么：** 用户发起行程规划后 libai 生成的行程路线质量好不好（主观质量）。

**数据来源：** libai 生产环境每次生成行程，dufu 自动将「用户意图 + 生成路线结果」异步写入 `TripEvalSet` 表，是对真实生产流量的自动采样。

**评测触发：** `TripEvalTask`（QSchedule 定时任务）轮询 DB 中 status=SAVED 的记录，分页批量捞取，每条用 `CompletableFuture` 并发处理。

**评测方式：用 LLM 评 LLM 的输出（Strategy 模式）**
通过配置的 Spring Bean 名动态加载 `TripEvalSetProcessor` 实现类。以 `RouteProcessor` 为例：
1. 将行程数据整理成结构化 Prompt：用户意图 + POI 列表 + POI 详情 + 相邻 POI 距离信息
2. 遍历多个 promptKey（每个对应一个评测维度），分别调用 LangChain4j LLM 打分
3. 多维度分数取平均，写回 `TripEvalSet.score`

**评测指标：** 由 Prompt 配置决定，多维度打分（路线合理性、POI 顺序地理合理性、是否满足用户意图等），最终是一个数值分。

---

### 流程二：Test/Dataset — 评测"特定功能的输出是否正确"

**评测的是什么：** 针对某个具体功能（如 POI 聚合），验证 AI 实际输出和标准答案是否一致（客观正确性）。

**数据来源：** 人工维护的标注数据集，带版本号，每条数据包含 `input`（输入）和 `expectedOutput`（标准答案）。

**评测流程：**
1. 人工创建测试模板（绑定数据集 + 模型 + 类型），手动触发运行
2. sushi 创建 `TestRecord` + 批量初始化 `TestCase`，通过 **QMQ 消息**将每条用例发给 libai
3. libai 真实执行 AI 推理，将结果通过 QMQ 发回
4. `EvalsResultConsumer` 消费结果，对比 `actualOutput` 与 `expectedOutput`

**评测指标：**
- 默认：字符串精确匹配，完全一致 `passed=1`
- `poiAggregate` 类型：提取两侧 poiId 列表做结构化比对
- 汇总为整个 TestRecord 的**通过率**

---

### 两套对比

| | TripEvalSet | Test/Dataset |
|---|---|---|
| 数据来源 | 生产流量自动采样 | 人工维护标注数据集 |
| 评分方式 | LLM 多维度打分 | 精确匹配 / 自定义规则 |
| 触发方式 | QSchedule 定时轮询 | 手动触发，QMQ 异步执行 |
| 评测目标 | 监控生产行程质量（主观） | 验证功能正确性（客观） |

## Q4. Agent 的记忆系统是怎么实现的？

### 1. 会话内节点间共享：LangGraph State

同一次请求内，所有节点共享一个 `State` 字典，通过 `Command(update={...})` 写入、`state.get(...)` 读取，请求结束即消失。

### 2. 多轮对话上下文：历史消息持久化

**存储**：每轮对话结束后，用户输入和 AI 的各类输出按 `conversationId` + `messageType` 写入 DB。不同类型的 AI 回复（行程卡片、酒店推荐、交通信息等）对应不同的 `messageType`。

**加载（STD）**：下一轮请求进来时，`get_history_messages()` 从 DB 查询该会话历史，最多回溯 4 轮（可配置），用 **STD 级别**将每条历史消息完整转换后存入 `state["messages"]`，供业务执行节点（如 `trip_react_node`）直接使用。STD 保留完整内容——行程卡片含每天的 POI 描述、酒店价格评分、交通编号时间；酒店推荐含名称、位置、档次、标签、推荐理由；交通信息含编号、价格、出发到达站时间。

### 3. 意图识别和思考节点的轻量上下文（NANO）

`trip_react_reception_node`（意图识别）和 `thinker_node`（深度思考）不直接使用 `state["messages"]`，而是在节点内部自己从 `state["history_message_converters"]` 取历史消息，用 **NANO 级别**重新组装一份轻量上下文喂给 LLM。

NANO 的压缩规则：**AI 的历史回复**被压缩为一句固定摘要（如"AI展示了行程规划信息，详情略。"、"AI展示了推荐的酒店信息，详情略。"），**用户消息不压缩，原文保留**。

这样做的原因：意图识别只需要判断"用户这句话想做什么"，深度思考只需要了解对话大致走向，两者都不需要知道历史 AI 回复里具体的价格、编号等细节。用 NANO 把 AI 的大块结构化数据替换为一句话摘要，大幅减少 token 消耗，同时保留用户消息原文确保意图判断准确。

> 注：代码中定义了 MINI 级别，但从未被调用，是死代码。

### 4. 行程数据引用：routeId

行程规划完成后，调用 dufu 接口将完整行程数据存入 dufu 的数据库，返回一个 `routeId`，消息表只存这个 ID。下一轮需要将行程作为历史上下文时，`TravelPlannerConverter` 用 `routeId` 重新调 dufu 接口拉取完整数据，再按 STD/NANO 转换后注入上下文。

**注意**：这个项目没有跨会话的记忆，每个新会话都是全新开始。

### Q4-sub. 会话多轮时，跨轮次的上下文记忆是怎么做的？

项目里有两套机制，一套在用，一套已下线。

**在用的：历史消息持久化**

每轮对话结束后，用户输入和 AI 的各类回复按 `conversationId` + `messageType` 存入消息表。下一轮请求进来时，`get_history_messages()` 按会话 ID 查询历史，最多回溯 4 轮，组装成 `state["messages"]` 供业务节点使用。

**已下线的：transcriptor 节点**（可以说未下线一直在使用）

transcriptor 是一个意图压缩节点，在子图入口与 reception 并行触发。它拿到当前对话历史后，用 LLM 将用户的旅行意图提炼成一个结构化的 `TranscriptContext` JSON 对象，包含 `hotel_intent`、`flight_intent`、`trip_intent` 等分类意图，每类下面有 `constraints`（正向约束）、`negative_constraints`（负向约束）、`background_memory`（被覆盖的旧约束）、`curr_travel_intention`（词槽）等字段，这些字段都是 LLM 按 prompt 规则填充的，不是单独的数据库字段。

跨轮次机制：本轮 transcriptor 运行完后，TranscriptContext 作为完整 JSON 字符串 **insert 一条新记录**存入 dufu 数据库。下一轮请求初始化时，`query_transcript_context()` 按 `conversationId` 取 `create_time` 最新的那条，加载到 State 中供 transcriptor 作为"上一轮的意图上下文"输入。

约束的继承和更新规则由 prompt 控制：没有冲突时，上一轮的所有约束原样继承；用户修改某个条件时，旧约束下沉到 `background_memory`（仅作参考，不作为有效约束），新约束进入 `constraints`；负向约束（如"不要青年旅社"）同理，无冲突则一直继承，仅有新消息明确反向才会被覆盖。

## Q5. 用户发出一个请求后，整个后端系统的运行流程是什么？

**① 接入层**

FastAPI `/aiChat/sendMessage` 接收请求，经 `TraceMiddleware` 注入链路追踪 ID，进入 `run_agent_workflow()`。

**② 请求初始化**

- 生成 `conversation_id`（没有则新建）、`message_id`、`workflow_id`
- `get_history_messages()` 从 DB 加载最多 4 轮历史消息，组装后存入 State
- `query_transcript_context()` 加载上一轮 TranscriptContext（开关控制）
- 将当前用户消息先写入 DB
- 调外部接口获取用户地理位置（每次请求都调），结果注入 State，用于 transcriptor prompt 中的默认出发城市，以及 Langfuse 链路追踪 tag
- 尝试从 Redis 读取 RouteState 缓存：仅首轮对话且消息内容命中远程配置正则 pattern 才生成 cache key，命中则直接复用上次行程规划结果，同时跳过 thinker
- 判断是否跳过 thinker（点击标签、固定入口词等场景直接跳过）

**③ LangGraph 图启动**

将完整 State 传入 `graph.astream_events()`，图开始运行并返回异步事件流：

- 父图 START 同时触发 **thinker** 和 **子图**，两者完全并行
- thinker 做深度思考，流式向前端推送 thinking 内容
- 子图 START 同时触发 **trip_react_reception_node** 和 **transcriptor**

**④ Reception 意图识别**

四级优先级：pre_router_sector 规则路由 → 关键词路由（仅首轮）→ BERT v2（置信度 ≥ 0.85）→ LLM 兜底。识别结果写入 State，然后并行双发：`trip_react` + `question_association`。

**⑤ trip_react 执行**

`trip_react_node` 从 State 读取 reception 结果，根据意图类型从远程配置读取对应工具 key 列表，只将该意图相关的工具 Schema 绑定给 LLM（动态绑定工具集，避免干扰和浪费 token）。

LLM 收到用户消息 + 工具定义后，通过 Function Calling 决定调用哪个工具、传什么参数，返回结构化的 `tool_calls`。系统执行工具（实际调用 dufu API），根据工具执行结果决定下一跳：
- 工具有对应卡片格式（如火车票、酒店列表）→ `goto="trip_function_call_output"` 走卡片输出节点
- 行程规划工具 → 转发至 `travel_plan_trigger`，按召回类型走三条分支，各 planner 节点并行查询酒店、交通、天气，汇聚后生成行程
- LLM 直接回复或工具返回字符串 → `goto=END`
- 意图不匹配工具 → `goto="chatbot"`

**⑥ 流式输出**

`process_events()` 监听 astream_events 事件流，每个事件按节点名路由到对应 handler，handler 将 LangGraph 内部事件格式化为 SSE dict，写入当前请求独占的内存队列（`OutputQueueService`）。thinker 未结束前队列不消费；thinker 结束后 `consume_output_queue()` 开始消费，通过 SSE 实时推送给前端。

**⑦ 收尾**

workflow 完成后：transcriptor 结果 insert 一条新记录到 DB；所有 AI 回复消息批量写入 DB；推送问题推荐结果；发送 `CONVERSATION_ENDING` 事件关闭 SSE 连接。

## Q6. 为什么把酒店、景点等拆成独立的节点，而不是作为工具调用？

真正的问题是：**为什么不在 trip_react_node 里直接用工具调用处理所有业务，还要额外拆出独立节点？**

**1. 各业务互不干扰，单独维护**

酒店、交通、路线的查询逻辑、外部服务、数据结构完全不同。拆成独立节点后每个节点只负责一件事，出问题容易定位，业务逻辑互不影响，各自独立演进。

**2. 每个节点有自己专属的 prompt**

酒店节点的 prompt 只描述"把自然语言解析成酒店查询参数"这一件事，输出格式固定，few-shot 全针对酒店场景，可以单独调优。如果用一个大 prompt 覆盖所有领域，LLM 每次要处理大量无关内容，容易出错，改一处也容易影响其他部分。

**3. 依赖关系用图结构表达更清晰**

行程规划场景下，酒店和交通的查询都依赖路线规划的结果——`travel_hotel_planner` 和 `travel_transport_planner` 都从 State 读路线数据。用图节点来表达这种依赖（路线节点先跑 → 写结果到 State → 下游节点读取），比在一个大节点里手动管理调用顺序和中间变量更直观。

**4. 酒店咨询需要多轮参数补全**

`hotel_consult_sector_node` 里 LLM 解析参数后检查 `ready` 字段，不完整就追问用户，完整才继续查询。工具调用是一次性的，LLM 决定参数后直接执行，没有"检测缺失 → 等用户补充 → 继续"的能力。

## Q7. 多节点拿到数据后怎么合并产生行程方案？最后输出给用户的是什么格式？输出队列的意义在哪里？

### 数据合并

route_graph 是纯串行 pipeline，没有多节点数据合并的问题。数据在各步骤中逐步累积在 State 里：

```
build_param → select_processor → get_route_ref → route_ref_to_json → standardization → summary → after_summary
```

到 `summary` 步骤时 `route_store_result` 已经完整组装好，直接输出。

### 最终输出格式

`summary_output` 在最后节点中连续调用三次 `dispatch_component_out_message`，向 SSE 通道发出三条独立消息：

| 顺序 | eventType | 内容 |
|---|---|---|
| 第1条 | MARKDOWN | 标题文字："上海3天2晚行程推荐" |
| 第2条 | MARKDOWN | 推荐语段落文字（LLM生成）|
| 第3条 | TRIP_CARD | 完整行程 JSON（每天景点顺序等结构化数据）|

三条消息连续入队，前端几乎同时收到，但是三条独立 SSE 消息，前端可以收到第1条就渲染标题，不用等第3条。

### 输出队列的意义

对 TRIP_CARD 这种必须等完整数据的事件，流式本身没有加速意义——前端拿到半份数据也构不成卡片。

队列的真正作用是**统一的事件推送框架**：同一条 SSE 连接要服务多种事件类型——普通对话（LLM逐token流式）、thinking过程（流式）、MARKDOWN（流式）、TRIP_CARD（整块）。统一走队列是合理的架构选择，不需要为 TRIP_CARD 单独拆一套同步返回机制。

## Q8. 行程规划中，景点之间的距离关系如何处理？酒店如何获取？

### 一、景点距离的两种处理方式

**方式1：预计算距离矩阵，辅助 LLM 规划（RAG 召回路径）**

在 RAG 召回路径（`handleRagRef`）中，dufu 会对所有候选景点两两计算直线距离，构建一个完整的 N×N 距离矩阵（JSON 嵌套 Map 形式，key 为"景点名(ID)"，value 为到其余各景点的距离，同时包含米和公里两种单位）。这个矩阵连同景点详情、天气信息一起作为结构化 prompt 传给 LLM，辅助 LLM 做景点的天粒度分组和游览顺序编排。

这样设计是因为职责分离：LLM 做不到可靠的球面三角计算，工程代码提前算好距离，LLM 只需要做语义推理——"这两个景点只有 500 米可以连着走，那两个 15 公里最好分天"。

**方式2：TSP 算法重排景点顺序（用户编辑触发）**

用户手动编辑某天行程后，可触发"魔法棒"功能（`dayPathAutoFix`），用 TSP 算法对当天景点的游览顺序进行优化，使总路程最短：≤15 个景点用 DP 精确求解，>15 个景点用贪心+2-opt 启发式。重排后固定顺序为：**交通 → 景点（TSP 排序）→ 酒店**。

两种方式的两点距离底层都调用同一个计算函数 `OpenPathTspSolver.distMeters`。

### 二、Haversine 公式：经纬度如何算成米

地球是球体，两个经纬度点之间的距离是球面上的弧长，不能直接用经纬度差做平面勾股定理。

Haversine 公式的思路是：
1. 把经纬度差转换为弧度
2. 用球面几何的半正矢公式算出两点在球面上张开的圆心角
3. 圆心角 × 地球半径（6371km）= 弧长距离

之所以不用简单的平面公式，是因为经度线并不等距——在赤道上经度 1° 约 111km，但在北京（纬度约 40°）经度 1° 只有约 85km，纬度越高经度间距越小。Haversine 通过引入 cos(纬度) 修正项来补偿这个收缩，保证不同纬度的计算精度。最终误差在 0.5% 以内，对城市内景点排序完全够用。

### 三、根据景点查询酒店的逻辑

酒店查询以**当天最后一个景点**的坐标为位置参数（兜底取次日第一个景点），向外部去哪儿酒店接口发起搜索。选取这一个坐标点而非整天景点的中心点，是因为用户当天游览结束后就近入住更合理。

接口返回多个候选酒店后，dufu 侧取其中**最多 2 个**：
- 第一个：**综合推荐**（外部接口综合排序第一）
- 第二个：**价格最低**（从返回列表中取价格最低的）

复杂的质量过滤和排序逻辑完全交给外部酒店服务，dufu 只做结果取用，不做额外筛选。

## Q9. 路线生成后，酒店和交通并行执行，需要共享什么上下文？可能产生什么冲突？如何解决？

### 一、共享的上下文（只读，无冲突）

两个节点都从 State 中读取以下数据：

| 数据 | State key | 说明 |
|---|---|---|
| 路线结果 | `travel_route_planner_recommend_routes` / `travel_planner_param_by_travel_route_planner_node` | 酒店需要每天最后景点坐标；交通需要目的地城市 |
| 用户意图 | `ext_map`（含 `trip_planner_get_route_request`）| 出发地、目的地、日期、天数等 |
| 用户信息 | `c` | cookie，调用下游接口鉴权用 |

这些都是**只读共享**，并行读取不产生任何冲突。

### 二、冲突点：`content` key 被两个节点同时写

hotel 和 transport 节点都往同一个 `content` key 写各自的卡片数据：
- hotel 写：酒店推荐列表 JSON
- transport 写：机票/火车推荐列表 JSON

LangGraph 并行分支每个节点完成时立即把 update 写入 checkpoint。`content` 如果是普通字段（`str`），后完成的分支直接覆盖先完成的，结果不确定——谁后写谁赢，另一份数据丢失。

### 三、解决方案

**方案一：拆成独立 key（最直接）**

两个节点各写各的 key（`hotel_content`、`transport_content`），互不干扰，`travel_planner` 分别读取两个字段组装输出。

**方案二：Reducer 定义合并语义（LangGraph 惯用方式）**

```python
from typing import Annotated
import operator

class State(TypedDict):
    content: Annotated[list, operator.add]
```

hotel 写 `["酒店卡片JSON"]`，transport 写 `["交通卡片JSON"]`，框架收到两个 update 时自动调用 `operator.add` 做列表拼接，`travel_planner`（`defer=True`，等所有分支写完才执行）读到完整合并后的列表，两份数据都保留。

Reducer 把"覆盖"变成"追加"——不是新值替换旧值，而是新值追加到旧值后面。拆 key 适合两份数据语义完全不同的场景；Reducer 适合多个节点写同类型、需要聚合的数据。

## Q10. 并行启动多个 Agent 节点 vs 单 Agent 并行工具调用，各自的优缺点是什么？

> 前提：信息不足时先追问用户，信息完整后才触发并行执行，因此执行阶段不存在"部分节点需要追问"的混乱情况。

### 独立并行节点

**优点：**
- **各自独立的专属 prompt**：酒店节点有针对酒店推荐优化的 prompt（强调性价比、位置、评分权重），交通节点有专门处理航班/火车的 prompt（强调时刻表、中转、价格区间），各自在各自领域表现更好
- **故障隔离**：酒店接口超时不影响交通节点，各自有独立错误处理，可以返回部分结果
- **业务逻辑内聚**：复杂的筛选、格式转换逻辑封装在各自节点内，互不干扰，新增产品维度只需加节点
- **独立可配置**：不同节点可以用不同模型、不同限流、不同超时策略

**缺点：**
- **状态协调复杂**：并行写同一个 `content` key 需要 Reducer 或拆独立 key，`defer=True` 等框架机制带来额外负担
- **综合阶段不自动**：跨领域推理需要汇聚节点专门设计，不是自动的

### 单 Agent 并行工具调用

**优点：**
- **综合阶段上下文统一**：同一个 LLM 发出工具调用，也读取两份结果做综合，天然可以做跨领域联合推荐（如根据多个航班选项和多个酒店选项综合推荐最优组合）
- **状态简单**：一个 Agent 写一次 State，没有并行写冲突，不需要 Reducer
- **推理成本低**：一次 LLM 决策 + 一次 LLM 综合，共两次推理

**缺点：**
- **prompt 无法专化**：一个 prompt 要同时覆盖多个领域，要么 prompt 臃肿，要么各领域表现不如专属 prompt 精准
- **单点故障**：一个工具失败，LLM 拿到残缺上下文，处理方式完全依赖 LLM 判断，可控性不如节点级错误处理
- **扩展性差**：新增产品维度需要修改同一个 Agent 的工具列表和 prompt，耦合度高

### 注意：并行阶段均无法跨领域推理

无论哪种方式，工具/节点**执行阶段**都是并行的，谁都拿不到对方的结果，无法互相影响。跨领域推理只发生在**综合阶段**（两份结果都返回后）。真正有执行依赖（如先查航班时间再查酒店）必须用串行，不能并行。

### 选择依据

各领域业务复杂、需要专属 prompt 精准处理、需要独立容错 → 独立并行节点；查询逻辑简单、需要综合多个结果做联合推荐、成本敏感 → 单 Agent 并行工具调用。

## Q11. 将多个子 Agent 打包成一个 Agent 后，之前单独复用的通用性会不会受到影响？Agent 数量会不会膨胀？

不会导致 Agent 数量膨胀，LLM 有能力根据上下文判断当前场景只需要哪些工具，只调用相关工具，不调用无关工具。

真正受影响的是 **Prompt 的专化程度**。

独立的酒店 Agent 有一套专门为酒店推荐场景优化的 prompt——角色定义、约束条件、输出格式都围绕酒店逻辑设计，在任何需要查酒店的场景下直接复用，效果稳定。

合并成一个行程规划 Agent 后，prompt 需要同时覆盖酒店和交通两个领域，变成一个更通用的规划 prompt。当这个 Agent 被用在"只修改酒店"这类单一场景时：

- 行程规划的背景描述和约束（比如"结合路线选酒店位置"）成了干扰信息
- 输出格式是为完整行程规划设计的，不一定符合单独修改酒店的预期
- prompt 越通用，在特定场景下的表现越不如专属 prompt 精准

所以合并后的代价不是 Agent 数量膨胀，而是**各场景下 prompt 的精准度下降**，专属场景的表现不如独立 Agent。独立节点的价值正在于此：prompt 高度专化，复用时不需要为不同场景重新调整。

### Q12-sub2. 小红书帖子入库全流程（数据采集 → 分类提取 → Route 入库）是怎样的？

#### 一、数据采集与入库

**全量同步**把400万帖子按时间切成小窗口，上午（0-12点）和下午（12-24点）用不同的窗口大小，因为不同时段发帖密度不同。每次定时任务只处理最近的 N 个窗口，防止单次压力过大。每个窗口调一次 Materials API，批量写入 DB，写入用 `insertOrUpdate` 保证幂等，重复跑不产生脏数据，写入时 `status=0`。

**增量同步**固定处理昨天全天，复用相同的窗口切分逻辑，每天补充新帖子。

写入 DB 和后续处理**完全解耦**，单独一个扫描任务轮询 `status=0` 的记录提交到下游，两阶段互不影响。

#### 二、分类

扫描到待处理帖子后，先过分类线程池。

- 模型：本地部署的 **QWen3-8B**（chat 生成模型），prompt 从 QConfig 配置中心动态加载
- 输入帖子标题和正文，LLM 输出类别枚举（路线攻略 / 单景点种草 / 无关内容等）、置信度、判断依据
- LLM 输出自然语言，业务层解析成结构化结果写回 DB，更新 `category` 和 `status`
- LLM 调用前有 Rate Limiter 控速

分类结果决定帖子走哪条下游流水线。

#### 三、POI 提取

针对分类为"含景点"的帖子，进入 POI 提取线程池。

- 模型：本地部署的 **QWen3-8B-poi**（独立实例，不同端口），`maxTokens=12000`，是分类模型的3倍，因为要输出多个景点的结构化 JSON
- prompt 从 QConfig 加载，LLM 从正文提取景点名称及相关信息
- 提取出的是用户写的非标准名称，经过 **POI 挂靠**：调用内部服务（携程 POI 数据，Redis 存储），把非标准名称映射到系统内标准 POI ID
- 挂靠成功的结果落库，至此 POI 提取结束

#### 四、路线提取 → sushi 评估 → Route 入库

针对分类为"路线攻略"的帖子，进入路线提取线程池。Route 记录不是在这里创建的，要经过 sushi 质量评估才创建。

**Step 1：路线结构提取**

- 模型：**DeepSeek-V3.2（阿里云）**，通过 `AiAccountConfig` 从 QConfig 的 `ai-account.properties` 动态读取，字段为 `ai.xhs_route_extract.*`，支持热更新
- LLM 从正文提取多日行程结构：第几天、去了哪些地方、顺序是什么
- 提取结果同样经过 **POI 挂靠**，景点名映射到标准 POI ID
- 生成 **TripRoute**（中间态，不是最终 Route）

**Step 2：sushi LLM 质量评估（第一道质量门槛）**

- dufu 调 sushi 的 `/insertRouteList`，把 TripRoute 推过去
- sushi 用 LLM 对路线质量打分，评估行程合理性、POI 完整度、序列逻辑等
- **评分 < 3.5：丢弃**，dufu 不会创建任何 Route 记录
- **评分 ≥ 3.5**：sushi 回调 dufu 的 `/tripRouteToRoute`

**Step 3：dufu 创建 Route 记录**

- 收到回调后，dufu 才正式创建 Route，`status=SCORING`
- 聚合任务定时计算热度/POI/序列综合得分，自动推进 SCORING → CHECKING
- CHECKING 进入人工审核队列，运营在 sushi 后台审核通过后调 dufu `/audit`，变为 DISPLAY，进入召回

#### 整体漏斗

```
400万帖子
    ↓ 分类过滤（无关内容丢弃）
    ↓ 路线提取（LLM无法抽取有效行程的丢弃）
    ↓ sushi质量评分 < 3.5 丢弃
  Route(SCORING)
    ↓ 聚合任务自动推进
  CHECKING
    ↓ 人工审核
  DISPLAY（进入召回）
```

### Q12-sub3. 小红书帖子的向量化与存储是怎么做的？

#### 一、整体架构

存储分两套并行：**Marco Polo（向量库）+ ES（关键词索引）**，分别服务于不同的召回路径，两者独立写入、独立查询，且都与帖子入库和审核流程完全解耦，由各自独立的定时任务驱动。

#### 二、向量化流程（Marco Polo / pgvector）

`XhsNoteEmbeddingTask` 定时任务按城市 + 类别分页扫描 DB，提交到独立的 embedding 线程池并发处理。

**文本处理**：没有分块，整篇帖子作为一个整体输入 embedding 模型，一篇帖子对应一个向量。小红书帖子文字有上限，约 1000+ token，QWen3-Embedding-4B 支持 8K 输入，完全不需要切分。

**Embedding 模型**：本地部署的 **QWen3-Embedding-4B**，输出 2560 维向量，向量连同 noteId、city、noteContent、poiList 一起 POST 给 Marco Polo 的 `/note_embedding_insert` 接口存入 pgvector。

#### 三、ES 存储

`XhsNoteEsSyncTask` 定时任务按城市 + 类别把 DB 里的帖子同步到 ES，索引名 `xhs_note_condition`。

**为什么按城市 + 类别分批**：把大任务拆成小任务，单次失败只影响一个城市，可单独重跑；同时只同步路线类帖子（category=2），无关分类不进索引，保持索引干净。

**翻页方式**：游标翻页，取上一页最后一条记录的 id 作为游标（`WHERE id > lastId`），避免 offset 深翻页时数据库扫描大量记录导致性能劣化。

**核心字段设计**：`content` 字段用 **MultiField 多字段映射**，同一段文字同时建两份倒排索引：

- **`content`（主字段）**：`ik_smart` 粗粒度分词，BM25 相关性查询，查询时 boost=2.0，精确匹配优先
- **`content.search`（子字段）**：写入时 `ik_max_word` 细粒度分词，查询时 `ik_smart`，boost=1.0，切词更细覆盖更多查询词

存储代价是同一段内容建了两份倒排索引，空间翻倍。召回时两个字段都查，至少命中一个即可，子字段兜底提召回率，主字段高权重保排序质量。

其他字段：`city` 精确过滤（Keyword），互动数据和时间字段支持排序和过滤。

#### 四、数据时效性

两个定时任务都是异步批量同步，路线审核通过变成 DISPLAY 后不会立刻出现在召回结果里，要等定时任务下次跑完才生效，存在一定延迟。

## Q13. 智能调整路线是怎么做的？（只看智能模式）

#### 前提：数据从哪来

首次生成路线时，`route_store` 接口把**路线数据**和**用户意图**（出发地/目的地/天数/日期/人数/风格标签）绑定同一个 `routeId` 一起落库。用户意图是更早在 `travel_intent_sector_node` 由 LLM 从对话中提取的结构化数据（`curr_travel_intention_dict`）。

#### 触发

前端传 `ext_map.llm_adjust_route_flag=true` + `ext_map.routeId`，在 `pre_router_sector` 短路，直接进 `adjust_route_sector_node`，配置开关（`intelligent_adjust_route_switch` 或白名单）命中后走智能模式。

#### 执行链路

**Step 1：取回原始数据**

```
route_detail_more(routeId)
→ routeDetail（原路线结构）
→ userIntent（原始意图，当初落库时存的）
```

**Step 2：LLM 重构路线**

单次调用 `LLM(intelligent_adjust_route)`，强制 `json_object` 输出。

Prompt 注入三个信息：原路线、原意图、用户当前这条修改指令。

Prompt 里写了 Chain of Thought 三阶段，LLM 在一次推理中内部完成（不是代码分阶段调用，完全由 prompt 驱动）：
- Phase 1：对比原意图和用户指令，只更新用户明确提及的维度（天数/日期/目的地/人数/风格），其余原样保留
- Phase 2：基于新意图重排行程，增删改景点，保证地理顺路、同日不跨城市、天数严格一致
- Phase 3：格式化输出，`full_desc` 用 `[[景点名]]` 引用格式，`ready=true` 表示成功

**Step 3：景点挂靠**

```
save_extract_route(新路线)
→ 后端把景点名称匹配到 poiId

query_poi_link_count(poiIdList)
→ 过滤掉内容不足的无效 POI

re.sub([[景点名]] → 有效 POI 生成跳转链接，无效的保留纯文本)
```

**Step 4：落库**

```
do_route_store(updated_travel_intention, state)
→ 用更新后的意图 + 新路线重新写入数据库
→ 生成新 routeId（新 userIntent 也随之落库，供下次调整继续使用）
```

**Step 5：推送前端**

```
① Markdown 标题（成功提示文案）
② Markdown 正文（full_desc，含景点跳转链接）
③ TRIP_CARD（新路线卡片，含新 routeId）
```

## Q14. 小红书笔记导入的逻辑思路是什么？并行是如何实现的？

整个流程分两轮交互完成，只看主动点击"导入小红书笔记"按钮、走智能生成的主路线。

### 第一轮：内容解析

系统用 LLM 解析用户粘贴的笔记内容，判断是否是一篇旅行路线文章，并从中提取结构化意图：出发地、目的地、开始/结束日期、天数。如果内容不够清晰（`ready=false`），直接回复引导用户补充，流程结束。解析成功后，将这份意图存入 state，进入第二轮。

### 第二轮：行程生成

**1. 路线生成（智能生成）**

携带笔记原文和提取出的意图，调用智能生成，LLM 以笔记内容为参考直接规划路线。生成完成后，进行 POI 链接——将路线中的 `[[景点名]]` 匹配到系统 POI 库，补充坐标、ID 等结构化数据。

路线准备好后，将目的地、天数、日期等写入 state，触发后续并行分支。

**2. 并行 + 串行的混合规划**

路线生成节点通过 `goto=[交通, 天气]`（LangGraph Command 的 goto 传入列表）同时启动两条分支：

```
路线生成
    ├─────────────────────────┐
    ▼                         ▼
交通查询                   天气查询
    ▼
酒店推荐
    ▼                         ▼
              汇总（等待两条分支都到达）
```

- **左链路（串行）**：先查交通，再推荐酒店。酒店推荐依赖交通返回的路线信息，所以必须串行。
- **右链路（并发）**：天气查询与左链路同时进行，相互独立。
- **特殊情况**：如果出发地与目的地相同，或行程只有 1 天，跳过交通查询，直接从酒店开始。

**3. 汇总写入（fan-in）**

汇总节点标记为 `defer=True`，LangGraph 框架会等到酒店和天气两条分支都完成后才执行它。此时从 state 中读取路线、交通、酒店、天气的全部结果，统一调用 `do_route_store` 完成行程持久化，返回 `routeId` 给前端。

### 并行机制本质

并行不是一个节点内调用多个工具，而是图层面的多节点并发——`Command(goto=[node_a, node_b])` 让 LangGraph 同时激活多个节点；`defer=True` 是扇入机制，让汇总节点等所有上游分支到齐后再执行。

## Q15. 项目中用到了哪些模型，各自承担什么场景，为什么选择这个模型？

**BERT (hfl/chinese-roberta-wwm-ext-large) —— 意图分类**

与两个 LLM 节点并行运行，18ms 完成分类。v3 开关开启时 BERT 结果直接生效；关闭时由 LLM 结果兜底。用 BERT 的原因是分类任务不需要生成能力，速度远快于 LLM，作为前置过滤可以拦截大量确定性意图，节省 LLM 调用次数。

**gpt-4o-2024-11-20 —— 意图分类兜底 + 结构化提取**

BERT 的兜底模型，同时也承担所有结构化提取场景：旅行意图提取、酒店筛选条件解析、交通参数提取、小红书内容识别等。这类任务核心是按格式准确输出结构化 JSON，GPT-4o 在 instruction following 上表现稳定，比最新模型成本低。

**kimi-k2.5 —— V5 Function Calling**

V5 模式下工具调用的执行模型。Kimi 在工具调用格式化和多工具场景的准确性上有专项优化，V5 是单轮 Function Calling，工具调用准确率直接决定结果质量。

**gpt-5-chat —— 核心生成任务**

AI 捏路线、小红书导入生成路线、智能调整路线、行程风格生成。这些任务对创意和综合推理要求最高，是用户直接感知的核心体验，用最强模型保证质量。

**aliyun/qwen3.5-flash —— 极速流式（thinker）**

thinker 节点专用，流式向用户展示思考过程，延迟比质量更重要，flash 版速度最快。

**gemini-2.5-flash-lite —— 超高频轻量（问题推荐）**

每次对话都并行触发的问题推荐节点，质量要求低、调用频率最高，用最轻量的模型控制成本和延迟。

**deepseek-v3.2-web —— 联网搜索**

内置联网能力，搜索和生成合一，不需要额外调搜索 API，专用于需要实时外部信息的场景。
