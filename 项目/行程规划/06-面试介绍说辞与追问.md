# 面试介绍说辞与追问应对

> **原则：** 介绍项目时不夸大自己的贡献，不过度拔高技术难度。如实说明自己的参与程度，重点展示对系统的理解深度。

---

## 一、项目介绍说辞（1-2 分钟版）

> 适用于"介绍一下你参与过的项目"这类开场问题。

---

**参考说法：**

"我参与过一个 AI 行程规划系统的开发，叫 LibAI。这个系统的核心是帮用户生成旅游行程，用户输入目的地和出行时间，系统会自动规划每天的路线，同时给出酒店、交通、天气的建议。

技术上，后端用 FastAPI 做异步服务，核心是基于 LangGraph 搭建的 AI 工作流——把整个规划过程拆成多个节点，比如意图识别、路线规划、酒店查询、交通查询，这几个查询任务是并行执行的，最后汇总结果。

用户侧用 SSE 流式输出，不同类型的内容用不同的事件类型推送，文本是逐步流出的，行程卡片是完整数据一次推送。

此外系统还有一个特色功能，就是支持把小红书的旅游攻略笔记直接导入为可执行的行程，背后是用爬虫拿到笔记内容，再让 LLM 把非结构化的文字转成标准的行程格式。"

---

**说辞要点解析：**

- 先讲用户价值（做什么用），再讲技术方案（怎么做）
- "并行执行"、"流式输出"是实际亮点，可以展开
- 小红书导入作为差异化功能一带而过，如果面试官感兴趣会追问

---

## 二、面试官的常见追问

### 追问 1：你在这个项目里做了哪些具体的工作？

**为什么会问：** 很多人把别人做的东西说成自己做的，面试官会通过追问判断参与程度。

**如实回答的思路：**
说清楚自己负责了哪些模块，哪些是看过但没有直接写的，哪些是完全不了解的。比如：
- "我主要负责了 XXX 模块"
- "路线生成那块是同事写的，但我了解它的设计思路"
- "监控这一层我没有直接参与"

---

### 追问 2：LangGraph 和直接写 if/else 流程控制比，有什么优势？

**为什么会问：** 考察是否真的理解 LangGraph 的价值，还是只是"用了一个框架"。

**回答：**

直接写 if/else 也能实现，但 LangGraph 解决了几个具体问题：

一是**状态传递**。节点之间共享一个 State 对象，不用手动在函数之间传参，也不用担心哪个中间变量漏传了。

二是**并行控制**。fan-out/fan-in 的并行模式，用 if/else 自己实现要处理 `asyncio.gather`、异常收集、部分失败处理，容易写错。LangGraph 封装了这些。

三是**可视化和调试**。LangGraph 可以把图结构可视化出来，方便团队沟通和排查问题节点。

当然代价是引入了一个框架，有学习成本，对于非常简单的流程确实没必要用。

---

### 追问 3：流式输出是怎么做的，前端怎么接收？

**为什么会问：** 流式输出是这类 AI 项目的必答题，考察是否真正了解实现细节。

**回答：**

后端用 FastAPI 的 `StreamingResponse`，`media_type` 设为 `text/event-stream`，响应会持续推送直到结束。

每条事件的格式是：
```
event: MESSAGE
data: {"text": "这是一段文字"}

event: TRIP_CARD
data: {"routeId": "xxx", "days": 5, ...}
```

前端可以用原生的 `EventSource` 接收，也可以用 `fetch` + `ReadableStream` 手动处理。

我们有多种事件类型，文本内容是逐 token 推的，结构化的卡片数据（比如行程、酒店列表）是等全部数据就绪再一次推送，不分块。

---

### 追问 4：如果 LLM 返回结果很慢，怎么处理？

**为什么会问：** 考察对实际生产问题的思考，LLM 延迟是 AI 项目的常见痛点。

**回答：**

有几个层面：

第一是**流式输出本身就是缓解手段**，用户看到 AI 在"打字"，感知延迟比等一个完整结果低很多。

第二是**并行化**，路线/酒店/交通/天气同时查，避免串行等待。

第三是**缓存**，相同意图（同目的地、同天数、同风格）的结果缓存在 Redis 里，命中缓存直接返回。

第四是**超时兜底**，单个节点超时不能拖垮整个请求，需要设置超时并返回降级结果。

---

### 追问 5：提示词是怎么管理的，会不会经常改？

**为什么会问：** Prompt 管理是 AI 工程里特有的工程问题，考察实际经验。

**回答：**

项目有两套机制：

本地是 Markdown 文件，和代码一起部署，改了要重新发版。

生产环境用 Langfuse 管理提示词，提示词存在平台上，服务启动时拉取，也支持运行时热更新。这样改提示词不需要重新部署服务，方便快速迭代。

提示词确实经常改，尤其是意图识别和路线生成这两块，用户的输入方式多样，边界情况很多，要持续优化。

---

### 追问 6：多轮对话的上下文是怎么维护的？token 超出了怎么办？

**为什么会问：** 长对话的 token 管理是 LLM 应用的实际挑战。

**回答：**

上下文维护分两层：

一是消息历史直接放在 State.messages 里，每轮请求都带上。

二是关键信息（旅行意图：目的地、日期、人数这些）提取后单独存在 Redis，下次对话直接读，不依赖消息历史里的原文。

Token 超出的处理：有一个消息精简模块，分三个级别（STD/MINI/NANO），对话越长，历史消息压缩程度越高。早期的轮次可能只保留摘要，最近几轮保留原文。工具调用的结果（ToolMessage）使用后从历史中移除，因为它只是中间过程。

---

### 追问 7：系统出了问题怎么排查？

**为什么会问：** 考察对线上运维的理解，是否只会写代码不懂运维。

**回答：**

每个请求进来时，中间件会注入一个 `qtraceid`，所有后续的日志、LLM 调用、外部 API 请求都会带上这个 ID。

排查问题时，先用 `qtraceid` 在 Langfuse 里找到这个请求的完整 LLM 调用链，看每次调用的输入 prompt 和输出是什么，确认是哪一步出了问题。如果是外部 API（携程、地图服务）的问题，再去 OpenTelemetry 的 trace 里看。

业务指标（成功率、各节点耗时）在 QMonitor 里，有异常会触发告警。

---

### 追问 8：有没有遇到什么印象深刻的 Bug 或技术问题？

**为什么会问：** 这是考察真实参与度的好问题，没参与过很难回答。

**说明：** 这个问题需要结合你**实际参与的部分**来回答，以下提供几个项目中真实存在的潜在问题点，供参考：

- **并行节点部分失败**：4 个规划节点并行，如果天气查询超时，行程能不能正常返回？（需要每个子任务独立超时，失败只影响天气部分，不影响路线和酒店）
- **消息历史过长**：用户反复修改行程，消息越来越多，某次请求发现 token 超限。
- **LLM 输出格式不稳定**：路线生成节点要求 LLM 按固定格式输出，但偶尔格式不对导致解析失败，需要加解析兜底。
- **小红书笔记内容提取不完整**：部分笔记有反爬机制，Readability 提取到的内容残缺，LLM 无法生成完整行程。

---

## 三、可以主动抛出的亮点

面试时可以在介绍完基本情况后，主动提一两个点，引导面试官往你熟悉的方向问：

1. **"我们的并行规划设计比较有意思……"** → 引出 LangGraph fan-out、异步并行
2. **"流式输出这块做了一些区分……"** → 引出 SSE 多事件类型设计
3. **"提示词管理有一个比较实用的方案……"** → 引出 Langfuse 热更新
4. **"小红书导入功能有一些有趣的技术挑战……"** → 引出非结构化内容处理

---

## 四、不要说的话

- ❌ "这是一个非常复杂的系统" — 面试官会追问"哪里复杂"，容易被问穿
- ❌ 把整个系统说成自己一个人做的 — 如实说明分工
- ❌ "我们用了最先进的 LangGraph/LLM 技术" — 技术选型要说出为什么用，不是用来堆砌的
- ❌ 背诵技术细节而不解释原因 — 面试官想听"为什么这样做"，不是"做了什么"
