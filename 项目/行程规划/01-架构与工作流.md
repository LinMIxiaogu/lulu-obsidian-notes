# 架构与 LangGraph 工作流

---

## 为什么用 LangGraph？

普通 LLM 调用是无状态的。旅行规划需要：
- 多轮对话记忆用户意图
- 同时调用多个 API（机票 + 酒店 + 天气）
- 条件分支（用户要调整路线 vs 新建行程）
- 中间状态向前端流式推送

**LangGraph** 解决了这些问题：它把整个对话处理流程建模为一张**有向图**，每个功能是一个节点（Node），节点之间有明确的状态传递和路由规则。

---

## 主工作流全貌

```
用户消息进入
     │
     ▼
┌─────────────┐
│  reception  │  ← 接待节点：判断用户意图，决定路由
└──────┬──────┘
       │ intent 分类
       ├────────────────────────────────────┐
       ▼                                    ▼
┌─────────────┐                    ┌──────────────────┐
│   thinker   │                    │  drawer_chat     │ ← 抽屉聊天场景
│  (思考节点)  │                    └──────────────────┘
└──────┬──────┘
       │
       ▼
┌─────────────────┐
│  transcriptor   │  ← 解析消息历史，提取当前对话上下文
└────────┬────────┘
         │
    ┌────┴────────────────────────────────────┐
    │                                         │
    ▼                                         ▼
┌──────────────────────┐         ┌─────────────────────────┐
│ travel_intent_sector │         │ hotel/transport_consult  │
```

## 状态对象 (State)

整张图共享一个 `State` 对象，各节点读取和更新自己关心的字段。

```python
class State(MessagesState):
    # === 基础信息 ===
    common_state: CommonState       # 会话ID、消息ID、用户名
    
    # === 工作流控制 ===
    workflow_id: str                # 当前工作流ID
    next: str                       # 下一跳节点名
    intent: str                     # 意图分类结果
    thinking: str                   # thinker节点的推理内容
    
    # === 旅行规划状态 ===
    route_state: RouteState         # 路线中间状态
    hotel_state: HotelState         # 酒店中间状态
    transport_state: TransportState # 交通中间状态
    
    # === 各规划器输出（fan-in 收集）===
    travel_planner_param_by_travel_route_planner_node:     List[Dict]
    travel_planner_param_by_travel_hotel_planner_node:     List[Dict]
    travel_planner_param_by_travel_transport_planner_node: Dict
    
    # === 缓存控制 ===
    cache_state: CacheState
    
    # === 消息历史（继承自 MessagesState）===
    messages: List[BaseMessage]
```

> **关键理解：** `messages` 字段存所有对话历史，其他字段是规划过程中的中间结果。节点更新 State 后，LangGraph 自动传递给下一个节点。

## 节点分类速查表

| 节点名 | 类型 | 作用 |
|--------|------|------|
| `reception_node` | 路由节点 | 分析用户意图，决定进入哪个 Sector |
| `thinker_node` | AI节点 | 开启 thinking 模式，深度理解需求 |
| `transcriptor_node` | AI节点 | 提取对话上下文，精简消息历史 |
| `travel_intent_sector_node` | AI节点 | 识别出发地/目的地/日期/人数等参数 |
| `travel_plan_trigger_node` | 控制节点 | 触发并行规划，fan-out 分发任务 |
| `travel_route_planner_node` | AI节点 | 生成每日路线 |
| `travel_hotel_planner_node` | API节点 | 查询并推荐酒店 |
| `travel_transport_planner_node` | API节点 | 查询机票/火车票 |
| `travel_weather_planner_node` | API节点 | 查询天气预报 |
| `travel_planner_node` | 汇总节点 | 合并所有规划结果，调用 route_store 落库 |
| `hotel_consult_sector_node` | AI节点 | 酒店咨询（已有行程中查酒店）|
| `transport_consult_sector_node` | AI节点 | 交通咨询（查机票、火车票详情）|
| `adjust_route_sector_node` | AI节点 | 用户要求调整现有路线 |
| `question_recommend_node` | AI节点 | 根据上下文推荐用户可能继续问的问题 |
| `travel_xhs_import_*_node` | 多个节点 | 小红书笔记导入专用流程 |

## 图的构建方式（builder.py）

```python
# 伪代码示意
graph = StateGraph(State)

# 添加节点
graph.add_node("reception", reception_node)
graph.add_node("thinker", thinker_node)
graph.add_node("travel_intent_sector", travel_intent_sector_node)
# ...

# 设置入口
graph.set_entry_point("reception")

# 添加条件路由
graph.add_conditional_edges("reception", route_by_intent, {
    "travel_plan": "thinker",
    "hotel_consult": "hotel_consult_sector",
    "transport_consult": "transport_consult_sector",
    "adjust_route": "adjust_route_sector",
    "xhs_import": "travel_xhs_import_sector",
    "chat": "nodes_by_chat",
})

# 并行 fan-out
graph.add_edge("travel_plan_trigger", ["route_planner", "hotel_planner", 
                                        "transport_planner", "weather_planner"])
# fan-in
graph.add_edge(["route_planner", "hotel_planner", 
                 "transport_planner", "weather_planner"], "travel_planner")
```

## 流式输出架构

节点产出内容 → 不直接写响应，而是放入事件队列：

```
LangGraph 节点执行
     │  产出 event
     ▼
OutputQueueService (内存队列)
     │
     ▼
StreamDispatcher (事件分发器)
     │  根据 event_type 分发
     ├─→ OutputStreamHandler       → MESSAGE 文本块
     ├─→ ThinkingOutputHandler     → THINKING 思考过程
     ├─→ AssistantAgentCardHandler → AGENT_CARD 专家卡片
     ├─→ FunctionCallToolHandler   → TOOL_CALL 工具调用
     └─→ TripCardHandler           → TRIP_CARD 行程卡片
          │
          ▼
     SSE Event 推送给前端
```

前端收到的事件类型举例：

```json
// 文本消息（流式）
{"event": "MESSAGE", "data": "根据您的需求，我为您规划了..."}

// 思考过程
{"event": "THINKING", "data": "用户想去三亚5天，需要考虑..."}

// 行程卡片（结构化数据）
{"event": "TRIP_CARD", "data": {"days": 5, "routeList": [...]}}

// 推荐问题
{"event": "QUESTION_RECOMMEND", "data": ["需要安排接送机吗？", "要推荐亲子酒店吗？"]}
```
