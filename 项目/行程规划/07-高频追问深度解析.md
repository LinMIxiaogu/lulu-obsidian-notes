# 高频追问深度解析

> 以下四个话题是 AI 工程岗面试中最常被深挖的。内容基于项目实际代码，引用了具体的函数名和实现细节，回答时做到有据可查。

---

## 一、意图识别是怎么做的？

### 核心结论先说

意图识别是**三层递进**结构，不是一个节点做完所有事：

```
用户输入
   │
reception_node       ← 第一层：粗粒度分类（去哪个 Sector）
   │
transcriptor_node    ← 第二层：上下文整理，提取四类意图的约束条件
   │
travel_intent_sector_node ← 第三层：细粒度抽取（出发地/目的地/日期/人数等词槽）
```

---

### 第一层：Reception 节点（粗分类）

**分类目标**（Intent 类别）：

| 意图类别                       | 说明            |
| -------------------------- | ------------- |
| `chatbot`                  | 闲聊、非旅游相关      |
| `travel_intent_sector`     | 旅行规划（新建行程）    |
| `transport_consult_sector` | 交通咨询（查机票/火车）  |
| `hotel_consult_sector`     | 酒店咨询          |
| `adjust_route_sector`      | 调整已有路线        |
| `trip_consult_sector`      | 旅游咨询（问景点、问攻略） |
| `drawer_chat`              | 抽屉聊天场景        |

**判断方式是三路竞争，按优先级依次尝试：**

1. **规则快速路由**（`pre_router_sector()`）：最快，处理明确的模式匹配，命中直接返回
2. **BERT 分类模型**（`http_request_service.get_bert_reception_v2()`）：调用内部 BERT 服务，中等延迟
3. **LLM 兜底**（`get_chat_open_ai_by_biz("reception").with_structured_output(ReceptionResult)`）：前两者都不确定时兜底，最慢

输出结构：
```python
class ReceptionResult(TypedDict):
    next: str      # 目标节点名，如 "travel_intent_sector"
    thinking: str  # LLM 的判断依据（调试用）
```

---

### 第二层：Transcriptor 节点（上下文整理）

**职责**：把多轮对话历史中散落的意图信息汇总成结构化对象。

**输出（TranscriptContext TypedDict）：**
```python
{
    "current_intent": "用户当前意图的一句话摘要",
    "hotel_intent": {
        "constraints": "明确想要的条件（如海景房）",
        "hotel_query": "剔除日期城市后的纯条件描述",
        "background_memory": "被新条件覆盖的旧条件",
        "negative_constraints": "明确不想要的条件（如不要靠路边）"
    },
    "flight_intent": {...},
    "train_intent": {...},
    "trip_intent": {
        "label_constraints": "旅行偏好标签（亲子/探险/海滩等，非地点日期）"
    },
    "curr_travel_intention": {
        "departure": "出发地",
        "arrival": "目的地",
        "startDate": "2026-04-01",
        "endDate": "2026-04-05",
        "days": 5,
        "adultCount": 2,
        "childCount": 1
    }
}
```

**为什么要单独一个节点做这件事？**
因为用户可能说"不要带泳池的"（否定约束）、"改成5号到"（覆盖旧日期）、"再加一个大人"（增量修改），这些语义变化很难用简单规则处理，用一个专门的 LLM 节点来维护意图状态更可靠。

---

### 第三层：Travel Intent Sector 节点（词槽抽取）

提示词文件：`prompts/travel_intent_sector.md`

**词槽抽取规则（部分）：**

| 字段    | 规则                                    |
| ----- | ------------------------------------- |
| 地名标准化 | 帝都→北京、魔都→上海、机场三字码 PEK→北京              |
| 日期推断  | 固定节日（元旦、劳动节、国庆）可推算具体日期；春节等农历节日**不推断** |
| 相对日期  | 基准时间从 `<<CURRENT_TIME>>` 注入，"后天"依此计算  |
| 天数校验  | 范围 [1, 365]；与日期区间冲突时，以日期为准            |
| 人数    | 成人 [1, 30]，儿童 [0, 10]                 |
| 禁止输出  | null 值、空字符串一律不输出，只输出有值字段              |

**优先级逻辑：**
```
ai_generate_trip_route_intention（用户之前已确认过的）
    ↓ 无则
transcript_curr_travel_intention（Transcriptor 整理的）
    ↓ 无则
LLM 重新从当前消息抽取
```

---

### 面试追问预案

**Q：为什么不用一个节点做所有意图识别？**

三个节点职责不同、消耗不同：Reception 需要最快（BERT 比 LLM 快），Transcriptor 做状态维护（需要完整历史，但不需要旅行词槽），TravelIntentSector 做细粒度抽取（只在确认进入规划流程后才调用）。拆开可以按需调用，不每次都跑最贵的节点。

**Q：BERT 路由和 LLM 路由有什么差异？**

BERT 分类模型是训练好的分类器，速度快（毫秒级）、成本低，但对边界 case 和新模式泛化差。LLM 泛化能力强，但延迟高、成本高，只作为兜底。两者共存是效率和准确率的折中。

**Q：用户改变主意怎么处理？**

Transcriptor 节点在每轮对话都运行，它维护的 `hotel_intent.background_memory` 字段专门记录"被覆盖的旧条件"。新意图进来后，旧约束并不直接删除，而是移入 background_memory，供调试和日志使用。LLM 看到新的 constraints 会覆盖旧的，保证行为正确。

## 二、上下文压缩是怎么做的？

### 核心机制：消息转换器 + 三级压缩

不是把历史消息截断丢弃，而是**对每条历史消息单独做内容压缩**，保留条数但缩减每条的长度。

**实现文件：** `framework/message/message_context.py`

---

### 三个压缩级别

```python
class ContextLevel(Enum):
    STD  = "std"   # 完整内容
    MINI = "mini"  # 保留关键信息，去掉细节
    NANO = "nano"  # 1-2 句摘要
```

---

### 按消息类型定制压缩策略

不同类型的 AI 回复有不同的转换器（都继承 `BaseMessageConverter`）：

| 消息类型                                  | MINI 级              | NANO 级                        |
| ------------------------------------- | ------------------- | ----------------------------- |
| 普通 AI 回复                              | 同 STD               | 固定文案 "AI回复了用户，详情略。" + 最后 2 句话 |
| 行程规划结果（TravelPlannerConverter）        | 只保留景点和酒店名称，去掉描述和小贴士 | "AI展示了行程规划信息，详情略。"            |
| 酒店推荐（HotelConsultSectorConverter）     | 只保留酒店名+价格+评分        | "AI展示了推荐酒店信息，详情略。"            |
| 交通推荐（TransportConsultSectorConverter） | 只保留起止站信息            | "AI展示了交通信息，详情略。"              |
| 旅行意图（TravelIntentSectorConverter）     | 同 STD               | "AI总结了用户的旅行需求，详情略。"           |

**NANO 级别最后2句话的提取逻辑（`get_text_suffix` 函数）：**
- 按 `。！？.!---` 等标点分句
- 取最后 N 句（N 可配置，默认 2 句）
- 如果末尾内容不足 20 字，直接用固定摘要文案代替

---

### 各节点使用哪个级别

| 节点                          | 压缩级别              | 原因                 |
| --------------------------- | ----------------- | ------------------ |
| `thinker_node`              | **NANO**          | 思考阶段只需要摘要，节省 token |
| `trip_react_reception_node` | **NANO**，最多取近 5 轮 | 分类任务不需要细节          |
| `chatbot`                   | **STD**           | 完整回答需要完整上下文        |
| `reception`                 | **STD**           | 分类需要完整意图信息         |

---

### 面试追问预案

**Q：为什么不直接截断历史消息（只保留最近 N 条）？**

截断会丢失早期的关键信息。比如用户第 1 轮说了"我带了一个老人"，第 10 轮问"酒店有没有无障碍设施"，截断掉第 1 轮就没法关联。压缩保留了所有轮次，只是把细节去掉，关键信息（人物、地点、需求）还在。

**Q：NANO 级别的"最后 2 句"有什么意义？**

LLM 输出的最后几句通常是总结性语句（"为您规划了以上行程，如需调整..."），保留尾部比保留头部对后续推断更有用。这是一个经验性的设计，不是理论最优。

**Q：这样压缩会不会丢失重要信息，导致 AI 回答不准？**

会有风险。所以只有 thinker 和 reception 这些做"判断"的节点用 NANO。真正生成内容的节点（chatbot、travel_intent_sector）用 STD，保留完整上下文。分层使用是关键。

## 三、RAG 用到了吗？怎么用的？

### 结论先说

**用了，但不是教科书式的 RAG（没有本地向量数据库 + Chunk + Rerank 那套）**。是三种形式：

1. **路线召回**：基于行程参数做检索，返回历史相似路线
2. **小红书笔记 Embedding 检索**：根据用户目的地检索相似笔记，生成路线
3. **知识库 HTTP 检索**：调用内部 HTTP 服务查知识库，结果注入 prompt

---

### 形式 1：路线召回

用户提出行程需求后，先不急着让 LLM 生成，而是**先去数据库检索有没有匹配的历史路线**：

```python
# graph/nodes_by_travel_plan.py
result = route_service.get_recommend_route_list(travel_intention, state)

if result.get("routeRecommendType") == "route_retrieve":
    # 使用召回路线，跳过 AI 生成
    goto = "route_retrieve_planner"
elif result.get("routeRecommendType") == "embedding_by_xhs_note":
    # 使用小红书笔记 embedding 路线
    goto = "generate_route_use_embedding_router"
else:
    # 没有召回结果，走 AI 生成
    goto = "generate_text_travel_plan"
```

State 中会记录召回路线的来源 ID（`trip_route_meta_route_id`），落库时也会存进去，用于数据追溯。

**这属于 RAG 吗？** 本质上是"先检索再生成"的思路，检索对象是结构化的历史行程数据，不是文本向量检索，算是 RAG 的变体。

---

### 形式 2：小红书笔记 Embedding 检索

当召回类型是 `embedding_by_xhs_note` 时，系统用用户的目的地意图去检索相似的小红书笔记（Embedding 向量相似度），再基于匹配的笔记内容生成路线。

```python
# graph/builder.py 中注册了专门节点
sub.add_node("generate_route_use_embedding_xhs_note_route_planner", ...)
sub.add_node("generate_route_use_embedding_xhs_note", ...)
sub.add_node("generate_route_use_embedding_xhs_note_ex", ...)
```

**这个流程的路由逻辑：**
```python
# 先判断目的地是否是热门城市
if is_top_destination:
    goto = "generate_route_use_embedding_xhs_note_route_planner"  # 用笔记生成
else:
    goto = "generate_text_travel_plan"  # 用纯文本生成
```

**热门城市走笔记 Embedding 的原因**：热门城市有大量高质量小红书攻略，Embedding 检索出来的内容质量高，比纯 LLM 生成更接地气。

---

### 形式 3：知识库 HTTP 检索

chatbot 节点和 trip_react 节点都有 RAG 逻辑：

```python
# graph/nodes.py（chatbot_node）
retrieval_result = http_request_service.retrieval_chatbot_kb(
    query=state.get("human_input_content", "")
)
if chatbot_kb_switch and retrieval_result:
    # 把检索结果注入 prompt
    chatbot_knowledge_dict = retrieval_result
```

```python
# graph/trip_react_node.py
if trip_react_rag_enable:
    retrieval_result = http_request_service.retrieval_chatbot_kb(
        query=human_input_content,
        top_k=5,
        similarity_threshold=0.85,  # 相似度阈值
    )
```

**知识库内容**：旅游相关的 FAQ、景点信息、攻略知识——具体内容存在内部服务里，通过 HTTP 接口访问，项目不直接维护向量库。

`similarity_threshold=0.85` 说明只有相似度足够高的文档才会被注入，避免低质量检索结果干扰 LLM。

---

### 面试追问预案

**Q：为什么不用本地向量数据库（Pinecone/Weaviate）？**

内部服务化的选择。Embedding 检索服务由公司统一维护，各业务通过 HTTP 调用，避免每个项目都维护一套向量数据库。缺点是依赖内部服务的稳定性，没有本地控制权。

**Q：RAG 的检索质量怎么保证？**

知识库检索设了 `similarity_threshold=0.85` 的相似度门槛，低于门槛不注入。路线召回是基于结构化参数（目的地、天数、标签）匹配，不是模糊文本相似度，准确率更高。小红书笔记 Embedding 只在热门城市开启，因为非热门城市的笔记数量不足，强行召回质量差。

**Q：RAG 的结果是怎么注入到 prompt 里的？**

检索结果作为变量注入提示词模板，类似：
```
## 参考知识
{retrieval_knowledge}

## 用户问题
{user_query}
```

知识内容在 prompt 里是显式可见的，LLM 会优先参考这些内容回答。

## 四、多 Agent 有吗？

### 结论先说

**严格来说不是"多 Agent"系统，是单图多节点架构。** 但有几个设计点值得提：

- 整个系统是**一张大图**（父图 + 路线子图），所有节点共享同一个 State
- `route_graph` 是唯一的**子图（subgraph）**，有独立的处理器注册表
- 各 Sector 节点（hotel、transport、trip_consult 等）在概念上是独立的功能域，但技术上是同一图的节点

---

### 图的层次结构

```
父图（build_graph）
├── thinker_node          ← 思考，与主流程并行
└── parallel_node（子图 build_parallel_graph）
    ├── reception_node
    ├── transcriptor_node
    ├── travel_intent_sector_node
    ├── transport_consult_sector_node
    ├── hotel_consult_sector_node
    ├── adjust_route_sector_node
    ├── travel_plan_trigger_node
    │   ├── travel_route_planner_node
    │   ├── travel_hotel_planner_node       ← 四节点并行
    │   ├── travel_transport_planner_node
    │   └── travel_weather_planner_node
    ├── travel_planner_node（汇总落库）
    └── route_graph_node（路线子图）
            └── build_route_graph（内部子图）
                ├── select_processor_node
                ├── build_param_node
                ├── get_route_ref_node
                ├── route_ref_to_json_node
                ├── standardization_node
                └── route_store_node
```

---

### 路线子图（route_graph）的特殊设计

路线规划逻辑放在独立子图里，内部有**处理器注册表模式**：

```python
# workflows/routing/route/route_workflow.py
route_processor_register = {
    "RouteRetrievalProcessor":       RouteRetrievalProcessor,      # 召回路线
    "ImportContentRouteProcessor":   ImportContentRouteProcessor,  # 小红书导入
    "TextRouteProcessor":            TextRouteProcessor,           # 纯文本生成
    "EmbeddingNoteRouteProcessor":   EmbeddingNoteRouteProcessor,  # Embedding生成
}
```

每个 Processor 实现同一套方法（`build_route_param / get_route_ref / standardization`），子图根据路线类型选择对应的 Processor 执行。如果某个 Processor 执行失败，可以 fallback 到 `TextRouteProcessor`（纯 LLM 生成，最不依赖外部）。

**这是策略模式（Strategy Pattern）在 Agent 工作流里的应用。**

---

### 节点间怎么"通信"

不是消息队列，而是通过 `Command` 对象修改共享 State：

```python
# 单跳转
return Command(
    update={"next": "hotel_consult_sector", "intent": "hotel"},
    goto="hotel_consult_sector"
)

# 并行 fan-out
return Command(
    update={...},
    goto=["travel_route_planner_node", "travel_hotel_planner_node",
          "travel_transport_planner_node", "travel_weather_planner_node"]
)
```

节点之间没有直接调用，全部通过 goto + State 共享数据。

---

### 和"真正的多 Agent"有什么区别

| 维度 | LibAI 的设计 | 真正的多 Agent |
|------|------------|--------------|
| Agent 数量 | 1 个图（含 1 个子图）| 多个独立 Agent 进程 |
| 通信方式 | 共享 State + goto | 消息队列 / RPC / Handoff |
| 状态隔离 | 无隔离，全局共享 | 每个 Agent 有自己的状态 |
| 独立部署 | 不能，紧耦合 | 可以独立扩缩容 |
| 适合场景 | 流程固定、需要低延迟 | 流程动态、需要高弹性 |

---

### 面试追问预案

**Q：为什么不做成多 Agent 而是单图多节点？**

行程规划的流程是相对固定的（意图识别 → 规划 → 汇总），用多 Agent 的好处不明显，反而引入了 Agent 间通信的复杂性和网络开销。单图共享 State 更简单、延迟更低，对这个业务场景更合适。

**Q：route_graph 子图为什么要单独拆出来？**

路线规划这块逻辑最复杂（多种处理器、fallback 机制、结构化步骤）。拆成子图有两个好处：
1. 独立测试：路线子图可以单独跑单元测试
2. 逻辑隔离：不把复杂的 Processor 注册表逻辑堆在主图里，主图保持清晰

**Q：如果要改成真正的多 Agent 架构，你会怎么拆？**

可以考虑把独立性强的 Sector 拆出来：`hotel_consult_agent`、`transport_consult_agent` 各自独立，通过 LangGraph 的 `handoff` 机制或消息队列协作。主要代价是需要解决状态同步问题（当前用共享 State 非常方便），以及每个 Agent 的独立部署和监控。

**Q：并行的四个规划节点（路线/酒店/交通/天气）如果有一个超时，怎么处理？**

这是一个实际的工程问题。LangGraph 的 fan-in 会等所有并行节点完成才继续，如果某个节点卡死会阻塞整个流程。需要在节点层面设置超时，超时后返回空结果或降级数据，而不是让 fan-in 永久等待。这是项目里需要关注的一个潜在问题点。
